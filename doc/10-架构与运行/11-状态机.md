# 11 - 状态机

> 当前权威设计：ADR 0038。状态写入统一经 `plan_update`，check 只记录审核结果。

## Goal 状态

```text
Task Plan:          active ──tasks all done + update──→ done
Phase/Goal Plan: pending ──proposal confirmed──→ active
                                      │
                                      ├── technical failure / user stop / agent blocker ──→ paused
                                      │                                                │
                                      └── all phases done + goal check approved        └──resume──→ active
                                          + update ──→ done
```

| 状态 | 含义 |
|---|---|
| `pending` | Phase/Goal Plan 正在语义预审或等待用户确认 |
| `active` | 当前 Plan 正在执行，或正在修复 rejected check |
| `paused` | 自动续跑停止，等待 `/dgoal resume` 或用户处理 |
| `done` | 完成守卫通过并由 `plan_update(target=goal,status=done)` 收口 |

完成后先持久化 done / 清空活动 Plan，再尝试更新 UI；TUI 异常不能阻断闭环。

## Phase 状态

```text
pending ──task 开始──→ in_progress ──完成守卫 + plan_update──→ done
   │                         │
   └────────阻塞─────────────┴──→ blocked ──解除──→ in_progress
```

中间状态可由 task 聚合，但 phase 不会因 task 状态聚合自动 done：

- Phase Plan：task 全部 `done` 后，`plan_update` 可写 phase done；blocked 表示尚未完成。
- Goal Plan：还必须有当前 plan revision 的 `phase_check=approved`。
- Task Plan：隐藏 phase 不作为工具状态目标，goal 完成直接检查平铺 task。

## Task 状态

```text
pending ──→ in_progress ──→ done
   │               │
   └──────────────→ blocked ──→ in_progress
```

- 不能从 pending 跳过执行直接 done，也不能从 in_progress 回到 pending。
- done 必须带可独立复验的 `evidence`，且不回退；发现遗漏时新建 follow-up task。
- blocked 必须有 `blockedReason`；解除为 in_progress 时清除旧阻塞原因。
- 未完成依赖会阻止 task 进入 in_progress。

## Check 状态

每次 check 写 `CheckRecord`：

```text
running → approved | rejected | audit_error
```

记录包含 scope、report、checkedAt 与 plan revision。它与 phase/goal 状态正交：

- approved：允许后续 `plan_update` 通过对应完成守卫；不自动 done。
- rejected：Plan 保持 active，agent 修复；写操作增加 revision 后旧记录失效。
- audit_error：不是业务拒绝，goal 进入 `paused(audit_error)`。

## Plan Revision

`task_plan` 替换、`plan_create`、会改变受审事实的 `plan_update` 都增加 revision，并清理相应 check：

- task/phase 变化使所属 phase check 与 goal check 失效；
- phase done 等全局进度变化使 goal check 失效；
- 纯读不改变 revision；check 运行期间 revision 若变化，本轮结果丢弃，不写回新 revision。

## Paused

新运行时保留固定可靠性出口：

| `pauseReason` | 来源 | resume |
|---|---|---|
| `user_abort` | 用户中断 | 恢复原 Plan |
| `model_error` | 主模型连续错误 | 恢复重试 |
| `audit_error` | 审核器候选耗尽/协议异常/timeout | 清理对应审核故障范围后重试 |
| `no_progress` | 连续 3 轮无工具进展 | 清零空转计数后继续 |
| `agent_blocked` | `plan_update(target=goal,status=paused,reason=...)` | 保留原因，等待用户解除 |

Check 的业务拒绝不会切换 Goal 状态，也不会按固定拒绝次数自动暂停；Plan 持续保持 `active`，直到修复后重审或命中上述技术熔断。`/dgoal resume` 只有在恢复 prompt 成功入队后才保持 active；发送失败会恢复原 paused 原因，避免假恢复后永久停滞。
