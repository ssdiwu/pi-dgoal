# 11 - 状态机

> dgoal 三层状态机。决策依据见 `../决策档案/0004-rejected与paused的状态语义.md`、`../决策档案/0005-step-blocked状态.md`，术语见 `术语表.md`。

## Goal 状态机

```
pending ──→ active ──→ done              (正常路径)
              │  ↑
              ↓  │ rejected               (终审不过，硬约束重回，prompt 钉问题)
              │  │  ×3终审不过
              ↓  ↓
            paused (audit_failed_3x) ──resume清零──→ active
            paused (user_abort/model_error/audit_error/no_progress) ──resume不清零──→ active
```

| 状态 | 含义 |
|---|---|
| `pending` | 启动中，START prompt 尚未发出（背景固化、计划提案、用户确认期间）；**不计入正式执行耗时** |
| `active` | loop 正常推进；**用户确认计划后才开始正式计时** |
| `rejected` | 终审（最后 phase 的大建检）不过的回环态。每轮 prompt 钉着未过问题，agent 修完可再触发终审 |
| `paused` | 停止续跑。两种语义，见下 |
| `done` | 终审通过，goal 关闭，loop 结束 |

**done 的终结保证（UI 边界容错）**：`finalizeGoal` 落 `done` 后，会调用主程序实现的 UI 边界（`planOverlay` 的完成闪现、状态栏清空）来展示最终状态。这些 UI 调用用 try/catch 包裹——主程序 TUI 渲染异常不阻断 goal 状态清空（`currentGoal` 置空 + `persistGoal(null)`）。状态机一致性优先于最终 UI 展示：UI 展示失败时 goal 仍正确终结，只是不显示最终完成浮层。

## Phase 状态聚合与建检门

phase 的 `in_progress` / `blocked` 等中间状态可由 task 状态聚合；task 全部进入终态只表示该 phase 具备建检条件，不能自动把 phase 写成 `done`。phase 正式完成的唯一入口是 `dgoal_check`，只有独立建检通过后才调用 `setPhaseCompleted` 写入 `done`。

## Paused 的两种语义（pauseReason 字段）

`GoalState.pauseReason` 区分，resume 时按 reason 决定是否清零 rejected 计数：

| pauseReason | 触发 | resume 行为 |
|---|---|---|
| `user_abort` | 用户主动中断 | 不清零，恢复原状继续 |
| `model_error` | 模型错误重试耗尽 | 不清零，恢复原状继续 |
| `audit_error` | 审核器自身出错/中断/无结论 | 不清零，恢复原状继续 |
| `no_progress` | 连续 3 轮正常结束但无工具调用（空转保护） | 不清零，恢复原状继续 |
| `audit_failed_3x` | 终审连续 3 次不过（能力到顶） | **清零** rejected 计数，给 agent 重新机会 |

**为什么 audit_failed_3x 进 paused 而非直接清退出**：paused 不自动续跑（现有代码只在 active 续跑），不烧 token。用户可 resume 清零重试，或搁置。直接清退出会丢失 goal 上下文。

**为什么两种 paused 要分开**：异常中断是瞬时/外部故障，resume 重试合理（本就无 rejected 计数概念）；能力到顶是 agent 修不好，resume 清零给新机会。用 reason 字段区分 resume 语义，避免"同状态不同恢复行为"的混乱。

**无进展熔断（no_progress）**：`agent_end` 检测到正常结束（`stopReason=stop`）但本轮没有任何工具调用时，连续 3 轮即暂停。发生工具调用（`tool_execution_start`）或 plan/goal 状态推进时重置计数。计数在 startGoal / clearGoal / resyncGoalFromSession / finalizeGoal 时清零，不跨 goal/session 继承。这是空转保护，避免 agent 反复回复"未完成"却烧 token。

## Rejected 计数

- 终审不过 → goal 进 rejected，`rejectedCount += 1`。
- `rejectedCount` 达 3 → 转 `paused`（pauseReason=`audit_failed_3x`），停止续跑。
- phase 建检的有效拒绝只回当前 phase 的 `in_progress`，不累计 goal 的 `rejectedCount`，也不触发三次暂停。
- 候选技术/协议故障不计入 `rejectedCount`；候选各在本次审核中只尝试一次，成功候选按审核范围粘住复用。
- resume（audit_failed_3x）→ 清零 `rejectedCount`，回 active。
- resume（audit_error）→ 按 `auditErrorScope` 清除对应 phase/goal 的候选故障记录，保留另一范围的健康候选；旧 goal 缺少范围字段时兼容为全量清除，回 active。
- resume（其他 reason）→ 不动 `rejectedCount`，回 active。

## Phase 状态机

```
pending ──→ in_progress ──→ done      (task 聚合，done 唯一入口是 dgoal_check)
              │    ↑
              ↓    │
            blocked              (有 task blocked 且无 in_progress task)
```

- **中间态由其下 task 聚合**：有 in_progress 的 task→in_progress；有 blocked 且无 in_progress→blocked；全 task 终态只表示具备建检条件，不自动写入 done。
- **done 唯一入口是 `dgoal_check`**（阶段建检门）。agent 不能用 dgoal_plan 直接标 phase done。
- **空 phase（未拆 task）可直接标 blocked**：整阶段判定不可行时，不进去拆 task。

## Task 状态机

```
pending ⇄ in_progress ──→ done        (done 不回退)
              ↑    │
              │    ↓
              └── blocked                  (blocked 可回退 in_progress)
```

- 四态：`pending | in_progress | done | blocked`。
- **done 不回退**：发现错了新建接续 task（blockedBy 指向原 task），保持历史不失真。
- **blocked 可回退 in_progress**：外部 blocker 解除后重试合理。与 done 区分（done 是"真做完"，blocked 是"暂时卡住"）。
- **blocked 必带 reason**：可被 dgoal_check 审计（真 blocker 还是偷懒）。
- **blocked 不触发 goal 停止**：task 级诚实失败，goal 级停止靠终审 rejected。
