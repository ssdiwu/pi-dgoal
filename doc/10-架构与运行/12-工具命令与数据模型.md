# 12 - 工具、命令与数据模型

> agent 与 dgoal 状态机交互的工具、用户侧命令、持久化数据模型。决策依据见 `adr/0002`、`adr/0003`，术语见 `术语表.md`。

## 工具（统一 `dgoal_` 前缀）

agent 与 dgoal 交互统一用工具，命名 `dgoal_` 前缀对齐 `/dgoal` 命令。原 `loop_complete` 已改名 `dgoal_done`。

| 工具 | 职责 | 操作重量 |
|---|---|---|
| `dgoal_propose` | 启动闸门提交 goal + phases（+ 可选初始 task），触发确认 UI | 中（触发 UI） |
| `dgoal_plan` | task/phase 的 CRUD：建/改状态/依赖/标 task completed/标空 phase blocked | 轻（纯本地快操作） |
| `dgoal_check` | phase completed 的唯一入口（阶段建检门）；最后一个 phase 的 check 即终审 | 重（spawn 独立子进程） |
| `dgoal_done` | ≈ 标最后一个 phase completed（走 dgoal_check 终审）+ 关 goal | 重（触发终审） |

### 关键不变式

- **dgoal_plan 不能标 phase completed**——只能标 task。phase completed 必须走 dgoal_check（建检不可绕过）。
- **dgoal_check 是 phase completed 唯一入口**。颗粒度：task 级不 check（太细），phase 级阶段建检，goal 级终审。
- **dgoal_done ≈ 标最后 phase completed**：语义上 dgoal_done 不是独立动作，而是"完成最后 phase"的自然结果（走终审）。

## 命令（用户侧）

`/dgoal` 命令 + 子命令，用户控制，不是 agent 工具。每个子命令支持全拼 + 单字母别名（`s/p/r/c`）：

```
/dgoal <goal>       启动（触发启动闸门）
/dgoal status | s   查看当前 goal 和迭代
/dgoal pause  | p   暂停自动续跑，保留 goal
/dgoal resume | r   恢复暂停的 goal
/dgoal clear  | c   清除当前 goal
```

## 数据模型

```ts
interface LoopGoal {
  id: string;
  objective: string;              // goal 文本简述
  status: "pending" | "active" | "rejected" | "paused" | "done";
  startedAt: number;
  updatedAt: number;
  iteration: number;
  contextSummary?: string;        // 启动背景固化（现有）
  plan?: TaskPlan;                // phase + task 两层（新）
  verification?: string;          // goal 级验证：跨 phase 全局说明（新）
  pauseReason?: "user_abort" | "model_error" | "audit_error" | "audit_failed_3x";  // 新
  rejectedCount?: number;         // 终审连续不过计数，×3 转 paused(audit_failed_3x)（新）
}

interface TaskPlan {
  phases: Phase[];
  nextId: number;
}

interface Phase {
  id: number;
  subject: string;               // 阶段性目标
  description?: string;
  status: "pending" | "in_progress" | "completed" | "blocked";  // 由其下 task 聚合，空 phase 可直接 blocked
  tasks: Task[];
  blockedReason?: string;         // 空 phase 直接 blocked 时的原因
}

interface Task {
  id: number;
  subject: string;
  description?: string;
  activeForm?: string;           // in_progress 时浮层显示
  status: "pending" | "in_progress" | "completed" | "blocked";
  blockedBy?: number[];          // 依赖图（可跨 phase，深度不限 → 涌现分解）
  evidence?: string;             // 完成证据：可被 dgoal_check 独立复验的命令/文件/测试结果，非自述
  blockedReason?: string;        // blocked 时必带
}
```

### evidence 形态

task 级 evidence 必须是**可被 dgoal_check 独立复验**的形态（建检模式启示）：命令（如"跑 `npm test auth` 全过"）、文件、测试结果——不是 agent 的文字自述（后者 agent 能编，前者 check 能复验）。

goal 级 `verification` 是跨 phase 的全局说明，与 task 级 evidence 互补。

### verification 必填（ADR 0007）

`dgoal_propose` 提交的 `verification` 是**必填**，不是可选：没有可验收完成口的 goal 不应进入启动闸门。

- **来源**：可参考 `contextSummary` 里的“验收标准”，但必须由主代理在 `dgoal_propose` 中显式提交（不做静默自动回填）。
- **工具层校验**：`validateProposalInput` 校验 `verification` 非空（trim 后）；空则在工具层直接拒绝（`details.error: "no verification"`），不推给用户确认。
- **空话拦截**：工具层只做“非空”，不做“是不是空话”的启发式判断；空话靠 `buildProposePrompt` 文案引导 + 终审 auditer 兜底。
- **用户确认**：`verification` 在启动闸门确认框里与 goal / phases 并列展示；用户只做这一次总确认。

## 持久化

复用现有 `dgoal-state` custom entry（`api.appendEntry("dgoal-state", { goal })` + `loadGoal` 按 customType 过滤 last-write-wins）。不另起载体。三层内容（goal/phase/task）都持久化——AI 可见 + compact 后可恢复。

**工具名不是持久化键**：dgoal 恢复靠 custom entry 的 customType，与工具名无关（区别于 rpiv-todo 靠 toolResult.details + 工具名 replay）。故 `loop_complete`→`dgoal_done` 改名安全。
