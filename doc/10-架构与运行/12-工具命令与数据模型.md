# 12 - 工具、命令与数据模型

> agent 与 dgoal 状态机交互的工具、用户侧命令、持久化数据模型。决策依据见 `../决策档案/0002-启动闸门与工具回调提交计划.md`、`../决策档案/0003-dgoal-check单工具承载自检与终审.md`，术语见 `术语表.md`。

## 工具（统一 `dgoal_` 前缀）

agent 与 dgoal 交互统一用工具，命名 `dgoal_` 前缀对齐 `/dgoal` 命令。原 `loop_complete` 已改名 `dgoal_done`。

| 工具 | 职责 | 操作重量 |
|---|---|---|
| `dgoal_propose` | 启动闸门提交 goal + phases（+ 可选初始 task）、冻结独立验收条件和完成后用户复核项，以及可选边界信号（`nonGoals` / `guardrails` / `budget`），触发确认 UI | 中（触发 UI） |
| `dgoal_plan` | task/phase 的 CRUD：建/改状态/依赖/标 task done/标空 phase blocked | 轻（纯本地快操作） |
| `dgoal_check` | phase done 的唯一入口（阶段建检门）；单 phase 时一次审核同时核验 phase 与 goal，记录统一审核凭据（ADR 0018） | 重（spawn 独立子进程） |
| `dgoal_done` | 在所有 phase 都已通过后声明 goal 完成；单 phase 复用统一审核凭据直接关闭（不重复终审），多 phase 触发 goal 级终审（含 phase(id)/goal/user_review 三路归因）；唯一关 goal 入口 | 重（触发终审） |

### 关键不变式

- **dgoal_plan 不能标 phase done**——只能标 task。phase done 必须走 dgoal_check（建检不可绕过）。
- **dgoal_check 是 phase done 唯一入口**。颗粒度：task 级不 check（太细），phase 级阶段建检。单 phase 时一次审核同时核验 phase 与 goal，记录统一审核凭据；多 phase 即使最后一个 phase 通过，也还没完成 goal，需 dgoal_done 终审。
- **dgoal_done 是 goal 级终审唯一入口**：只有当所有 phase 都已通过 `dgoal_check` 后，才允许调用 `dgoal_done` 触发终审并关闭 goal。

### paused 状态下的工具边界

goal 存在但处于 `paused` 时，工具不得误报为"目标不存在"，而是返回结构化 paused 结果（含 `pauseReason` + `/dgoal resume` 指引）：

| 操作 | paused 下 |
|---|---|
| `dgoal_plan list` / `get` | ✅ 允许只读 |
| `/dgoal status` | ✅ 允许只读 |
| `dgoal_plan create` / `update` | ❌ 拒绝，提示 resume |
| `dgoal_check` | ❌ 拒绝，提示 resume |
| `dgoal_done` | ❌ 拒绝，提示 resume |

### phaseId 与 phaseNumber

`dgoal_check` 和 `dgoal_plan`（create/list）的 phase 定位参数支持两种写法，二选一（同时提供会被拒）：

- `phaseId`：真实 phase ID（旧 plan 可能非连续，如 `#1/#4/#8`）
- `phaseNumber`：阶段序号（1-based，第几个阶段）

phase 找不到时返回完整阶段列表（阶段序号 + 真实 phaseId + 标题），当前 phase 高亮，避免模型静默猜测。

## 命令（用户侧）

`/dgoal` 命令 + 子命令，用户控制，不是 agent 工具。每个子命令支持全拼 + 单字母别名（`s/p/r/c`）：

```
/dgoal <goal>       显式目标启动（触发启动闸门）
/dgoal              裸启动：承接当前前文共识进入启动闸门；无前文时提示改用 /dgoal <goal>
/dgoal status | s   以 center modal 查看完整 plan 状态（goal / phase / task）
/dgoal pause  | p   暂停自动续跑，保留 goal
/dgoal resume | r   恢复暂停的 goal
/dgoal clear  | c   清除当前 goal
```

**路由约定（v0.5.2）**：裸 `/dgoal`（空 args）不再等于 `status`；它走启动闸门的“路径 B”（承接前文，由主代理用 `dgoal_propose` 归纳 objective）。查看状态统一用显式 `/dgoal s`。

## 数据模型

```ts
interface GoalState {
  id: string;
  objective: string;              // goal 文本简述；裸 /dgoal 启动时 pending 期间可暂为承接占位，确认后被 proposal.objective 覆盖
  status: "pending" | "active" | "rejected" | "paused" | "done";
  startedAt: number;
  updatedAt: number;
  iteration: number;
  contextSummary?: string;        // 启动背景固化（只补背景，不替代 objective / verification）
  plan?: TaskPlan;                // phase + task 两层
  verification?: string;          // goal 级验证：跨 phase 全局说明
  acceptanceCriteria?: AcceptanceCriterion[]; // 冻结的 LLM 可独立验收条件；旧 session 缺失时按空处理
  userReviewItems?: string[];      // 完成后交给用户复核，不阻塞 done
  nonGoals?: string[];            // 启动闸门确认的“不做什么”边界
  guardrails?: string[];          // 启动闸门确认的高风险边界 / 明确不碰什么
  budget?: string;                // 启动闸门确认的成本预估 / 轮次边界
  pauseReason?: "user_abort" | "model_error" | "audit_error" | "audit_failed_3x" | "no_progress";
  auditErrorScope?: "phase" | "goal"; // audit_error 来源范围；resume 只重置该范围候选
  rejectedCount?: number;         // 终审连续不过计数，×3 转 paused(audit_failed_3x)
  phaseFeedbackById?: Record<string, PhaseCheckFeedback>; // 阶段建检未通过的原始报告（按 phaseId）
  finalFeedback?: FinalCheckFeedback; // 终审未通过的原始报告（resume 后保留）
  auditorCandidates?: { // 当前 goal 内按审核范围隔离的候选健康状态
    phase?: { selectedModelId?: string; failedModelIds?: string[] };
    goal?: { selectedModelId?: string; failedModelIds?: string[] };
  };
}

interface AcceptanceCriterion {
  criterion: string;               // LLM 可独立判定的完成条件
  evidence: string;                // 可由受限工具/命令/工件独立复验的证据
}

interface CheckFeedback {
  report: string;                 // 检查 agent 原始报告，不做 summary / 压缩
  createdAt: number;
}

interface PhaseCheckFeedback extends CheckFeedback {
  phaseId: number;
}

interface FinalCheckFeedback extends CheckFeedback {
  rejectedCount: number;
}

interface TaskPlan {
  phases: Phase[];
  nextId: number;
}

interface Phase {
  id: number;
  subject: string;               // 阶段性目标
  description?: string;
  acceptanceCriteria?: AcceptanceCriterion[]; // 新 proposal 必填；旧 session 缺失时不迁移
  status: "pending" | "in_progress" | "done" | "blocked";  // 中间态可由 task 聚合；done 仅由 dgoal_check 写入，空 phase 可直接 blocked
  tasks: Task[];
  blockedReason?: string;         // 空 phase 直接 blocked 时的原因
}

interface Task {
  id: number;
  subject: string;
  description?: string;
  activeForm?: string;           // in_progress 时浮层显示
  status: "pending" | "in_progress" | "done" | "blocked";
  blockedBy?: number[];          // 依赖图（可跨 phase，深度不限 → 涌现分解）
  evidence?: string;             // 完成证据：可被 dgoal_check 独立复验的命令/文件/测试结果，非自述
  blockedReason?: string;        // blocked 时必带
}
```

### phase ID 分配与旧 session 兼容

新 proposal 转 `TaskPlan` 时，先为全部 phase 预分配连续真实 ID `1..N`，再从 `N+1` 起为 task 分配全局唯一 ID；phase 内初始 `blockedBy` 仍按局部 task 序号解析到这些 task ID。

加载既有 session 时不迁移、不重编号 `TaskPlan.phases[].id`、`nextId`、`phaseFeedbackById`、历史建检报告或 `task.blockedBy`。因此旧 plan 可保留非连续 phase ID（如 `#1/#4/#8`），调用方应使用真实 `phaseId` 或 1-based `phaseNumber`，不得将两者静默互猜。

### feedback 生命周期（v0.5.2）

- `phaseFeedbackById[phaseId]`：`dgoal_check` 明确未通过时写入；该 phase 通过后清除；`auditor_error` / aborted / 无有效结论不写。
- `finalFeedback`：`dgoal_done` 明确未通过时写入；终审通过或 goal 清除后自然结束；`paused(audit_failed_3x)` → `/dgoal resume` 时保留。
- feedback 是 **agent-facing 修复输入**：原文会注入后续 prompt 的 `<check_feedback>`，但**不展示**在 `/dgoal s` modal、aboveEditor 浮层或底部状态栏。

### 审核器选模配置

独立审核子进程（`dgoal_check` / `dgoal_done` 共享的 `runIsolatedCheck()`）默认继承当前会话模型。首次实际审核时，只有全局和受信任项目级的 `pi-dgoal.json` 文件都不存在，dgoal 才会原子创建全局模板：

```json
// ~/.pi/agent/pi-dgoal.json
{
  "$comment": "将各候选列表按回退顺序填为 provider/model[:thinking]；保持 null 则继承当前会话模型。",
  "phaseAuditorModels": null,
  "goalAuditorModels": null
}
```

阶段建检与目标终审是两个独立审核范围，可分别配置最多 3 个有序模型候选与 Pi 思考等级；具体项是持久化专用设置，不随主会话换模或 Pi 重载变化：

```json
// ~/.pi/agent/pi-dgoal.json 或 .pi/pi-dgoal.json
{
  "phaseAuditorModels": ["openai-codex/gpt-5.6-sol:medium", "minimax-cn/MiniMax-M3:high"],
  "goalAuditorModels": ["openai-codex/gpt-5.6-sol:xhigh", "minimax-cn/MiniMax-M3:high"]
}
```

旧单值 `phaseAuditorModel`、`goalAuditorModel` 与共享 `auditorModel` 字段继续兼容，不自动改写用户文件。

解析顺序：

1. 项目 `.pi/pi-dgoal.json`（候选链整体使用）
2. 全局 `~/.pi/agent/pi-dgoal.json`
3. 配置没有可用候选时才使用当前 `ctx.model`

补充约束：

- 每个候选在一次审核中最多调用一次；技术/协议错误或缺终止标记的部分输出才切换下一个候选，不做同模型三次续审。
- 当前 goal、同一审核范围内产生有效结论的候选会写入 `auditorCandidates` 并优先复用；phase 与 goal 两个范围彼此隔离。
- `/dgoal resume` 从 `audit_error` 恢复时只清除 `auditErrorScope` 对应范围的候选故障状态；旧 goal 缺失范围字段时兼容为全量清除。业务 `REJECTED` 不清除健康候选，也不触发换模。
- 项目级配置只有在 `ctx.isProjectTrusted()` 为真时才会读取。
- `phaseAuditorModels` 只供 `dgoal_check` 阶段建检使用；`goalAuditorModels` 只供 `dgoal_done` 触发的目标终审使用。
- 值支持 `provider/model[:thinking]`；末尾 Pi 标准等级（`off`/`minimal`/`low`/`medium`/`high`/`xhigh`/`max`）用于选 thinking。模型 ID 本身可含 `/` 或 `:`（custom/gateway 路径或 tag），dgoal 只拒绝畸形 provider、内嵌空白/控制字符和空路径/tag 段。
- 同一来源内复数字段 > 对应单值字段 > 共享 `auditorModel`；先比较来源（受信任项目级 > 全局）且不混合两个来源的候选链。复数字段 `null` 显式继承当前会话模型并阻断继续降级；空数组非法，非法/重复项按下标告警，只保留前 3 个合法首次项。
- 预检通过与审核 child 同隔离边界的 Pi `get_available_models` 结构化结果完成：先完整匹配模型 ID，再识别末尾标准 thinking 后缀。成功结果在当前 Pi 进程缓存、`/reload` 清除；预检失败保留候选交由运行时判定，预检确认不可用的项才跳过并继续配置降级。
- 非法值、坏 JSON、读文件失败或预检确认全部不可用都会继续走剩余配置优先级，最终可显式降级当前 `ctx.model`；这不等同于运行时候选耗尽，后者必须 `audit_error` 暂停。
- 模板仅在全局和受信任项目级配置文件均不存在时创建，使用独占写入，绝不覆盖已有文件；已有但不可读或格式错误的文件只会告警并降级；当前范围未配置或为 `null` 且无其他配置问题时，每个 Pi 进程首次审核提示一次候选链配置入口；存在配置问题时只发对应告警。
- 该配置只影响独立审核子进程选模与思考等级，不改变主执行线程模型；决策演进见 ADR 0013、ADR 0014 与 ADR 0015。

### 冻结验收契约

新 proposal 必须同时提供 goal 级 `acceptanceCriteria` 与每个 phase 的 `acceptanceCriteria`。每项由 `criterion`（完成条件）和 `evidence`（可独立复验的证据）组成；任一项为空、非字符串或混入脏条件时不得静默删除，启动闸门直接拒绝提案。`userReviewItems` 只记录完成后需要用户实际使用、观察或主观判断的事项，不替代独立验收条件，也不阻塞 phase / goal done。

phase 建检和 goal 终审只允许依据启动闸门冻结的 `acceptanceCriteria`、task 及其 evidence、直接相关的代码/文档事实判定。所有进入审核 prompt 的动态字段都经过 `escapeXml()`，防止动态内容闭合 XML 标签边界（`escapeXml()` 只防止标签闭合，不消除普通文本形式的 prompt injection）。审核器不得从 `AGENTS.md`、README 或自身判断临时增加人工体验完成门；这类发现只能作为非阻塞的用户复核建议，最终合并进完成回复。重审时上一轮反馈中的越权人工体验完成门不继续作为完成门，只按冻结契约重审。

旧 session 没有这些字段时按缺失兼容加载，不迁移、不重编号；旧 goal 继续沿用其原有 verification/task evidence 审核路径；最旧 goal 若连 verification 也缺失，则终审使用本次 `dgoal_done` 的 verification 与 task evidence 作为兼容依据，不凭空新增人工完成门。阶段审核拒绝报告中的非阻塞用户复核建议同样合并到 `userReviewItems`，最终统一交付。

### evidence 形态

task 级 evidence 必须是**可被 dgoal_check 独立复验**的形态（建检模式启示）：命令（如"跑 `npm test auth` 全过"）、文件、测试结果——不是 agent 的文字自述（后者 agent 能编，前者 check 能复验）。

goal 级 `verification` 是跨 phase 的全局说明，与 task 级 evidence 互补。

### verification 必填（ADR 0007）

`dgoal_propose` 提交的 `verification` 与 goal/phase `acceptanceCriteria` 是**必填**，不是可选：没有可验收完成口的 goal 不应进入启动闸门。

- **角色**：新 goal 的冻结完成门只有 `acceptanceCriteria`；`verification` 是 goal 级验收说明（帮助 agent 和用户理解完成标准），不单独作为终审完成门。旧 session 缺少 `acceptanceCriteria` 时，终审兼容沿用 `verification`。
- **来源**：可参考 `contextSummary` 里的“验收标准”，但必须由主代理在 `dgoal_propose` 中显式提交（不做静默自动回填）。
- **工具层校验**：`validateProposalInput` 校验 `verification`、goal `acceptanceCriteria` 和每个 phase 的 `acceptanceCriteria` 非空（trim 后）；任一缺失或无有效 `criterion/evidence` 时在工具层直接拒绝，不推给用户确认。
- **证据形态拦截**：工具层用轻量 evidence 形态启发式拒绝明显不可独立复验的证据，例如纯开发者声明、模型判断、完成说明、人工签字/认可等；可接受的 evidence 应包含命令、测试输出、文件/路径、URL/API 响应、日志或截图等可由受限工具复查的锚点。
- **人工体验检测**：`dgoal_propose` 先执行结构校验，再调用当前会话 LLM 做计划级语义预审；人工动作、自述/主观完成条件即使搭配命令、路径或测试输出，也必须在写入 `pendingProposal` 前被拒绝或改写到 `userReviewItems`。`rewrite` 必须为每个被移除或改变的原始 criterion 返回精确 `sourceCriterion` → `userReviewItem` 迁移映射；缺失或无关映射也 fail-closed。预审异常、用户中断或无当前模型时 fail-closed，不留下半激活 proposal。
- **提示与终审边界**：`buildProposePrompt` 的第 5 条仍负责引导主代理提交前二次复核；审核器两级 system prompt 的“人工条件兜底”只复核已经冻结的契约，不能替代启动前语义预审，也不能在运行期扩容完成门。
- **空话拦截**：工具层的 evidence 形态启发式只做保守结构拦截，不继续扩展关键词；完整语义判断由启动前 LLM 预审负责，冻结后的 phase/goal 审核只按已冻结 acceptanceCriteria 判定。
- **用户确认**：`verification` 在启动闸门确认框里与 goal / readiness / phases 并列展示（标注为验收说明，完成门以 `acceptanceCriteria` 为准）；用户只做这一次总确认。

### proposal 就绪度自检（L0-L3）

`dgoal_propose` 除了校验 `objective` / `verification` / `acceptanceCriteria` / `phases` 非空，还会基于提案字段做 **plan 级就绪度评估**：

- `objective` + `verification` + goal/phase `acceptanceCriteria` + `phases` 齐备，且 dgoal 自带独立 verifier（`dgoal_check` / `dgoal_done`）时，提案至少到 **L2**。
- 若额外补齐 `nonGoals`（不做什么）、`guardrails`（高风险边界）、`budget`（成本预估 / 轮次边界），则可到 **L3**。
- 缺什么不在工具层硬拒（除 `objective` / `verification` / `acceptanceCriteria` / `phases` 这四类硬门槛外），而是在启动闸门确认 UI 里**显式展示等级与缺口提示**，尤其要暴露 `nonGoals` 缺口。

这套评估是**计划提交前的就绪度评估**，不是运行中的自主权档位，也不引入 Cobus 式项目级 `loop-audit` / badge。

### dgoal_done 可核对完成文本（降理解债）

`dgoal_done` 除了 `summary`（改了什么 + 为什么）和 `verification`（怎么验证）两个必填字段，还支持两个可选字段，以及启动闸门冻结的 `userReviewItems`：

- `whatChanged`：主要改动点清单（文件/模块 + 关键变化），方便用户快速核对。
- `userReview`：仍需用户亲自核对的点——尤其是 agent 无法自验、需要人确认理解的部分（意图债）。
- `userReviewItems`：启动计划已声明的完成后体验/视觉复核项；终审通过后与 agent 补充项、审核器发现的非阻塞建议合并输出。

完成回复信号（`buildCompletionReplySignal`）会把这些字段结构化成可核对文本，而不是笼统宣布“已完成”。终审任务（`buildAuditorTask`）也会收到 `whatChanged`，用来核验改动是否与声称一致；`userReview` 只作参考，不在终审范围内（意图债由人还，不由 agent 审）。

对应三债模型：理解债靠 agent 解释恢复（`whatChanged` / `summary` / `verification`），意图债 agent 还不了（`userReview` 提示人核对）。

## 持久化

vNext 使用新的 `dgoal-goal-vnext` custom entry（`api.appendEntry("dgoal-goal-vnext", { goal })` + `loadGoal` 按 customType 过滤 last-write-wins）。旧 `dgoal-state` entry 被故意忽略，不迁移、不恢复、不自动清除；升级后用户必须重新 `/dgoal`。三层内容（goal/phase/task）与终审修复账本都持久化——AI 可见 + compact 后可恢复。

**工具名不是持久化键**：dgoal 恢复靠 custom entry 的 customType，与工具名无关（区别于 rpiv-todo 靠 toolResult.details + 工具名 replay）。故 `loop_complete`→`dgoal_done` 改名安全。终审修复仍是 goal 的展示投影，不创建 goal 级 task 或额外 phase。
