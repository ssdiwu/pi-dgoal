# 12 - 工具、命令与数据模型

> agent 与 dgoal 状态机交互的工具、用户侧命令、持久化数据模型。决策依据见 `../决策档案/0002-启动闸门与工具回调提交计划.md`、`../决策档案/0003-dgoal-check单工具承载自检与终审.md`，术语见 `术语表.md`。

## 工具（统一 `dgoal_` 前缀）

agent 与 dgoal 交互统一用工具，命名 `dgoal_` 前缀对齐 `/dgoal` 命令。原 `loop_complete` 已改名 `dgoal_done`。

| 工具 | 职责 | 操作重量 |
|---|---|---|
| `dgoal_propose` | 启动闸门提交 goal + phases（+ 可选初始 task），触发确认 UI | 中（触发 UI） |
| `dgoal_plan` | task/phase 的 CRUD：建/改状态/依赖/标 task done/标空 phase blocked | 轻（纯本地快操作） |
| `dgoal_check` | phase done 的唯一入口（阶段建检门）；即使是最后一个 phase，也只负责该 phase 的独立建检 | 重（spawn 独立子进程） |
| `dgoal_done` | 在所有 phase 都已通过后，声明 goal 完成并触发 goal 级终审；唯一的关 goal 入口 | 重（触发终审） |

### 关键不变式

- **dgoal_plan 不能标 phase done**——只能标 task。phase done 必须走 dgoal_check（建检不可绕过）。
- **dgoal_check 是 phase done 唯一入口**。颗粒度：task 级不 check（太细），phase 级阶段建检；即使最后一个 phase 通过，也还没完成 goal。
- **dgoal_done 是 goal 级终审唯一入口**：只有当所有 phase 都已通过 `dgoal_check` 后，才允许调用 `dgoal_done` 触发终审并关闭 goal。

## 命令（用户侧）

`/dgoal` 命令 + 子命令，用户控制，不是 agent 工具。每个子命令支持全拼 + 单字母别名（`s/p/r/c`）：

```
/dgoal <goal>       显式目标启动（触发启动闸门）
/dgoal              裸启动：承接当前前文共识进入启动闸门；无前文时提示改用 /dgoal <goal>
/dgoal status | s   以 center modal 查看完整 plan 状态（goal / phase / task）
/dgoal pause  | p   暂停自动续跑，保留 goal
/dgoal resume | r   恢复暂停的 goal
/dgoal clear  | c   清除当前 goal
```

**路由约定（v0.5.2）**：裸 `/dgoal`（空 args）不再等于 `status`；它走启动闸门的“路径 B”（承接前文，由主代理用 `dgoal_propose` 归纳 objective）。查看状态统一用显式 `/dgoal s`。

## 数据模型

```ts
interface LoopGoal {
  id: string;
  objective: string;              // goal 文本简述；裸 /dgoal 启动时 pending 期间可暂为承接占位，确认后被 proposal.objective 覆盖
  status: "pending" | "active" | "rejected" | "paused" | "done";
  startedAt: number;
  updatedAt: number;
  iteration: number;
  contextSummary?: string;        // 启动背景固化（只补背景，不替代 objective / verification）
  plan?: TaskPlan;                // phase + task 两层
  verification?: string;          // goal 级验证：跨 phase 全局说明
  pauseReason?: "user_abort" | "model_error" | "audit_error" | "audit_failed_3x";
  rejectedCount?: number;         // 终审连续不过计数，×3 转 paused(audit_failed_3x)
  phaseFeedbackById?: Record<string, PhaseCheckFeedback>; // 阶段建检未通过的原始报告（按 phaseId）
  finalFeedback?: FinalCheckFeedback; // 终审未通过的原始报告（resume 后保留）
}

interface CheckFeedback {
  report: string;                 // 检查 agent 原始报告，不做 summary / 压缩
  createdAt: number;
}

interface PhaseCheckFeedback extends CheckFeedback {
  phaseId: number;
}

interface FinalCheckFeedback extends CheckFeedback {
  rejectedCount: number;
}

interface TaskPlan {
  phases: Phase[];
  nextId: number;
}

interface Phase {
  id: number;
  subject: string;               // 阶段性目标
  description?: string;
  status: "pending" | "in_progress" | "done" | "blocked";  // 由其下 task 聚合，空 phase 可直接 blocked
  tasks: Task[];
  blockedReason?: string;         // 空 phase 直接 blocked 时的原因
}

interface Task {
  id: number;
  subject: string;
  description?: string;
  activeForm?: string;           // in_progress 时浮层显示
  status: "pending" | "in_progress" | "done" | "blocked";
  blockedBy?: number[];          // 依赖图（可跨 phase，深度不限 → 涌现分解）
  evidence?: string;             // 完成证据：可被 dgoal_check 独立复验的命令/文件/测试结果，非自述
  blockedReason?: string;        // blocked 时必带
}
```

### feedback 生命周期（v0.5.2）

- `phaseFeedbackById[phaseId]`：`dgoal_check` 明确未通过时写入；该 phase 通过后清除；`auditor_error` / aborted / 无有效结论不写。
- `finalFeedback`：`dgoal_done` 明确未通过时写入；终审通过或 goal 清除后自然结束；`paused(audit_failed_3x)` → `/dgoal resume` 时保留。
- feedback 是 **agent-facing 修复输入**：原文会注入后续 prompt 的 `<check_feedback>`，但**不展示**在 `/dgoal s` modal、aboveEditor 浮层或底部状态栏。

### evidence 形态

task 级 evidence 必须是**可被 dgoal_check 独立复验**的形态（建检模式启示）：命令（如"跑 `npm test auth` 全过"）、文件、测试结果——不是 agent 的文字自述（后者 agent 能编，前者 check 能复验）。

goal 级 `verification` 是跨 phase 的全局说明，与 task 级 evidence 互补。

### verification 必填（ADR 0007）

`dgoal_propose` 提交的 `verification` 是**必填**，不是可选：没有可验收完成口的 goal 不应进入启动闸门。

- **来源**：可参考 `contextSummary` 里的“验收标准”，但必须由主代理在 `dgoal_propose` 中显式提交（不做静默自动回填）。
- **工具层校验**：`validateProposalInput` 校验 `verification` 非空（trim 后）；空则在工具层直接拒绝（`details.error: "no verification"`），不推给用户确认。
- **空话拦截**：工具层只做“非空”，不做“是不是空话”的启发式判断；空话靠 `buildProposePrompt` 文案引导 + 终审 auditer 兜底。
- **用户确认**：`verification` 在启动闸门确认框里与 goal / phases 并列展示；用户只做这一次总确认。

## 持久化

复用现有 `dgoal-state` custom entry（`api.appendEntry("dgoal-state", { goal })` + `loadGoal` 按 customType 过滤 last-write-wins）。不另起载体。三层内容（goal/phase/task）都持久化——AI 可见 + compact 后可恢复。

**工具名不是持久化键**：dgoal 恢复靠 custom entry 的 customType，与工具名无关（区别于 rpiv-todo 靠 toolResult.details + 工具名 replay）。故 `loop_complete`→`dgoal_done` 改名安全。
