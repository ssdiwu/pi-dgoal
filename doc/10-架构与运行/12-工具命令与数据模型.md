# 12 - 工具、命令与数据模型

> 当前权威设计：ADR 0038。公共 agent 工具固定为八个两词名称。

## 工具矩阵

| 工具 | 主要参数 | 作用 |
|---|---|---|
| `task_plan` | `objective`, `tasks[]` | 直接建立或原子替换 Task Plan |
| `phase_plan` | goal 契约, `phases[]` | 提交 Phase Plan，进入语义预审与用户确认 |
| `goal_plan` | goal 契约, 带 phase 契约的 `phases[]` | 提交 Goal Plan，进入语义预审与用户确认 |
| `plan_create` | task 字段 + Phase/Goal Plan 的 phase 定位 | 只新增 task |
| `plan_read` | `target` + 可选 `id`/`phaseNumber` | 读取 Plan、phase 或 task；Task Plan 只暴露 plan/task |
| `plan_update` | `target`, `status` 及目标字段 | 唯一 agent 执行状态写工具 |
| `phase_check` | `phaseId` 或 `phaseNumber` | Goal Plan 的 phase 独立审核；只写 CheckRecord |
| `goal_check` | `summary`, `verification` + 可选变更/风险说明 | Phase/Goal Plan 的整体独立审核；只写 CheckRecord |

### `plan_update` 目标

- `target=task`：更新 subject/description/activeForm/status/evidence/blockedReason/依赖。
- `target=phase,status=done`：只允许 Phase/Goal Plan；task 必须全部 done，Goal Plan 还要求当前 revision 的 phase approval。
- `target=goal,status=done`：按 Plan 类型检查完整完成链并关闭 Plan。
- `target=goal,status=paused,reason=...`：agent 主动暂停；reason 必须具体且有界。

### 关键不变量

1. `plan_create` 不创建 phase；Phase/Goal Plan 的 phase 在启动时冻结。
2. Task Plan 的隐藏 phase 不接受 phase 定位，也不出现在 `plan_read`。
3. `phase_check` / `goal_check` 不写 done。
4. `plan_update` 不能绕过与 `planType` 对应的 check 守卫。
5. `phaseId` 与 `phaseNumber` 二选一，同时提供会被拒绝。
6. task 只能依赖同一或更早 phase 的 task；后续 phase 依赖会被拒绝，避免顺序死锁。
7. task 进入 done 必须带可复验 evidence；blocked 必须带 blockedReason，且不算完成。
8. plan 写操作同时清理旧审核检查点；check 运行期间 revision 若变化或原 Plan 被清除，结果丢弃，不写回新 Plan。
9. paused 下 `plan_read` 仍可用；写操作、check 与完成收口需先 resume。

## 用户命令

```text
/dgoal <objective>   启动显式 dgoal，由 agent 推荐 Phase Plan 或 Goal Plan
/dgoal               承接前文启动显式 dgoal
/dgoal status | s    打开完整状态 Modal
/dgoal pause  | p    用户暂停
/dgoal resume | r    恢复
/dgoal clear  | c    清除当前 Plan
/dgoal help   | h    解释当前能力与状态
```

真实用户也可用明确祈使句要求“使用/启动 dgoal”，复用同一显式启动闸门。Task Plan 无需用户命令，由 agent 在普通明确多步任务中按需建立。

## 核心数据模型

```ts
type PlanType = "task" | "phase" | "goal";
type GoalStatus = "pending" | "active" | "paused" | "done";
type ItemStatus = "pending" | "in_progress" | "done" | "blocked";
type CheckStatus = "approved" | "rejected" | "audit_error";

type AcceptanceCriterion = {
  criterion: string;
  evidence: string;
};

type CheckRecord = {
  status: CheckStatus;
  report: string;
  checkedAt: number;
  revision: number;
};

type Task = {
  id: number;
  subject: string;
  description?: string;
  activeForm?: string;
  status: ItemStatus;
  blockedBy?: number[];
  blockedReason?: string;
  evidence?: string;
};

type Phase = {
  id: number;
  subject: string;
  description?: string;
  acceptanceCriteria?: AcceptanceCriterion[]; // Goal Plan 必需
  status: ItemStatus;
  tasks: Task[];
  check?: CheckRecord;                         // 仅 Goal Plan 使用
};

type TaskPlan = {
  phases: Phase[];
  nextId: number;
  revision: number;
};

type GoalState = {
  id: string;
  objective: string;
  planType: PlanType;
  status: GoalStatus;
  plan: TaskPlan;
  verification?: string;                       // Phase/Goal Plan 必需
  acceptanceCriteria?: AcceptanceCriterion[]; // Phase/Goal Plan 必需
  goalCheck?: CheckRecord;
  contextSummary?: string;
  nonGoals?: string[];
  guardrails?: string[];
  userReviewItems?: string[];
  pauseReason?: string;
  pauseReasonDetail?: string;
  startedAt: number;
  updatedAt: number;
  iteration: number;
};
```

`dgoal-plan-v1` 不读取旧持久态，也不生成 verification policy、runtime budget 或 implicit start 字段。

## Proposal 语义预审

`phase_plan` / `goal_plan` 复用启动闸门语义预审：

- 代码层校验结构、状态、Plan 类型与用户授权。
- 当前会话模型保留独立验收条件，把主观体验迁移到 `userReviewItems`，真实人工 blocker 才拒绝。
- 预审支持 `approved / rewritten / rejected / technical_error`，技术错误不伪装成计划内容问题。
- 用户确认后才把 pending proposal 激活为当前 Plan。

Task Plan 不运行 proposal 预审；它也不包含冻结 acceptance contract。

## 独立审核器

`phase_check` / `goal_check` 共用隔离审核运行时。默认继承当前模型，可配置：

```json
{
  "phaseAuditorModels": null,
  "goalAuditorModels": null,
  "proposalSemanticReviewIdleTimeoutSeconds": 60
}
```

也可把前两项设为最多 3 个 `provider/model[:thinking]` 候选。受信任项目级 `.pi/pi-dgoal.json` 整体优先于全局 `~/.pi/agent/pi-dgoal.json`；复数字段优先于旧单值兼容字段。业务 rejected 不切模型，只有技术/协议异常切候选。

审核进程的 model idle timeout 为 180 秒，工具执行 idle timeout 为 1800 秒；phase/goal 的固定审核技术超时上限分别为 900/1800 秒，并由同一次审核的候选共享。这些是不可配置的技术熔断，不是用户运行预算策略。

## 持久化

新运行时只读取 `dgoal-plan-v1` custom entry，last-write-wins：

```ts
api.appendEntry("dgoal-plan-v1", { goal });
```

旧 `dgoal-state` 与 `dgoal-goal-vnext` 被故意忽略，不迁移、不恢复。`session_start`、`session_tree`、`session_compact` 会重同步；八个工具在内存状态缺失时可从当前 session 惰性恢复。

## Evidence 与审核边界

- task 进入 done 必须带 evidence；它应是可复验命令、文件、测试结果或外部可观察事实，不写主 agent 自述。
- phase/goal auditor 只核启动时冻结的 acceptance contract 和直接相关回归，不能临时增加人工体验完成门。
- `CheckRecord.revision` 必须与当前 `plan.revision` 一致才可作为完成守卫。
- UI 更新失败只能降级提示，不能改变已经落盘的状态事实。
