# 12 - 工具、命令与数据模型

> 当前权威设计：ADR 0038 + ADR 0042。公共 agent 工具固定为八个两词名称，goal / 可见 phase / task 使用必填 Description。

## 工具矩阵

| 工具 | 主要参数 | 作用 |
|---|---|---|
| `task_plan` | `objective`, `description`, `tasks[]`（各含 description） | 直接建立或原子替换 Task Plan |
| `phase_plan` | goal 契约与 description, `phases[]`（各含 description） | 提交 Phase Plan，进入语义预审与用户确认 |
| `goal_plan` | goal 契约与 description, 带 phase 契约/description 的 `phases[]` | 提交 Goal Plan，进入语义预审与用户确认 |
| `plan_create` | task subject/description + Phase/Goal Plan 的 phase 定位 | 只新增 task |
| `plan_read` | `target` + 可选 `id`/`phaseNumber` | 读取 Plan、phase 或 task；聚合/单项结果共享当前 frontier 的直接原因与下一合法动作，并只投影最新适用审核信息；不暴露原始 JSON；Task Plan 只暴露 plan/task |
| `plan_update` | `target`, `status` 及目标字段 | 唯一 agent 执行状态写工具 |
| `phase_check` | `phaseId` 或 `phaseNumber` | Goal Plan 的 phase 独立审核；只写 CheckRecord |
| `goal_check` | `summary`, `verification` + 可选变更/风险说明 | Phase/Goal Plan 的整体独立审核；只写 CheckRecord |

### `plan_update` 目标

- `target=task`：更新 subject/description/status/evidence/blockedReason/依赖；Task Plan 的最后一个 task 带 evidence 进入 done 时自动关闭 goal。
- `target=phase,status=done`：只允许 Phase/Goal Plan；task 必须全部 done，Goal Plan 还要求当前 revision 的 phase approval。
- `target=goal,status=done`：Phase/Goal Plan 按完整完成链关闭 Plan；Task Plan 正常由最后一个 task 自动收口。
- `target=goal,status=paused,reason=...`：agent 主动暂停；reason 必须具体且有界。

### 关键不变量

1. `plan_create` 不创建 phase；Phase/Goal Plan 的 phase 在启动时冻结。
2. Task Plan 的隐藏 phase 不接受 phase 定位，也不出现在 `plan_read`。phase 与 task 使用独立 ID namespace，二者都从 `1` 开始；task ID 在整个 Plan 内唯一且不按 phase 重置。
3. `phase_check` / `goal_check` 不写 done。
4. `plan_update` 不能绕过与 `planType` 对应的 check 守卫。
5. `phaseId` 与 `phaseNumber` 二选一，同时提供会被拒绝。
6. task 只能依赖同一或更早 phase 的 task；后续 phase 依赖会被拒绝，避免顺序死锁。
7. task 进入 done 必须带可复验 evidence；blocked 必须带 blockedReason，且不算完成。
8. plan 写操作同时清理旧审核检查点；check 运行期间 revision 若变化或原 Plan 被清除，结果丢弃，不写回新 Plan。
9. paused 下 `plan_read` 仍可用；写操作、check 与完成收口需先 resume。
10. goal、用户可见 phase 与 task 的 description 必须非空；Goal description 在 Phase/Goal Plan 确认后冻结，phase/task description 可由显式 `plan_update` 修订并使旧审核失效。Description 指导执行，但不单独构成 auditor 完成门。

## 用户命令

```text
/dgoal <objective>   启动显式 dgoal，由 agent 推荐 Phase Plan 或 Goal Plan
/dgoal               承接前文启动显式 dgoal
/dgoal status | s    打开完整状态 Modal
/dgoal pause  | p    用户暂停
/dgoal resume | r    恢复
/dgoal clear  | c    清除当前 Plan
/dgoal help   | h    解释当前能力与状态
```

真实用户也可用明确祈使句要求“使用/启动 dgoal”，复用同一显式启动闸门。Task Plan 无需用户命令，由 agent 在普通明确多步任务中按需建立。

## 核心数据模型

```ts
type PlanType = "task" | "phase" | "goal";
type GoalStatus = "pending" | "active" | "paused" | "done";
type ItemStatus = "pending" | "in_progress" | "done" | "blocked";
type CheckStatus = "approved" | "rejected" | "audit_error";

type AcceptanceCriterion = {
  criterion: string;
  evidence: string;
};

type CheckRecord = {
  status: CheckStatus;
  report: string;
  checkedAt: number;
  revision: number;
};

type Task = {
  id: number;
  subject: string;
  description: string;
  status: ItemStatus;
  blockedBy?: number[];
  blockedReason?: string;
  evidence?: string;
};

type Phase = {
  id: number;
  subject: string;
  description: string;
  acceptanceCriteria?: AcceptanceCriterion[]; // Goal Plan 必需
  status: ItemStatus;
  tasks: Task[];
  check?: CheckRecord;                         // 仅 Goal Plan 使用
};

type TaskPlan = {
  phases: Phase[];
  nextId: number;  // 下一个 plan-global task ID；不分配 phase ID
  revision: number;
};

type GoalState = {
  id: string;
  objective: string;
  description: string;
  planType: PlanType;
  status: GoalStatus;
  plan: TaskPlan;
  verification?: string;                       // Phase/Goal Plan 必需
  acceptanceCriteria?: AcceptanceCriterion[]; // Phase/Goal Plan 必需
  goalCheck?: CheckRecord;
  nonGoals?: string[];
  guardrails?: string[];
  userReviewItems?: string[];
  pauseReason?: string;
  pauseReasonDetail?: string;
  startedAt: number;
  updatedAt: number;
  iteration: number;
};
```

`dgoal-plan-v2` 不读取 `dgoal-plan-v1` 及更早持久态，也不生成 `contextSummary`、verification policy、runtime budget 或 implicit start 字段。Task Plan 的隐藏 phase 不单独生成用户说明；内部 description 复用 goal description，且不向工具或 TUI 暴露。

## Proposal 质量检查与语义预审

`phase_plan` / `goal_plan` 提交前，主 agent 固定做一次精简质量检查：

- 核对端到端可观察结果是否有 phase/task 承载；
- 适用时核对核心对象/状态生命周期、生产者—消费者真实调用链与风险相称的失败/恢复路径；
- 核对 verification/acceptanceCriteria 与上述 Plan 路径一致；
- 简单目标可判定某项不适用；发现问题直接修正，不输出自检报告，也不新增模型调用、状态或 hard gate。

随后复用启动闸门语义预审：

- 代码层校验结构、状态、Plan 类型与用户授权。
- 当前会话模型保留可由未来审核器在允许工具范围内独立取得证据的验收条件，把主观体验迁移到 `userReviewItems`，真实人工 blocker 才拒绝。
- 依赖未来审核器不可取得证据的条件（例如没有可导出不可变审计记录支撑的“过去从未发生”断言、父会话记录、agent/worker/user 记忆或未文档化宿主遥测）不是独立验收，也不能迁移为 `userReviewItems`；语义预审必须拒绝，并要求删除绝对历史主张或收缩为可观察、可独立复验的范围。
- `buildProposePrompt` 与语义预审共享该证据边界：`guardrails` / `nonGoals` 是执行约束而非完成门；主 agent 提交前必须一并清除所有不可准入 condition。reject 可选带逐条件 `issues`（原 criterion、分类、原因、改写方向），用于一次反馈全部问题；旧的单 `reason` 拒绝保持兼容。
- 预审支持 `approved / rewritten / rejected / technical_error`，技术错误不伪装成计划内容问题。
- 用户确认后才把 pending proposal 激活为当前 Plan。

Task Plan 不运行上述 audited Plan 质量检查或 proposal 预审；它继续只做相关性、必要性、依赖和证据路径的轻量自检，也不包含冻结 acceptance contract。

## 只读 frontier 与最新审核投影

`plan_read` 和 `/dgoal s` 调用同一纯派生逻辑，不写状态、不增加 revision：

- 只解释当前 frontier，不列出未来 phase 的全部未满足条件；
- 原因覆盖当前 task、未完成 `blockedBy`、显式 blocker、缺 evidence、phase/goal check rejected，以及缺当前 revision approval；
- 下一合法动作只指向现有 `plan_create` / `plan_update` / `phase_check` / `goal_check` 或 `/dgoal resume`；
- 最新审核投影只组合现有 `CheckRecord`、当前 phase/goal feedback、task evidence 与 `finalAuditHistory.at(-1)` 的最新完成声明；内部较早修复索引不展示；
- 不新增审核计数、耗时、attempt artifact、恢复字段或 `doctor` 类一致性修复。

## 独立审核器

`phase_check` / `goal_check` 共用隔离审核运行时。默认继承当前模型，可配置：

```json
{
  "phaseAuditorModels": null,
  "goalAuditorModels": null,
  "proposalSemanticReviewIdleTimeoutSeconds": 60
}
```

也可把前两项设为最多 3 个 `provider/model[:thinking]` 候选。受信任项目级 `.pi/pi-dgoal.json` 整体优先于全局 `~/.pi/agent/pi-dgoal.json`；复数字段优先于旧单值兼容字段。业务 rejected 不切模型，只有技术/协议异常切候选。

审核进程的 model idle timeout 为 180 秒，工具执行 idle timeout 为 1800 秒；phase/goal 的固定审核技术超时上限分别为 900/1800 秒，并由同一次审核的候选共享。这些是不可配置的技术熔断，不是用户运行预算策略。

## 持久化

新运行时只读取 `dgoal-plan-v2` custom entry，last-write-wins：

```ts
api.appendEntry("dgoal-plan-v2", { goal });
```

旧 `dgoal-state`、`dgoal-goal-vnext` 与 `dgoal-plan-v1` 被故意忽略，不迁移、不恢复。v2 load guard 会按 Plan 类型严格复验 goal/plan 的三层 description、冻结验收契约、ID/`nextId`、状态、依赖图、check 结构与已删除字段；恢复中的 `pendingProposal` 还须重新通过完整 proposal 结构校验，任一处脏数据都会使整条 entry 失效，不能越过启动闸门。`session_start`、`session_tree`、`session_compact` 会重同步，并递增仅内存的 session generation：旧分支尚在返回的审核结果不得写入新分支，已发送但尚未派发的旧 continuation 会由 input handler 丢弃；八个工具在内存状态缺失时可从当前 session 惰性恢复。

## Evidence 与审核边界

- task 进入 done 必须带 evidence；它应是可复验命令、文件、测试结果或外部可观察事实，不写主 agent 自述。
- Description 是主 agent 的权威执行说明，但不是平行 acceptance contract；phase/goal auditor 只核启动时冻结的 acceptance contract 和直接相关回归，不能只凭 description 或人工体验临时增加完成门。
- `CheckRecord.revision` 必须与当前 `plan.revision` 一致才可作为完成守卫。
- UI 更新失败只能降级提示，不能改变已经落盘的状态事实。
