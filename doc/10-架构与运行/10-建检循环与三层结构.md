# 10 - 建检循环与三层结构

> 当前权威设计：ADR 0038。术语以 `../术语表.md` 为准。

## 三档 Plan

三档 Plan 共享 goal / phase / task、持久化、TUI 与自动续跑，只改变激活边界和独立审核密度：

| Plan | 用户确认 | phase check | goal check |
|---|---:|---:|---:|
| Task Plan | 否 | 否 | 否 |
| Phase Plan | 是 | 否 | 是 |
| Goal Plan | 是 | 是 | 是 |

Task Plan 使用一个隐藏 phase 复用内部结构，但工具、prompt 和 TUI 都只暴露平铺 task。Phase/Goal Plan 的 phase 在启动时冻结，运行中只允许新增 task。

## 建检循环

建检循环是“执行与独立判卷分离”的验证约束：

1. 主 agent 按 Plan 推进并提供可复验证据。
2. `phase_check` 或 `goal_check` 在 fresh context 的隔离 Pi 子进程中核验冻结条件。
3. check 只写审核记录，不写完成状态。
4. `plan_update` 根据当前 Plan 类型和最新审核记录决定是否允许写 done。
5. rejected 时继续修复；计划一变化，旧 approved 自动失效，再次 check。

Task Plan 没有独立 check，完成守卫只验证 task 状态。Phase Plan 只有 goal 级建检。Goal Plan 同时有 phase 与 goal 建检。

## 三层内容

- **Goal**：全局方向、Plan 类型、冻结验收契约与生命周期。
- **Phase**：阶段主干；Task Plan 的 phase 是内部隐藏容器。
- **Task**：按需分解的执行单元，带状态、evidence 与 `blockedBy` 依赖。

phase 不是普通工作面分组。只有真实中间里程碑才值得放进 Goal Plan 并冻结 phase 验收条件；代码、测试、文档通常是同一 phase 下的 task。

## 双可见性轴

| 内容 | 用户 TUI | agent context |
|---|---|---|
| Goal | 可见 | 可见 |
| Phase/Goal Plan 的 Phase | 默认可见 | 可见 |
| Task | Task Plan 默认可见；Phase/Goal Plan 默认折叠 | 可见 |
| Task Plan 隐藏 Phase | 不可见 | 不可见 |

Phase/Goal Plan 的 done phase 在每轮 context 中软遗忘其 task，只留 phase 标题；持久化与 `/dgoal s` 仍保留完整细节。

## 完成链

```text
Task Plan
  task_plan
  → plan_update(task, done) × N
  → plan_update(goal, done)

Phase Plan
  phase_plan + 用户确认
  → plan_update(phase, done) × N
  → goal_check
  → plan_update(goal, done)

Goal Plan
  goal_plan + 用户确认
  → [phase_check → plan_update(phase, done)] × N
  → goal_check
  → plan_update(goal, done)
```

**不变量**：approved 只是审核事实；只有 `plan_update` 可以改变完成状态和完成显示。
