# 10 - 建检循环与三层结构

> dgoal 架构基本盘。本文件是当前实现权威。决策依据见 `adr/0006`，术语见 `术语表.md`。

## 核心原理：建检循环

dgoal 的第一性原理：**定义了 goal，定义了完成后要 check——不通过则继续干活，通过则结束。**

两个粒度，统一用 `dgoal_check`：
- **阶段建检**（小建检）：一个 phase 要标 completed 时触发，审该 phase 成果。通过则 phase 进 completed，不通过回 in_progress。
- **终审**（大建检）：最后一个 phase 标 completed 时升级为审全 goal，即 `dgoal_done` 内部走的 check。通过则 goal 进 done，不通过进 rejected。

建检循环是**心智模型**，不是显式数据结构——两实例各有状态承载（phase 状态 / goal 的 rejected）。

## 范式定位

dgoal 是 goal 范式（持久 goal + check 审计 + Stop 语义），不是 plan mode。Task Plan 是给 goal 补"怎么做"的结构化脚手架，不退化成 plan mode。

> "Plans help the agent know what to do; Goal Mode defines when the agent is allowed to stop." —— dgoal 定义何时允许停（goal + check），Task Plan 补怎么做（phase + task）。

## 三层内容

```
Goal（全局目标，用户确认后冻结，loop 方向契约）
  └─ Phase（阶段性目标，状态由其下 task 聚合，TUI 默认显示）
       └─ Task（按需递归分解的细粒度执行单元，默认隐藏 Ctrl+O 展开）
```

- **Goal**：用户通过 `/dgoal` 发起，启动闸门确认后**冻结**。
- **Phase**：阶段性目标，状态由其下 task 聚合。标 phase completed 的唯一入口是 `dgoal_check`。空 phase（未拆 task）可直接标 blocked。
- **Task**：按需递归分解（ADaPT）的细粒度单元。`blockedBy` 依赖图表达分解关系，深度不限。

## 双可见性轴

| 层级 | 用户可见（TUI） | AI 可见（context） |
|---|---|---|
| Goal | 可见（heading） | 可见 |
| Phase | 可见（默认） | 可见 |
| Task | **默认隐藏**，Ctrl+O 展开 | 可见 |

用户轴避免 TUI 被海量细 task 淹没（10 行折叠预算，预留 Pi core 截断提示空间）。AI 轴保证 AI 知道每步服务哪个 phase/goal，避免局部最优全局无用。

## 可变性

- **Goal 层冻结**：用户确认的方向契约，进 loop 后不改。
- **Phase 层**：状态由 task 聚合；空 phase 可直接 blocked。
- **Task 层可调**：可增改（建/改状态/依赖）。**completed 不回退**——发现错了新建接续 task（blockedBy 指向原 task）。**blocked 可回退** in_progress（条件变化后重试）。

## 建检模式的不可绕过性

生成（推进 task/phase）与验证（check）分离：
- `dgoal_plan` 只能标 task（本地快操作），不能标 phase completed。
- 标 phase completed 必须走 `dgoal_check`（spawn 独立验收子进程审计，重操作）。
- 验证是**独立验收子进程**（fresh 上下文 + 受限核验工具），物理上拿不到主会话历史——不让学生判卷。

在工具边界上锁死了建检的不可绕过性。
