# v0.5.2 建检闭环与审核器运行时增强实施方案

> **版本归属**：0.5.2
> **状态**：grill 对齐中
> **一句话目标**：同时补齐建检反馈持久化、事件流化审核器、真实 `auditor_error` 的 3 次透明重试，以及建检闸门锁定，消灭“假超时 + 越阶段/越终审推进 + 修复现场丢失”这条断链。
>
> **决策依据**：`doc/决策档案/0011-建检反馈持久化到LoopGoal.md`（建检反馈持久化与生命周期）+ `doc/决策档案/0012-阶段建检闸门锁定与事件流化审核器.md`（事件流活性、闸门锁定、timeout 收窄）+ `doc/20-能力参考/26-Datawhale-LoopEngineering三文件循环参考.md`（失败报告原样转发的外部启发）。

## 1. Problem Statement

当前 dgoal 在 `dgoal_check`（阶段建检工具）或 `dgoal_done`（终审声明工具）未通过时，会把检查 agent 的报告放进当轮 tool result（工具结果）文本和 details（结构化结果）里；但报告没有进入 `LoopGoal` 持久状态。

这导致一个关键断点：如果主 agent 后续经历 compact（上下文压缩）、resume（恢复）或多轮 rejected（终审未过回环），它只能看到“上次终审未通过第 N/3 次”这类状态提示，不一定能看到原始失败报告。结果是 agent 可能凭记忆或摘要瞎修，违背建检循环“不让学生判卷、不过继续干”的基本盘。

用户侧不需要为每次中间建检失败介入；用户关心启动前确认、goal 完成结果，以及连续失败到 paused（暂停）这类需要介入的节点。建检反馈主要是 agent-facing（面向代理）的修复输入，但当轮 tool result 仍可以完整展示，供用户偶尔查看。

## 2. Solution

引入 `Check Feedback`（建检反馈）作为 `LoopGoal` 的短期持久字段：

- 阶段建检未通过时，把检查 agent 的原始报告完整写入 `phaseFeedbackById[phaseId].report`。
- 终审未通过时，把检查 agent 的原始报告完整写入 `finalFeedback.report`。
- 后续 system prompt（系统提示）注入独立 `<check_feedback>` block（区块），把当前有效反馈完整钉回给主 agent。
- 阶段建检通过后清除对应 phase feedback；终审通过或 goal 清除后自然清除 final feedback。
- 终审连续 3 次失败进入 `paused(audit_failed_3x)` 时保留 final feedback；`/dgoal resume` 清零 `rejectedCount`，但不清除 final feedback。

报告保存检查 agent 给出的原始报告，不生成 summary（摘要），不压缩，不做二手转述。若未来出现报告过长问题，应约束检查 agent 的报告格式，而不是在反馈链路里压缩。

## 3. User Stories

1. As a 主 agent, I want 阶段建检失败报告在下一轮仍完整可见, so that 我能基于真实失败项修正当前 phase。
2. As a 主 agent, I want 终审 rejected 后仍看到完整终审报告, so that 我能修全局问题并重新调用 `dgoal_done`。
3. As a user, I want 中间建检报告不额外刷进 `/dgoal s` modal 或 aboveEditor 浮层, so that 状态视图保持简洁，只在工具结果里偶尔查看过程。
4. As a maintainer, I want feedback 生命周期可预测, so that phase 通过后不会把旧失败报告带到后续 phase。

## 4. Implementation Decisions

### 4.1 数据模型

在 `LoopGoal` 中新增短期反馈字段：

```ts
interface CheckFeedback {
  report: string;
  createdAt: number;
}

interface PhaseCheckFeedback extends CheckFeedback {
  phaseId: number;
}

interface FinalCheckFeedback extends CheckFeedback {
  rejectedCount: number;
}
```

`LoopGoal` 增加：

```ts
phaseFeedbackById?: Record<string, PhaseCheckFeedback>;
finalFeedback?: FinalCheckFeedback;
```

说明：`phaseFeedbackById` 是按 `phaseId` 定位反馈，不表示允许多个 phase 并行失败。dgoal 仍保持阶段顺序执行；注入时只取当前 phase 的反馈。

### 4.2 写入规则

- `dgoal_check` 明确未通过：写 `phaseFeedbackById[phaseId] = { phaseId, report: result.output, createdAt }`，并保持/回退 phase 为 `in_progress`。
- `dgoal_done` 终审明确未通过：写 `finalFeedback = { report: audit.output, rejectedCount: newCount, createdAt }`，并进入 `rejected` 或 `paused(audit_failed_3x)`。
- `audit_error`（审核器自身出错）、`aborted`（中断）、`timeout`（超时）、`no output`（无输出）不是建检反馈，不写入 feedback，继续走现有异常/暂停路径。

### 4.3 清理规则

- phase 建检通过：清除 `phaseFeedbackById[phaseId]`。
- goal 终审通过：`finalizeGoal` 清空当前 goal，feedback 随之结束。
- `/dgoal clear`：清空当前 goal，feedback 随之结束。
- `/dgoal resume` from `paused(audit_failed_3x)`：保留 `finalFeedback`，只清零 `rejectedCount`。
- 新一轮同 phase 建检再次未通过：覆盖该 phase 的旧 feedback，保留最新原始报告。

### 4.4 Prompt 注入规则

新增 `buildCheckFeedbackBlock(goal)`：

- `goal.status === "active"`：定位当前未 done 的 phase，只注入该 phase 对应的 `phaseFeedbackById[phaseId]`。
- `goal.status === "rejected"` 或 `paused(audit_failed_3x)` resume 后继续修复：注入 `finalFeedback`。
- feedback block 放在 `<loop_plan>` 之后、循环规则之前。
- 使用独立 XML-like block，避免污染 plan 结构：

```text
<check_feedback type="phase" phaseId="N">
...检查 agent 原始报告...
</check_feedback>
```

或：

```text
<check_feedback type="final" rejectedCount="N">
...检查 agent 原始报告...
</check_feedback>
```

### 4.5 用户界面边界

- tool result 仍完整展示原始报告，让人和 AI 当轮都可读。
- `/dgoal s` modal 不展示原始报告。
- aboveEditor 浮层不展示原始报告。
- status bar 不展示原始报告。

### 4.6 不做自动判定器

Datawhale 推文里的 same failure（同一失败）、no progress（无进展）、regression（回归）不在本版本建自动判定器。dgoal 只负责稳定交还检查 agent 的原始报告；主 agent 基于报告自行判断是否同源失败、是否需要拆 task、是否升级 blocked（阻塞）。

## 5. Out of Scope

- 不做失败历史归档；只保留当前未解决反馈。
- 不生成 summary，不裁剪、不压缩原始报告。
- 不把建检反馈展示到 `/dgoal s` modal、aboveEditor 浮层或 status bar。
- 不做 same failure / no progress / regression 的自动识别。
- 不改独立审核子进程模型选择或容灾策略。
- 不替代项目自身测试命令；反馈只是让主 agent 更稳定地看到失败原因。

## 6. 切片拆解与验收标准

### 切片 1：数据模型与持久化兼容

**范围**：
- 扩展 `LoopGoal` 类型，新增 `phaseFeedbackById` 和 `finalFeedback`。
- 更新 goal schema 兼容校验，旧 state 没有 feedback 字段仍可加载。
- 新增反馈写入/清理的纯函数，避免散落 mutation。

**验收**：
- 旧版 `LoopGoal` fixture 能正常通过 load/validate。
- 写入 phase feedback 后 `persistGoal/loadGoal` 可恢复完整 report。
- 写入 final feedback 后 `persistGoal/loadGoal` 可恢复完整 report。

### 切片 2：阶段建检反馈写入与清理

**范围**：
- `dgoal_check` 未通过路径写入 `phaseFeedbackById[phaseId]`。
- `dgoal_check` 通过路径清理对应 phase feedback。
- 异常/中断/超时路径不写 feedback。

**验收**：
- 阶段建检 rejected 后，当前 goal 持久状态包含该 phase 原始报告。
- 同 phase 再次 rejected 会覆盖为最新报告。
- phase approved 后对应 feedback 被清理。
- audit error / aborted / no output 不写 feedback。

### 切片 3：终审反馈写入、保留与清理

**范围**：
- `dgoal_done` rejected 路径写入 `finalFeedback`。
- 3 次 rejected 进入 `paused(audit_failed_3x)` 时保留 `finalFeedback`。
- `/dgoal resume` 清零 `rejectedCount` 但保留 `finalFeedback`。
- 终审 approved 或 `/dgoal clear` 后随 goal 清空。

**验收**：
- 第 1/2 次终审 rejected 后，system 状态含完整 `finalFeedback.report`。
- 第 3 次终审 rejected 进入 paused 后，`finalFeedback.report` 仍存在。
- resume 后 `rejectedCount` 清零，`finalFeedback.report` 不丢。
- 终审通过后 current goal 清空。

### 切片 4：Prompt 注入

**范围**：
- 新增 `buildCheckFeedbackBlock(goal)`。
- `buildSystemPrompt(goal)` 在 `<loop_plan>` 后注入当前有效 feedback。
- active 只注入当前 phase feedback；rejected 注入 final feedback。
- 原始报告完整注入，不 summary、不压缩。

**验收**：
- active + 当前 phase feedback：system prompt 包含 `<check_feedback type="phase" phaseId="...">` 和完整报告。
- active + 非当前 phase feedback：不注入。
- rejected + final feedback：system prompt 包含 `<check_feedback type="final" ...>` 和完整报告。
- 无 feedback：不生成空 block。

### 切片 5：工具结果与 UI 边界回归

**范围**：
- 保持 `dgoal_check` / `dgoal_done` tool result 完整展示报告。
- 确认 `/dgoal s` modal、aboveEditor 浮层、status bar 不展示原始报告。
- 更新相关快照/纯函数测试。

**验收**：
- tool result 文本仍包含完整 `审核报告`。
- `/dgoal s` modal 渲染不包含 feedback report。
- PlanOverlay 渲染不包含 feedback report。

### 切片 6：文档同步

**范围**：

- 更新 `doc/10-架构与运行/11-状态机.md` 的 rejected/paused 反馈保留语义。
- 更新 `doc/10-架构与运行/10-建检循环与三层结构.md`，说明建检反馈是 agent-facing 修复输入。

## 7. 测试决策

优先使用现有公开接缝和纯函数测试，不测内部实现细节：

- 数据模型/持久化：复用现有 state load/persist 接缝。
- 阶段建检：扩展 `tool-execute-integration` 或相邻测试，验证 `dgoal_check` rejected/approved 对 feedback 的影响。
- 终审：扩展状态机/工具执行测试，覆盖 rejected、paused、resume 的 feedback 生命周期。
- Prompt 注入：扩展 `state-machine-and-prompt.test.ts`，断言 `<check_feedback>` block。
- UI 边界：扩展 `plan-status-dialog` / `plan-overlay-render` 相关测试，确保 report 不进入 modal/widget。

建议验证命令：

```bash
npm test
npm run test:rpc
```

涉及真实 TUI 展示较少，本版本不强制人工 TUI smoke；若改动触碰 `/dgoal s` 或 overlay 渲染实现，再补人工 smoke。

## 8. 风险与缓解

| 风险 | 缓解 |
|---|---|
| feedback 字段变成长期 audit log | 只保留当前未解决反馈；phase 通过即清，goal 清除即清 |
| 原始报告过长占 prompt | 本版不压缩；若真实出现，再约束 auditor 输出格式，不在链路里二手转述 |
| 旧 phase feedback 误注入后续 phase | 注入时按当前 phase 定位，phase 通过时清理对应 key |
| final feedback 在 resume 时被误清 | 明确 resume 只清 rejectedCount，不清 finalFeedback，补测试 |
| 用户界面被报告刷屏 | tool result 仍展示当轮报告；modal/overlay/status 不展示 |
| 自动失败判定过早引入复杂度 | 本版不建 failure history/fingerprint，交给主 agent 判断 |
