# v0.5.2 建检闭环与审核器运行时增强实施方案

> **版本归属**：0.5.2
> **状态**：grill 已对齐，待实施
> **一句话目标**：同时补齐建检反馈持久化、事件流化审核器、真实 `auditor_error` 的 3 次透明重试，以及建检闸门锁定，消灭“假超时 + 越阶段/越终审推进 + 修复现场丢失”这条断链。
>
> **决策依据**：`doc/决策档案/0011-建检反馈持久化到LoopGoal.md`（建检反馈持久化与生命周期）+ `doc/决策档案/0012-阶段建检闸门锁定与事件流化审核器.md`（事件流活性、闸门锁定、timeout 收窄）+ `doc/20-能力参考/26-Datawhale-LoopEngineering三文件循环参考.md`（失败报告原样转发的外部启发）。

## 1. Problem Statement

当前 dgoal 的建检闭环有四个断点，彼此相互放大：

1. **建检反馈不持久**：`dgoal_check`（阶段建检工具）或 `dgoal_done`（终审声明工具）未通过时，检查 agent 的原始失败报告只留在当轮 tool result（工具结果）里，没有进入 `LoopGoal`。compact（上下文压缩）/ resume（恢复）/ 多轮 rejected（终审未过回环）后，主 agent 可能只剩“上次没过”这种状态，却看不到真实失败现场。
2. **审核器协议过于依赖最终文本**：当前独立审核子进程主要消费 `text_delta` 和 `message_end`。模型在 `thinking`（思考）或 `toolcall`（工具调用）阶段明明还活着，父进程却可能因为看不到文本流而误判成 timeout（超时）。
3. **timeout 语义模糊**：主 agent 容易把“审核空闲超时”脑补成“也许后台还在继续审”，从而在当前 phase（阶段）尚未通过时，擅自推进后续 phase 或提前进入终审。
4. **阶段推进护栏不完整**：当前顺序约束主要落在 `dgoal_plan` 上；真实暴露出来的问题不是“工具本身有副作用”，而是**phase 没过却先去推进下一个 phase 或终审**。这属于 `Gate-Jumping Progression`（越闸门推进），需要显式堵住。

用户侧并不需要为每次中间建检失败介入；用户关心的是启动前确认、goal 完成结果，以及连续失败到 `paused`（暂停）这类 `User Intervention Point`（用户介入节点）。因此，这个版本的重点不是把更多正文塞进 TUI，而是把建检过程变成**主 agent 可修、用户可观察、状态机不乱跳**的正式闭环。

## 2. Solution

v0.5.2 统一补四件事：

1. **Check Feedback（建检反馈）持久化**：阶段建检和终审未通过时，把检查 agent 给出的原始失败报告完整写入 `LoopGoal`，后续主 agent 续跑时完整钉回。
2. **事件流化审核器**：把 `dgoal_check` 从“最终报告导向”改成“事件流导向”。父进程显式消费 `thinking_*`、`toolcall_*`、`text_delta`、`message_end` 等事件，产出正式的 `Check Liveness State`（建检活性状态）。
3. **真实 `auditor_error` 的 3 次透明重试**：只有审核器自身异常（子进程 / 事件流 / child 内模型调用失败等）才走内部透明重试。任一重试成功产出正常建检结论，就不再继续重试；3 次都失败才进入 `paused(audit_error)`。
4. **Check-Gate Lock（建检闸门锁定）**：阶段建检未通过、无结论或审核器异常时，goal 仍保持 `active`，但只能围绕当前 phase 修正、补证据并重试建检；不能越阶段推进，也不能越终审推进。

这里要特别收窄边界：

- v0.5.2 **拦的是推进规则**，不是对所有 `write` / `edit` / `bash` 做广义副作用审查。
- `LoopGoal` 继续只放**阶段性持久业务事实**；建检活性状态属于**运行时观察层**，默认不写入 `LoopGoal`。
- TUI 只展示**状态 + plan 结构**，不展示原始建检报告正文。
- 本轮不引入自动换模型 / 备用模型；审核器继续复用当前会话模型。

## 3. User Stories

1. As a 主 agent, I want 阶段建检失败报告在下一轮仍完整可见, so that 我能基于真实失败项修正当前 phase，而不是凭记忆瞎修。
2. As a 主 agent, I want 审核器在思考和调工具时也被正式观察, so that 我不会把“还在跑”误解成“后台异步”或“已经没事了”。
3. As a 主 agent, I want 真实 `auditor_error` 在工具内部先透明重试 3 次, so that 瞬时基础设施抖动不会过早打断 loop。
4. As a user, I want aboveEditor 和 `/dgoal s` 只显示状态与 plan 结构，不展示原始报告正文, so that 状态视图保持简洁。
5. As a maintainer, I want phase 没过时不能越阶段推进、不能越终审推进, so that dgoal 的状态机约束和真实执行边界一致。

## 4. Implementation Decisions

### 4.1 `LoopGoal` 只扩反馈，不承载运行时活性

在 `LoopGoal` 中新增短期反馈字段：

```ts
interface CheckFeedback {
  report: string;
  createdAt: number;
}

interface PhaseCheckFeedback extends CheckFeedback {
  phaseId: number;
}

interface FinalCheckFeedback extends CheckFeedback {
  rejectedCount: number;
}
```

`LoopGoal` 增加：

```ts
phaseFeedbackById?: Record<string, PhaseCheckFeedback>;
finalFeedback?: FinalCheckFeedback;
```

说明：

- `phaseFeedbackById` 按 `phaseId` 定位，不表示允许多个 phase 并行失败；dgoal 仍保持顺序推进。
- `LoopGoal` **不新增** `thinking` / `tool_running` / `latestSnippet` 之类字段；这些属于运行时观察态，不做持久化。

### 4.2 建检反馈写入与清理规则

- `dgoal_check` 明确未通过：写 `phaseFeedbackById[phaseId] = { phaseId, report: result.output, createdAt }`，并保持/回退 phase 为 `in_progress`。
- `dgoal_done` 终审明确未通过：写 `finalFeedback = { report: audit.output, rejectedCount: newCount, createdAt }`，并进入 `rejected` 或 `paused(audit_failed_3x)`。
- `auditor_error`、`aborted`、无有效结论等**不是建检反馈**，不写入 feedback。
- phase 建检通过：清除 `phaseFeedbackById[phaseId]`。
- 终审通过或 `/dgoal clear`：feedback 随 goal 一起清空。
- `/dgoal resume` from `paused(audit_failed_3x)`：保留 `finalFeedback`，只清零 `rejectedCount`。

### 4.3 事件流化审核器

独立审核子进程从“等最终报告文本”改成“显式消费运行事件”：

- `thinking_start`
- `thinking_delta`
- `thinking_end`
- `toolcall_start`
- `toolcall_delta`
- `toolcall_end`
- `text_delta`
- `message_end`

父进程维护一份**运行时内存态**的 `Check Liveness State`，至少覆盖：

- `starting`
- `thinking`
- `tool_running`
- `report_streaming`
- `approved` / `rejected` / `auditor_error`（最终收敛态）

**超时语义与时长（明确决策）**：

- **动作的定义**：任何有效事件（`thinking_*` / `toolcall_*` / `text_delta` / `message_end`）出现，就重置 idle timer（空闲计时器）。这是相对当前实现的关键变化——当前只把 `text_delta` 类文本流当成“有动作”，导致模型在思考或调只读工具时，2 分钟倒计时也在空跑。
- **超时时长（秒单位）**：从 `CHECK_IDLE_TIMEOUT_MS = 120_000` 改为 `CHECK_IDLE_TIMEOUT_SECONDS = 120`，内部 `* 1000` 传给 timer。读起来是“120 秒”，不是一串毫秒大数；语义从“2 分钟没出文本”改为“2 分钟没有任何动作”。配合事件流化后，2 分钟基本只会被“真卡死”触发。
- **可见倒计时（明确决策）**：建检运行时维护一个**剩余秒数**，随 `onUpdate` 活性状态流出，格式类似 `idle 118s/120s`。TUI 可显示它；每次有有效事件就跳回 120s。正常情况下一有事件就跳回（说明还活着），只有真卡死才会一路降到 0（说明真没动作了）。**位置口径**：倒计时走 `onUpdate` 工具执行流（`tool_execution_start → onUpdate → tool_execution_end`），归工具调用渲染层；它**不走** `ctx.ui.setStatus("dgoal", ...)` 底部状态栏，不与 goal 级状态（`active / paused / rejected / done`）混在一起。建检工具在跑时倒计时出现，工具一结束（通过/未过/异常）就随工具结果一起收掉。
- **倒计时实现边界**：真正的 kill 仍由 idle `setTimeout` 控制；倒计时只是额外起一个 1s `setInterval` ticker 更新“剩余秒数”这个展示字段，两者解耦，ticker 不触发 kill。Pi 的 `onUpdate` 本就有 1s throttle（`CHECK_PROGRESS_UPDATE_THROTTLE_MS = 1_000`），每秒刷一次倒计时正好落在节流窗口里，不会刷爆 TUI。倒计时是**运行时展示态**，不是 `LoopGoal` 持久态，不写进 `LoopGoal`；它属于 `Check Liveness State` 的展示元数据，和 `thinking / tool_running` 等状态并列。
- **可调参数**：`CHECK_IDLE_TIMEOUT_SECONDS` 作为可调参数；未来可基于实测下调到 1 分钟，本版本不调。
- timeout 收窄为：**事件流本身长期静默**、子进程起不来、或 child 明确模型调用异常等真实基础设施问题，归类为 `auditor_error`。
- `text_delta` 只是活性事件之一，不再是“是否活着”的唯一证据。

### 4.4 结果三态结构化返回与 `auditor_error` 3 次透明重试

`dgoal_check`（含终审）的结果正式收敛成三态，让主 agent 和 TUI 不再从自然语言文案里猜“这到底是没过、还在跑、还是审核器坏了”：

| 态 | `isError` | goal 状态 | 含义 |
|---|---|---|---|
| `approved` | false | phase→done | 建检通过 |
| `rejected` | **false** | active + 闸门锁定 | 建检有结论但未通过，**正常业务结果** |
| `auditor_error` | **true** | paused(audit_error) | 审核器自身异常（3 次重试耗尽） |

关键点：

- `rejected` 是“判卷器正常工作后给出不通过的结论”，不是错误。让它 `isError: false`，主 agent 才会把它当成“继续修当前 phase”的信号，而不是“出错了要处理异常”。
- `isError: true` 只留给 `auditor_error`，主 agent 才能一眼区分“判卷器正常，我要修东西” vs “判卷器坏了，我不能空转”。
- 这正好堵掉这次会话的根因：当前 `auditor_error`（timeout）返回 `isError: false`，agent 误以为“还能继续”，结果越界推进。

把“建检未通过”和“审核器坏了”彻底分开：

- **建检未通过**：正常业务结果，不重试审核器。
- **`auditor_error`**：审核器自身异常，先在一次 `dgoal_check` 调用内部透明重试 3 次。

具体语义：

- `attempt 1/3`、`2/3`、`3/3` 由工具内部消化，TUI 可显示当前尝试次数。
- 任意一次重试只要得到正常结论（通过或未通过），就立刻结束重试链路。
- 只有 3 次都仍是 `auditor_error`，才进入 `paused(audit_error)`。
- 这 3 次重试不是让主 agent 自己重复调用 3 次 `dgoal_check`，而是判卷器的自救。

### 4.5 建检闸门锁定与越闸门推进防护

当阶段建检未通过、无结论或审核器异常时：

- goal 保持 `active`
- 当前 phase 维持/回退 `in_progress`
- 进入 `Check-Gate Lock`

锁定后的允许动作：

1. 修当前 phase
2. 补当前 phase 的 evidence（证据）
3. 重试当前 phase 的 `dgoal_check`

锁定后的硬约束：

- 不能越阶段推进
- 不能越终审推进

v0.5.2 明确**不做**“所有工具都懂意图边界”的大系统；硬拦截只打在**推进节点**上：

- `dgoal_plan`：后续 phase 相关操作继续硬拒绝
- `dgoal_check`：当前 phase 未过时，不允许对后续 phase 发起建检
- `dgoal_done`：当前 phase / 前序 phase 未完成时，不允许提前进入终审

也就是说，第一版拦的是 `Gate-Jumping Progression`（越闸门推进），不是把每个 `write` / `edit` / `bash` 都拉进审判。

### 4.6 Prompt 注入规则

新增 `buildCheckFeedbackBlock(goal)`：

- `goal.status === "active"`：定位当前未 done 的 phase，只注入该 phase 对应的 `phaseFeedbackById[phaseId]`
- `goal.status === "rejected"` 或 `paused(audit_failed_3x)` resume 后继续修复：注入 `finalFeedback`
- feedback block 放在 `<loop_plan>` 之后、循环规则之前
- 保留原始报告，不做 summary、不压缩

示意：

```text
<check_feedback type="phase" phaseId="N">
...检查 agent 原始报告...
</check_feedback>
```

或：

```text
<check_feedback type="final" rejectedCount="N">
...检查 agent 原始报告...
</check_feedback>
```

### 4.7 用户界面边界

- tool result 仍完整展示原始报告，让人和 AI 当轮都可读。
- aboveEditor 浮层只展示状态与 plan 结构；建检进行中可显示活性状态和尝试次数，但**不展示报告正文**。
- `/dgoal s` modal 只展示状态与 plan 结构；**不展示报告正文**。
- status bar 不展示原始报告。

### 4.8 本轮明确不做

- 不做自动换模型 / 备用模型 / 审核器独立模型配置。
- 不把建检活性状态持久化进 `LoopGoal`。
- 不做 same failure / no progress / regression 自动判定器。
- 不做广义工具副作用审查。
- 不把原始建检报告展示进 `/dgoal s`、overlay、status bar。

## 5. Out of Scope

- 不做失败历史归档；只保留当前未解决反馈。
- 不生成 summary，不裁剪、不压缩原始报告。
- 不把建检反馈展示到 `/dgoal s` modal、aboveEditor 浮层或 status bar。
- 不做 same failure / no progress / regression 的自动识别。
- 不改独立审核子进程模型选择或容灾策略。
- 不把运行时建检活性状态写入 `LoopGoal`。
- 不替代项目自身测试命令；反馈只是让主 agent 更稳定地看到失败原因。

## 6. 切片拆解与验收标准

### 切片 1：反馈数据模型与持久化兼容

**范围**：
- 扩展 `LoopGoal` 类型，新增 `phaseFeedbackById` 和 `finalFeedback`
- 更新 state load/validate 兼容逻辑，旧 state 没有 feedback 字段仍可加载
- 新增反馈写入/清理纯函数

**验收**：
- 旧版 `LoopGoal` fixture 能正常 load/validate
- phase / final feedback 写入后 `persistGoal/loadGoal` 可恢复完整 report
- 不新增任何运行时活性字段到 `LoopGoal`

### 切片 2：阶段建检反馈写入与清理

**范围**：
- `dgoal_check` 未通过路径写 `phaseFeedbackById[phaseId]`
- `dgoal_check` 通过路径清理对应 phase feedback
- `auditor_error` / aborted / 无有效结论路径不写 feedback

**验收**：
- 同 phase 多次 rejected 会覆盖为最新报告
- phase approved 后对应 feedback 被清理
- `auditor_error` 不写 phase feedback

### 切片 3：终审反馈写入、保留与清理

**范围**：
- `dgoal_done` rejected 路径写 `finalFeedback`
- 3 次终审 rejected 进入 `paused(audit_failed_3x)` 时保留 `finalFeedback`
- `/dgoal resume` 清零 `rejectedCount` 但保留 `finalFeedback`
- 终审 approved 或 `/dgoal clear` 后随 goal 清空

**验收**：
- 第 1/2 次终审 rejected 后，state 含完整 `finalFeedback.report`
- 第 3 次终审 rejected 进入 paused 后，`finalFeedback.report` 仍存在
- resume 后 `rejectedCount` 清零，`finalFeedback.report` 不丢

### 切片 4：事件流化审核器

**范围**：
- `runIsolatedCheck()` 消费 `thinking_*` / `toolcall_*` / `text_delta` / `message_end`
- 建立运行时 `Check Liveness State`
- 任何有效事件都重置 idle timer
- timeout 收窄为真实基础设施异常

**验收**：
- child 只在 thinking / toolcall 时，不再被误判成空闲超时
- 任何 `thinking_*` / `toolcall_*` / `text_delta` / `message_end` 事件出现都会重置 idle timer
- `onUpdate` 能看到状态从 `starting → thinking → tool_running → report_streaming`
- 无事件流、spawn 失败、child 明确模型异常时，才归类为 `auditor_error`
- 超时参数以秒为单位（`CHECK_IDLE_TIMEOUT_SECONDS = 120`），不再出现裸毫秒魔数
- 剩余秒数倒计时随活性状态流出（格式如 `idle 118s/120s`），有事件跳回 120s，无事件降到 0
- 倒计时走 `onUpdate` 工具执行流，**不写进** `ctx.ui.setStatus("dgoal", ...)` 底部状态栏，不与 goal 级状态混在一起
- 倒计时 ticker（1s `setInterval`）只更新展示字段，不触发 kill；kill 仍由 idle `setTimeout` 控制
- 倒计时是运行时展示态，不写进 `LoopGoal`

### 切片 5：结果三态与 `auditor_error` 3 次透明重试

**范围**：
- `dgoal_check`（含终审）结果收敛成 `approved / rejected / auditor_error` 三态结构化返回
- `rejected` 返回 `isError: false`；`auditor_error` 返回 `isError: true`
- 在一次 `dgoal_check` / 终审调用内部封装 3 次透明重试
- 尝试次数进入运行时状态，供 TUI 显示
- 3 次都失败才走 `paused(audit_error)`

**验收**：
- `approved` 返回 `isError: false`，phase 进 done
- `rejected` 返回 `isError: false`，goal 保持 active + 闸门锁定
- `auditor_error` 返回 `isError: true`（3 次重试耗尽后）
- 第 1/2 次出现 `auditor_error` 后自动重试，不要求主 agent 手动再调
- 任意一次重试获得正常建检结论后立即停止重试
- 3 次都失败时 goal 进入 `paused(audit_error)`

### 切片 6：建检闸门锁定与越闸门推进拦截

**范围**：
- `dgoal_plan` 顺序防护继续保留/补强
- `dgoal_check` 拒绝对后续 phase 发起建检
- `dgoal_done` 拒绝在当前 phase 未过时提前进入终审
- rejected 后维持 `active + 当前 phase 锁定`

**验收**：
- phase N 未过时，对 phase N+1 调 `dgoal_check` 被拒绝
- phase N 未过时调用 `dgoal_done` 被拒绝
- phase rejected 后 goal 不 paused，仍 active，但只能继续修当前 phase

### 切片 7：Prompt 注入与 UI 边界回归

**范围**：
- 新增 `buildCheckFeedbackBlock(goal)`
- active 只注入当前 phase feedback；rejected 注入 final feedback
- overlay / `/dgoal s` / status bar 不展示报告正文
- overlay / `/dgoal s` 可展示活性状态与尝试次数

**验收**：
- system prompt 在正确状态下注入完整 `<check_feedback>` block
- `/dgoal s` 与 aboveEditor 不渲染 report 正文
- 工具结果仍包含完整 `审核报告`

## 7. 测试决策

优先使用现有公开接缝和纯函数测试，不测内部实现细节：

- 数据模型/持久化：复用现有 state load/persist 接缝
- 阶段建检：扩展 `tool-execute-integration` 或相邻测试，验证 `dgoal_check` rejected/approved/`auditor_error` 对 feedback 与状态的影响
- 终审：扩展状态机/工具执行测试，覆盖 rejected、`paused(audit_error)`、resume 的生命周期
- 事件流：补 `runIsolatedCheck` 相关测试，覆盖 thinking-only、toolcall-only、真实 idle、spawn 失败、3 次重试成功/失败
- Prompt 注入：扩展 `state-machine-and-prompt.test.ts`，断言 `<check_feedback>` block
- UI 边界：扩展 `plan-status-dialog` / `plan-overlay-render` 相关测试，确保 report 不进入 modal/widget，只展示状态与 plan 结构

建议验证命令：

```bash
npm test
npm run test:rpc
```

本版本触碰审核器运行时与 TUI 状态展示，**人工 TUI smoke 重新变成必做项**：至少覆盖一次阶段建检进行中、一次 rejected 修回环、一次真实 `auditor_error` 3 次重试耗尽后的 `paused(audit_error)`。

## 8. 风险与缓解

| 风险 | 缓解 |
|---|---|
| feedback 字段变成长期 audit log | 只保留当前未解决反馈；phase 通过即清，goal 清除即清 |
| 事件流化后 runtime 复杂度上升 | 活性状态只放运行时内存，不扩 `LoopGoal`；切片化测试 |
| thinking / toolcall 事件噪音过大 | TUI 只展示结构化状态与简短片段，不展示原始报告正文 |
| `auditor_error` 重试与 rejected 混淆 | 明确 rejected 是业务结果，`auditor_error` 是基础设施异常；两者走不同路径 |
| 过度扩大到工具拦截系统 | v0.5.2 只拦 `Gate-Jumping Progression`，不做广义工具审查 |
| 用户界面被报告刷屏 | tool result 展示当轮报告；modal/overlay/status 不展示正文 |
| 模型容灾把版本范围拉炸 | 明确本轮不做自动换模型 / 备用模型 |
