# v0.4.2 `/dgoal s` Modal 实施方案

> **版本归属**：0.4.2
> **状态**：已完成
> **一句话目标**：把 `/dgoal s` 从 5 行 `ctx.ui.notify` 升级为 top-center overlay modal，让用户能按需看完整 plan 状态（goal + 所有 phase + 所有 task），与持续显示浮层（aboveEditor widget）职责正交。
>
> **决策依据**：`doc/adr/0008-dgoal-s-modal-形态选型.md`（Variant A 选型） + `doc/20-能力参考/25-dgoal-s-modal变体探索参考.md`（3 变体取舍 + v1→v2→v3 bug 复盘）。

## 1. 背景

当前 `/dgoal s`（index.ts:1051 `showStatus`）调 `ctx.ui.notify` 输出 5 行：

```typescript
ctx.ui.notify([
    t("status.objective", { objective: currentGoal.objective }),
    t("status.state", { status: currentGoal.status }),
    t("status.iteration", { iteration: currentGoal.iteration }),
    contextPreview ? t("status.contextPreview", { preview: contextPreview }) : t("status.noContextPreview"),
    t("status.commands"),
].join("\n"), "info");
```

底栏 status bar 也只显示 `🔁 active #N`。

两个视图都不能告诉用户**当前 plan 的完整状态**。当 goal 包含 5+ phase / 20+ task 时，用户无法在 TUI 内一目了然看到"现在在做什么、已完成什么、被什么 block"。

调研 + 原型 + 决策链见 ADR 0008 + 25-变体探索参考。核心结论：选 Variant A（top-center overlay + scroll + 颜色 + emoji 标志），不选 B（无 scroll + 不显示 task），C 降级备选。

## 2. 核心定义

| 概念 | 0.4.2 定义 |
|---|---|
| Modal | `ctx.ui.custom()` 弹的 Component，用户主动唤起，按需看完整信息 |
| 浮层 | 已有的 `PlanOverlay` widget（`aboveEditor` placement），持续显示 phase 进度，task 默认隐藏 Ctrl+O 展开 |
| 状态栏 | 已有的 `setStatus("dgoal", ...)`，单行 indicator（`🔁 active #N` 等） |
| 三者职责 | 浮层=持续进度，状态栏=轻量 indicator，modal=按需详细查询 |

不重新定义术语，复用 0.2.0 已有（goal/phase/task/RenderLine 见 ADR 0008）。

## 3. 目标

0.4.2 做 6 件事（垂直切片，每步可独立验证；总代码量估算 ~200 行）：

1. **RenderLine 数据结构**：把现有 `renderPlanLines` 返回 `string[]` 升级为 `RenderLine[]`（`{ type, status?, text }`），让 render 阶段按 status 染色。
2. **`colorize(line, theme)` 纯函数**：按 line.type + status 映射 theme 颜色（heading accent+bold / phase completed success / phase in_progress accent+bold / phase blocked warning / phase pending muted / task completed dim / task in_progress accent+bold / task blocked warning / task pending muted）。
3. **`computeScrollOffset(data, currentOffset, total, maxVisible)` 纯函数**：返回新 offset / "exit" / null。vim 风格 j 下、k 上（不在函数内修改 caller state——v2 bug 教训）。
4. **`PlanStatusDialog` Component 类**：~150 行 Component + Focusable。render 输出边框 + heading 钉顶 + body 可滚动 + 底部 hint；handleInput 调 computeScrollOffset；invalidate 清缓存。
5. **重写 `showStatus` (index.ts:1051)**：从 notify 改成 `ctx.ui.custom(factory, { overlay: true, overlayOptions: { anchor: "top-center", width: "100%", maxHeight: "85%", margin: 1 } })`。无 goal 时保留 notify 兜底。
6. **i18n + 测试 + 文档 + 清理**：emoji/颜色走 theme；新增单元测试覆盖三个纯函数；`doc/10-架构与运行/13-启动闸门与TUI浮层.md` 已有 `/dgoal s` 章节（已在 v3 时同步），删 prototype 目录。

## 4. 非目标

0.4.2 不做：

- 不改 `PlanOverlay` widget（持续显示浮层）。modal 与 widget 是**两个** surface，职责正交（modal 按需 / widget 持续）。
- 不引入新工具或新命令。`/dgoal s` 命令名不变，语义不变（查询当前 goal 状态），只是输出形态升级。
- 不重写 `formatStatus` 或 status bar 文案。
- 不做 full-screen takeover（之前 A 试过被用户否定——"想往上移至少不遮 input"）。
- 不重做底部 status bar 视觉（保持单行 indicator）。
- 不做 i18n 完整迁移——emoji + 颜色是 Pi theme 自带的，中文/英文文案仍走 `t()` 软依赖 pi-di18n。
- 不修复 user-reported "毫无变化" bug（用户已说"先这样了"）。

## 5. 切片拆解与验收标准

### 切片 1：RenderLine 数据结构

**范围**：
- 在 `index.ts` 加 `RenderLine` 类型：`{ type: "heading" | "spacer" | "phase" | "task"; status?: PlanStatus; text: string }`。
- 把 `renderPlanLines` 返回类型从 `string[]` 改为 `RenderLine[]`（line 类型 = heading / phase / task / spacer）。
- heading 单独生成（不参与 scroll），保持 `🎯 <objective> (X/Y) ⏱️ <elapsed>` 格式。
- phase/task 行保持原 `├─ ${emoji} ...` / `│    ${icon} ...` 格式，但带 status 字段。
- emoji 改 `✅/🔄/⬜/🚧`（phase）+ `✓/◐/○/⚠`（task 保持）。
- 抽出两个纯函数：`buildBodyLines(goal)` 返回 `RenderLine[]`、`buildBodyLinesNoHeading(goal)` 返回 `RenderLine[]`（slice 2）。
- 抽出 `buildHeadingLine(goal)` 返回 string（用于钉顶渲染）。

**验收**：
- 单元测试 `buildBodyLines(mockGoal)` 返回 33 个 RenderLine（heading + spacer + 8 phase + 22 task）。
- `buildBodyLinesNoHeading(mockGoal).length === 31`。
- emoji 字符与 status 一致。
- `renderPlanLines` 调用方（widget 浮层 `PlanOverlay.update()`）改用新结构但不破坏现有渲染。

### 切片 2：`colorize(line, theme)` 纯函数

**范围**：
- 新增 `colorize(line: RenderLine, theme: Theme): string` 纯函数（~25 行）。
- 映射规则（见 ADR 0008 第 5 节）：
  - heading → `theme.fg("accent", theme.bold(text))`
  - spacer → `text` 不变
  - phase + in_progress → accent + bold
  - phase + completed → success
  - phase + blocked → warning
  - phase + pending → muted
  - task + in_progress → accent + bold
  - task + completed → dim
  - task + blocked → warning
  - task + pending → text（默认）

**验收**：
- 单元测试每个 status × (phase/task) 共 8 种组合输出包含对应 ANSI 颜色。
- 无 status 的 heading/spacer 走原样输出。

### 切片 3：`computeScrollOffset` 纯函数

**范围**：
- 新增 `computeScrollOffset(data: string, currentOffset: number, totalLines: number, maxVisible: number): number | "exit" | null` 纯函数（~30 行）。
- 识别：`escape`/`ctrl+c` → "exit"；`down`/`j` → +1；`up`/`k` → -1；`pagedown`/`ctrl+d` → +10；`pageup`/`ctrl+u` → -10；`end`/`G` → maxOffset；`home`/`g` → 0；其他 → null。
- maxOffset 边界：`Math.max(0, totalLines - maxVisible)`；上下 clamp。

**验收**：
- 单元测试每种键的预期输出（j = +1, k = -1, G = maxOffset, g = 0, ESC = "exit"）。
- 边界：currentOffset=0 按 j 不变；currentOffset=maxOffset 按 k 不变。

### 切片 4：`PlanStatusDialog` Component 类

**范围**：
- 新增 `class PlanStatusDialog implements Component, Focusable`（~150 行）。
- 字段：`focused = false` + render 缓存 (`cachedWidth`, `cachedLines`, `cachedScrollOffset`) + `scrollOffset = 0` + `maxVisible: number` + `theme` + `done: () => void`。
- `handleInput(data)`：调 computeScrollOffset → 收到 "exit" 调 done；收到 number 且 ≠ 当前 offset → 赋值 scrollOffset + invalidate；null 不动。
- `render(width)`：边框（圆角 ╭─╮ ╰─╯，border 色 + accent 标题）+ heading 钉顶（accent + bold）+ body 可滚动切片（每行 colorize 后 +1 空格缩进）+ 底部 hint（offset 指示 + 键位）+ 边框收尾。
- `invalidate()`：清三个缓存字段。
- 复用 切片 1-3 的纯函数；Component 自身不写新逻辑。

**验收**：
- 单元测试 render(width=80) 返回 line 数 = 边框(2) + heading(1) + body(maxVisible) + hint(1) = maxVisible + 4。
- 单元测试 handleInput 接收 `"j"` data → scrollOffset += 1 + invalidate。
- 单元测试 handleInput 接收 `"\x1b"` (ESC) → done 调一次。
- 缓存逻辑：第二次 render 相同 (width, scrollOffset) 返回 cached。

### 切片 5：重写 `showStatus`

**范围**：
- 改 `index.ts:1051` 的 `showStatus` 函数：
  - 无 `currentGoal` 时：TUI 模式仍弹 top-center modal 空状态；非 TUI / custom 不可用时 `ctx.ui.notify(t("status.noDgoal"), "info")` 兜底。
  - 有 `currentGoal` 时：调 `ctx.ui.custom<void>((_tui, theme, _kb, done) => new PlanStatusDialog(currentGoal, theme, () => done()), { overlay: true, overlayOptions: { anchor: "top-center", width: "100%", maxHeight: "85%", margin: 1 } })`。
- i18n 文案：底部 hint 走 `t("status.dialogHint", { shown })` 新 key（加入中英双语 fallback）。
- 导出 `PlanStatusDialog` 供测试 import。

**验收**：
- 单元测试覆盖 `showStatus(undefined ctx)`：TUI 模式走空状态 modal，非 TUI 模式调 notify 兜底。
- TUI smoke test：在 Pi TUI 跑 `/dgoal s`，弹 top-center overlay；j/k/↑↓/PgDn/PgUp/End/Home/ESC 都响应；ESC 干净退出；input + status bar 始终可见。

## 8. 风险与缓解

| 风险 | 缓解 |
|---|---|
| Component render 缓存失效（每秒 elapsed 刷新导致每秒重渲染） | 缓存 key 包含 `Date.now()` 量化到秒；或 elapsed 不走 render 缓存（独立计算） |
| maxHeight 85% 在小 terminal（< 25 行）下挡 input | terminal < 30 行是边缘场景；接受 modal 略挡 input，加 hint 提示"按 ESC 退出" |
| 颜色 + emoji 在某些 terminal 显示异常（黑白字形） | 走 theme tokens（用户已切主题会自适应）；emoji 用 box-drawing 类（✅🔄⬜🚧）兼容性好 |
| i18n 文案 `status.dialogHint` 漏 key 导致 fallback 报错 | 用现有 `t()` 软依赖模式（缺 key 时 fallback 到内置字符串），同 status.contextPreview |
| scroll handleInput 边界 bug（v2 已修但仍要小心） | 切片 3 强制走纯函数返回 offset，caller 赋值；不在纯函数内修改 state |
| 用户已反馈"毫无变化"——可能 Pi 没 reload / 文件没存 | CHANGELOG / commit 消息里提醒"记得 `/reload`"，人工 smoke test 确认 |
| 现有 widget 浮层（PlanOverlay）受 modal 影响 | 不应受影响——modal 走 overlay 系统，widget 走 widget 系统，两者独立；TUI smoke test 验证两者并存 |

## 9. 版本号说明

按 semver：
- `/dgoal s` 重写是**功能增强**（现有命令的输出形态升级），不是纯 bug fix → 应该是 minor bump（0.5.0）。
- 但本方案按用户指定标 0.4.2（patch）。**判断**：把这次升级视为"既有功能的视觉增强"而非"新功能"——和 0.4.0（命令别名）+ 0.4.1（容错）性质接近，仍是 0.4.x 系列内的迭代。

如果发版后收到反馈"应该 0.5.0"，下次 minor bump 时把这条 backport 进 0.5.0 changelog 即可（功能是叠加的不是破坏性的）。
