# v0.7.0 验收策略与运行预算实施方案

> Status：已实现（v0.7.0）。受 `30-路线图` 约束。

## 目标

让同一个 dgoal 在启动闸门中由主 agent 推荐、用户确认验收策略与预算策略：普通复合任务可用 `final_only` 避免逐 phase 审核成本；有真实中间验收门的任务仍可用 `phased`。有界任务在首次触顶后自动消耗一次预授权宽限，无上限任务可在会话内持续推进，但不绕过真实冲突暂停或 goal 级独立终审。主 LLM 直接提交可选背景；用户全局配置授权后，可受限地隐式启动轻量 `final_only` 有界 goal。

## 产品行为基线

### 启动闸门

`dgoal_propose` 提交计划时同时提交：

- `verificationPolicyRecommendation`：`final_only` 或 `phased`，及简短推荐理由；
- `budgetPolicyRecommendation`：`bounded`（附推荐上限）或 `unbounded`；
- 两种策略所需的验收信息。

现有确认 UI 保持一次阻塞交互：默认展示 LLM 推荐组合，用户可直接确认，或在同一选择列表改选其余策略组合；不增加第二个对话框。选择有界策略时采用该 proposal 附带的上限；要改上限则输入反馈并重新提案。用户确认后，实际策略写入 `GoalState` 并冻结。

新 proposal 在 `final_only` 下只要求 goal 级 `acceptanceCriteria`；`phased` 继续要求 goal 和每个 phase 的 `acceptanceCriteria`。旧 goal 不迁移，沿用原 phased 语义。

### 背景与隐式轻量启动

主 LLM 在 `dgoal_propose` 中按需提交 `contextSummary`，替代启动前独立摘要子进程；缺失背景不阻塞启动，已有旧 goal 的背景继续兼容。proposal 仍是建立 goal/phase/task 持久态的唯一入口。

默认只有用户 `/dgoal` 才能启动。全局 `pi-dgoal.json` 显式启用 `implicitFinalOnlyStart` 后，LLM 才可对范围具体、可独立验收的任务直接提交并执行 proposal。该路径允许受边界校验的本地项目文件工具读写和外部只读信息，但不允许通过 shell 执行任意 agent 可编辑脚本或测试代码；测试/脚本执行应转为显式 `/dgoal` 或独立审核子进程。禁止消息发送、发布/部署、`git push`、外部删除、账号/权限修改与付费资源消费；仅允许 `final_only + bounded`，默认基础上限为 24 个执行 turn、60 分钟有效运行时长、1 个终审修复回合，turn 宽限再给 24 个执行 turn，可由全局 `implicitFinalOnlyBudget` 覆盖。它仍经过结构校验和语义预审；通过后不弹 plan 审核框，改为非阻塞通知与已有浮层。项目配置不得开启这项权限。

### `final_only` 执行与呈现

phase 继续承担 Task Plan、浮层和上下文组织。一个 phase 下 task 全部完成后，主 agent 回顾该阶段并明确标记**阶段进度完成**；浮层使用已有完成划线展示，但该事实与“通过 `dgoal_check` 的阶段完成”分开持久化，不能作为独立审核证据。

实现采用 phase 的独立进度完成标记，而不重定义既有 `done`。`dgoal_plan` 新增一个只在 `final_only` 可用的 phase-progress 完成操作；主 agent 新建或重开该 phase 的 task 后，该标记失效并回到进行中。所有 phase 都阶段进度完成后才允许 `dgoal_done`。

`final_only` 不调用 `dgoal_check`；`dgoal_done` 是唯一独立审核入口。其运行中活性、候选回退与拒绝后的 `Goal Repair` 直接复用既有浮层、状态栏和 modal，不加专用面板或报告正文。

### `final_only` 审核收敛

主 agent 每次调用 `dgoal_done` 必须带验证包：本轮改动、冻结条件与命令/工件映射、最后一次改动后的自测、已知风险与未覆盖边界。

- 首次终审是诊断审：集中找全 blocker、真实回归与高风险证据缺口。
- 诊断审拒绝后的每次终审是窄确认审：只检查上一轮 blocker、该轮新增 diff、受影响回归测试与少量全局保护测试。
- 审核器不得新增完成门、偏好或无关 nits；新增 diff 造成的真实回归仍可拒绝。
- 验证包只提供定位上下文。独立证据复用仍严格限制于 ADR 0030 的审核 checkpoint。

审核账本必须保存诊断/确认模式、对应验证包和基线 workspace fingerprint，供重审 prompt 窄化范围；工作区不匹配时不复用任何 child 工具事实。

### 预算与暂停

有界策略由结构化上限表达执行轮次、墙钟时长和审核修复回合；缺省维度不限制。首次达到任一上限时，发一次非阻塞 `notify`，状态栏持续显示“宽限中”，并自动启用一次用户预授权的宽限。宽限结束后仍未完成则 `paused(budget_exhausted)`，保留实际消耗、最近审核反馈和恢复建议。

`unbounded` 取消执行轮次、墙钟时长和固定终审拒绝次数触发的预算暂停；它只在当前 Pi 会话继续运行。模型错误、无进展、审核技术错误、用户暂停与 `paused(agent_blocked)` 仍照现有安全语义处理。真实冲突绝不以 done 收束。

## 垂直切片

1. **提案、背景与启动权限**：移除独立背景摘要子进程与其 fail-closed 路径；扩展 proposal 的可选背景、策略推荐和组合选择；根据实际策略校验 acceptance criteria；确认后的实际选择进入 START prompt。加入全局 `implicitFinalOnlyStart` 权限、外部动作拒绝、默认 `24 turns / 60 min / 1 repair`、turn 宽限 24 与 `implicitFinalOnlyBudget` 覆盖、隐式路径的 `final_only + bounded` 硬限制与通知。验证默认拒绝、全局启用、项目配置不能越权、外部动作与越界策略拒绝、无背景启动、同框覆盖、反馈重提和旧 goal 兼容。
2. **`final_only` 阶段进度**：新增独立的阶段进度完成事实与 `dgoal_plan` 操作；浮层划线、上下文注入、重开 task 的失效规则；拦截 `final_only` 调用 `dgoal_check`，并允许全部阶段进度完成后 `dgoal_done`。验证其不伪装为阶段建检。
3. **验证包与收敛式终审**：扩展 `dgoal_done` 输入、审计账本与 auditor prompt；首次诊断、后续窄确认、冻结完成门不扩容、workspace 变化失效。验证审核 checkpoint 安全边界不回归。
4. **结构化预算与暂停**：记录消耗、一次宽限、`budget_exhausted` 暂停与 `unbounded` 对三次终审暂停的覆盖；接入非阻塞提示和状态栏。验证所有原有安全暂停不被无上限削弱。
5. **全链路回归与文档**：补 RPC/确定性生命周期测试，按覆盖面跑 context、RPC、全量 Bun；更新 README、架构文档、术语表、测试地图、CHANGELOG。涉及 TUI 的组合选择、宽限提示与划线呈现做人工 TUI smoke。

## 验收条件

- 用户能在一次启动确认交互中接受 LLM 推荐，或改选任意验收/预算策略组合；确认后的实际值可由持久态与 START prompt 复验。主 LLM 可选提交背景而不触发独立摘要；仅全局配置授权后，合格任务可隐式启动 `final_only + bounded`，项目配置和越界策略均被拒绝。
- `final_only` 多 phase goal 无需 phase 级验收条件或 `dgoal_check`，每个阶段进度完成仍在浮层划线，所有阶段完成后 goal 独立终审通过才关闭。
- `phased` 保持现有 phase 建检门与 goal 终审，不因新增策略放宽。
- 首次 `final_only` 拒绝后的审核只接收上一轮 blocker、修复新增 diff、受影响测试与保护测试作为工作范围；冻结完成门和 ADR 0030 checkpoint 边界不变。
- 有界预算首次耗尽不阻塞，提示与状态栏可见；宽限耗尽暂停。`unbounded` 不因预算或三次终审拒绝暂停，但仍能因真实用户决策死锁进入 `paused(agent_blocked)`。
- 确定性验证至少通过 `npm run test:context`、`npm run test:rpc`、`npm test`；TUI 交互按项目约定人工 smoke。

## 非目标

- 不新增 Plan Extension、第二套计划状态机、后台 worker、跨 session 目标、定时循环或多 goal 池。
- 不把主 agent 验证包或自测声明升级为独立审核证据。
- 不为无上限策略移除模型错误、无进展、审核错误和用户决策死锁的暂停出口。
