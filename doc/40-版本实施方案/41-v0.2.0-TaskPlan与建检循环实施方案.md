# v0.2.0 Task Plan 与建检循环实施方案

> **版本归属**：0.2.0
> **状态**：准备实施
> **一句话目标**：把 dgoal 从"单 objective 持续跑到完成"升级为"goal/phase/task 三层 + 建检循环"——启动闸门确认计划、loop 内按需分解推进、每阶段独立 check、终审通过才结束，并在 editor 上方用计划浮层显示 phase 进度。

## 1. 背景

0.1.x（现状）是"轻量 durable goal loop"：一个 `objective` + `loop_complete` 审核 + 自动续跑。三个摩擦点（见 `doc/10-架构与运行/` 和归档底稿）：

- 只有"第几轮"没有"第几步"（`formatStatus` 只输出 `🔁 active #N`）。
- Goal 只有 objective 文本，没有"打算怎么做"的结构。
- 没有计划浮层，全程一行 status bar。

0.2.0 把用户可见主对象从"goal 迭代轮数"改成**计划推进（phase 进度）**，并确立**建检循环**为第一性原理。

## 2. 核心定义

| 概念 | 0.2.0 定义 |
|---|---|
| 建检循环（Build-Check Loop） | dgoal 基本盘：定义 goal + 完成后 check，不过继续干，过则结束。心智模型，不建模（ADR 0006） |
| Goal（目标） | 用户 `/dgoal` 发起的全球目标，启动闸门确认后冻结 |
| Phase（阶段） | 阶段性目标，状态由其下 task 聚合；标 completed 唯一入口是 `dgoal_check` |
| Task（任务） | 按需递归分解（ADaPT）的细粒度单元，blockedBy 依赖图，默认对用户隐藏 Ctrl+O 展开 |
| 启动闸门 | `/dgoal` → 主代理 `dgoal_propose` 提交计划 → 确认 UI（确认/拒绝/反馈）→ 进 loop |
| 阶段建检 | phase 标 completed 时触发 `dgoal_check`（独立只读子进程），通过才 completed |
| 终审 | 最后一个 phase 标 completed 时，check 升级为审全 goal；即 `dgoal_done` 内部走的 check |

## 3. 目标

0.2.0 做 8 件事（垂直切片，每步 TUI 可验证）：

1. **数据模型 + 持久化**：扩 `LoopGoal` 加 `plan`(Phase/Task) + `pauseReason` + `rejectedCount` + `verification`。复用 `dgoal-state` custom entry。能建/存/恢复。
2. **dgoal_plan 工具**：task/phase CRUD（reducer 平移 rpiv-todo，去 tombstone 加 blocked，带 blockedBy 环检测）。
3. **aboveEditor 计划浮层**：抄 rpiv-todo overlay。显示 phase 进度，task 默认隐藏，Ctrl+O 展开。
4. **启动闸门 + dgoal_propose**：`/dgoal` → 主代理整理 plan → `dgoal_propose` → `ctx.ui.select` 确认 UI → 进 loop。含兜底（拷问 25：重试 2 次失败中止）。
5. **dgoal_check 阶段建检门**：phase completed 走 dgoal_check（spawn 独立子进程）。通过 completed，不过回 in_progress。
6. **dgoal_done + 终审 + 状态机**：`loop_complete` 改名 `dgoal_done`；终审走 dgoal_check；rejected/paused/pauseReason/rejectedCount×3 状态机。
7. **prompt 调和**：dgoal_plan/dgoal_check/dgoal_done guidance + before_agent_start 注入 plan 上下文 + 续跑 prompt 钉 rejected 问题。调和"先拆 plan"与"别停在计划"。
8. **全面改名同步**：`loop_complete`→`dgoal_done` 在 index.ts（prompt 文本 6 处）+ README + doc。

## 4. 非目标

0.2.0 不做：

- 不做跨 session goal / 多 goal 池（会话内单目标边界，AGENTS.md）。
- 不做任意深度嵌套树（用 blockedBy 依赖图涌现分解，ADR 0006）。
- 不做 plan mode 的"模式切换"（plan 是 goal 内结构，非独立模式）。
- 不做 `dgoal_pause` 工具（拷问 26 排除，blocked + 终审×3 传导足够）。
- 不做 evidence 结构化增强（现在文本字段，未来按真实痛点再做）。
- 不做建检历史可视化（建检循环心智模型刻意不建模，ADR 0006）。
- 不做 i18n（prompt/输出中文为主）。
- 不做用户手写 `.dplan.*`（plan 由 agent 在启动闸门生成并迭代，活在 custom entry 里）。

## 5. 切片拆解与验收标准

### 切片 1：数据模型 + 持久化

**范围**：
- 扩 `LoopGoal`：加 `plan?: TaskPlan`、`verification?: string`、`pauseReason?`、`rejectedCount?`。
- 新增 `TaskPlan` / `Phase` / `Task` 接口（见 `10-架构与运行/12`）。
- `persistGoal`/`loadGoal` 自然带上 plan（复用现有 custom entry，无需改持久化机制）。
- `isLoopGoal` 校验加 plan 字段容忍（旧 goal 无 plan 仍可加载）。

**验收**：
- 建 goal + plan，persistGoal 后 loadGoal 能完整恢复三层。
- 旧 dgoal-state entry（无 plan）仍能 loadGoal（向后兼容）。
- 现有测试 `test:rpc` + `test:context` 通过。

### 切片 2：dgoal_plan 工具

**范围**：
- 新增 `dgoal_plan` 工具，reducer 平移 rpiv-todo（`create/update/list/get`）。
- 适配：操作对象是 task（phase 内），去 tombstone，加 `blocked` 状态 + `blockedReason`。
- `blockedBy` 增量合并（add/remove）+ 环检测（detectCycle 预检）+ 自环拒绝 + 跨 phase 依赖。
- `evidence` 字段：标 completed 时可带（可复验形态）。
- 纯本地快操作，不 spawn 子进程。
- **不能标 phase completed**（留给 dgoal_check）。

**验收**：
- 建 task、改 status（pending⇄in_progress→completed，blocked 可回退 in_progress）、加/删 blockedBy。
- 环依赖被拒，自环被拒，blocked 必带 reason。
- 工具 details 持久化（persistGoal 带 plan）。

### 切片 3：aboveEditor 计划浮层

**范围**：
- 新增 `PlanOverlay` 类（抄 rpiv-todo `todo-overlay.ts`）。
- `setWidget("dgoal-plan", factory, { placement: "aboveEditor" })`。
- 显示 phase 进度（heading `🎯 objective (X/Y)` + 每行 phase）。
- task 默认隐藏，`Ctrl+O` 展开（需注册 keybinding 或监听）。
- completed 闪现（两 Set，agent_start 搬运）+ 12 行折叠。
- 刷新：`tool_execution_end`(dgoal_plan/dgoal_check) + `agent_end`。

**验收**：
- 有 plan 时浮层显示 phase 进度，符号 ○/◐/✓/⚠ 正确。
- task 默认不显示，Ctrl+O 展开。
- 空时自动隐藏。
- completed 闪现到下一轮 agent_start。

### 切片 4：启动闸门 + dgoal_propose

**范围**：
- 新增 `dgoal_propose` 工具：参数 `{ objective, verification?, phases: [{subject, description?, tasks?[]}] }`。
- execute：存到模块变量（pending proposal），触发确认 UI。
- `/dgoal <objective>` 改流程：先发指令让主代理读代码 + 调 dgoal_propose 提交，而非直接进 loop。
- `agent_end` 检测：本轮是否调了 dgoal_propose？调了 → `ctx.ui.select` 确认 UI（确认/拒绝/反馈）。
- 确认 → goal pending→active，发 START prompt 进 loop。
- 反馈 → `ctx.ui.editor` 取反馈 → `sendUserMessage` 喂回主代理 → 重新整理。
- **兜底（拷问 25）**：没调 dgoal_propose → 降级提示重试 2 次（共 3 次机会）→ 仍无 → 中止（清 goal，不进 active）。

**验收**：
- `/dgoal` 后主代理产出 plan，确认 UI 正确弹出 goal + phases。
- 确认进 loop，拒绝中止，反馈能多轮精炼。
- 主代理不调 propose 时，重试 2 次后中止，不卡死。

### 切片 5：dgoal_check 阶段建检门

**范围**：
- 新增 `dgoal_check` 工具：参数 `{ phaseId }`。
- execute：spawn 独立只读子进程（复用现有 auditor spawn 模式，`--no-session --mode json` + 只读工具）。
- 审该 phase 的成果（其下 task 的 evidence + phase subject）。
- 通过 → phase 进 completed（dgoal_check 内部改状态）。
- 不过 → phase 回 in_progress，审报告注入对话。
- prompt 强调"独立 FAIL 权限，不偏袒"（建检模式启示）。

**验收**：
- 调 dgoal_check(phaseId) 触发独立子进程审计。
- 通过则 phase completed，不过回 in_progress + 报告注入。
- dgoal_plan 无法直接标 phase completed（不变式守住）。

### 切片 6：dgoal_done + 终审 + 状态机

**范围**：
- `loop_complete` 改名 `dgoal_done`（工具名 + 全部 prompt 文本）。
- `dgoal_done` 内部：检查是否最后一个 phase → 是则走 dgoal_check 终审模式（审全 goal + verification）。
- 终审通过 → goal done，发完成信号。
- 终审不过 → goal rejected，rejectedCount++，prompt 钉问题。
- rejectedCount×3 → paused(audit_failed_3x)。
- resume 按 pauseReason 决定清不清零（audit_failed_3x 清零，其他不清）。
- `LoopStatus` 加 `rejected`、`done`（替 `complete`）。

**验收**：
- dgoal_done 触发终审，通过关 goal。
- 终审不过进 rejected，3 次进 paused(audit_failed_3x)。
- resume(audit_failed_3x) 清零重试，resume(其他) 不清零。
- 状态栏显示 rejected ×M。

### 切片 7：prompt 调和

**范围**：
- `dgoal_plan` guidance：建 plan 后立即执行第一步并标 in_progress；完成立即标不 batch；evidence 用可复验形态。
- `dgoal_check` guidance：独立 FAIL 权限。
- `dgoal_done` guidance：所有 phase completed（终审通过）才调。
- `buildSystemPrompt`："不要停在计划"改为"不要停在**纸面计划**"，明确建 plan 允许、停 plan 不动不允许。
- `before_agent_start`：注入当前 plan 快照（三层，AI 可见）。
- 续跑 prompt：rejected 时钉"上次终审未过，问题：X"。

**验收**：
- 人工 smoke test：agent 建计划后立即执行不停留。
- rejected 时续跑 prompt 带未过问题。

### 切片 8：全面改名同步

**范围**：
- index.ts：`loop_complete` → `dgoal_done`（工具名 + 6 处 prompt 文本）。
- `LoopStatus`：`complete` → `done`。
- README.md / README-zh.md / AGENTS.md / doc 同步。

**验收**：
- grep `loop_complete` 仅剩"原 loop_complete 已改名"的说明性引用。
- grep `"complete"` 状态名无残留。
- 测试通过。

## 6. 实施顺序与依赖

```
1 (数据模型) ──→ 2 (dgoal_plan) ──→ 3 (浮层)
                                      ↓
4 (启动闸门+dgoal_propose) ──→ 5 (dgoal_check) ──→ 6 (dgoal_done+状态机)
                                                          ↓
                                                    7 (prompt) ──→ 8 (改名同步)
```

切片 1 是所有的基础。2/3/4 可在 1 之后相对独立推进。5 依赖 2（phase/task 模型）。6 依赖 5（dgoal_check 终审模式）。7/8 收尾。

## 7. 验证策略

- **自动化**：每个切片补对应单元测试（reducer 纯函数可测、状态机转换可测、持久化往返可测）。`test:rpc` 扩展覆盖新工具/命令注册。
- **人工 smoke test**：完整建检循环行为（启动闸门确认 → loop 推进 → 阶段建检 → 终审 → rejected 回环 → paused）需在 Pi TUI 用真实模型验证。这是 AGENTS.md 明确的边界。
- **回归**：现有 `test:context`（背景固化）+ `test:rpc` 必须持续通过。

## 8. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 主代理启动闸门不产出 plan | 拷问 25 兜底：重试 2 次失败中止 |
| agent 滥用 task blocked 偷懒 | dgoal_check 审 block reason（真 blocker 还是偷懒）+ 终审×3 兜底 |
| 建检子进程成本（每 phase 一次） | 仅 phase 级 check（task 级不 check），颗粒度已优化 |
| Ctrl+O 展开与 pi-tui 交互 | 参考 rpiv-todo widget 模式，必要时降级为"长按/命令展开" |
| loop_complete 改名破坏现有用户 | 0.1.x 用户少（0.1.3 刚发），改名窗口期可接受；改名安全（工具名非持久化键） |

## 9. 版本号说明

0.1.x → 0.2.0 是 minor bump（向后兼容的数据模型扩展 + 新功能），非 breaking major。旧 dgoal-state entry 无 plan 字段仍可加载（切片 1 的向后兼容验收）。工具改名 `loop_complete`→`dgoal_done` 是 agent-facing 变更，但 0.1.x 用户基数小，在 0.2.0 一次性完成。
