## 模型把数组工具参数 stringify 成字符串，pi-ai 校验报 `*.0: must be number`

**现象**：模型（LLM）调用 `dgoal_plan` 时，把 `blockedBy` / `addBlockedBy` / `removeBlockedBy`（以及 `dgoal_propose` 的 `phases[].tasks[].blockedBy`）序列化成字符串 `"[]"` / `"[1,2]"`，而非真正的数组。pi 主程序的 pi-ai 入参校验层（`validateToolArguments`）在工具 `execute` 之前跑，先 `TypeBox` `Value.Convert` 再 `Compile.Check`：`Value.Convert` 把字符串转成类数组结构，Ajv 按元素逐项校验报 `blockedBy.0: must be number`，调用被拒，reducer 根本没机会执行。reducer 里原本的 `Array.isArray(...) ? ... : []` 防御在校验层之后，救不了。

关键：报错路径是 `blockedBy.0`（带 `.0` 索引）而非 `blockedBy: must be array`——这是 `Value.Convert` 把字符串当数组遍历的特征，看到 `.0` 就该怀疑入参被 stringify 了。

**做法**：用框架的 `prepareArguments` 钩子（`ToolDefinition.prepareArguments`），它在 `validateToolArguments` 之前执行，是专门为"校验前规整模型坏输入"设计的接缝。在 `prepareArguments` 里把字符串化的数组 `JSON.parse` 回 `number[]`，schema 保持严格 `Array<number>` 不放宽对 LLM 的契约（放宽成 `Union([Array, String])` 会让 LLM 误以为字符串是合法形态，反而鼓励 stringify）。reducer 入口同时保留 `coerceNumberArray` 兜底作为防御性二次清洗（直接调 `execute` 的测试路径绕过 `prepareArguments`）。

不要在 reducer 里加兜底就收工——校验在 reducer 之前，reducer 兜底救不了被校验拒掉的调用。要么用 `prepareArguments`（推荐，接缝正确），要么放宽 schema（不推荐，污染契约）。

**证据**：
- `index.ts` `dgoalPlanTool.prepareArguments` / `dgoalProposeTool.prepareArguments` + `coerceNumberArray` helper。
- `test/prepare-arguments-schema.test.ts`：`prepareArguments` 输出过严格 `Compile.Check`。
- 端到端复现：用真实 `pi-ai` `validateToolArguments` + 真实 `prepareArguments`，`blockedBy: "[]"` / `"[1,2]"` / `addBlockedBy: "[1]"` / propose `tasks[].blockedBy: "[1]"` 全部过校验并 coerce 成真数组。
- 2026-07-07 首次遇到，触发场景：agent 跑 `/dgoal` 建 task 时模型把空依赖序列化成 `"[]"`。

## `/tree` 导航后浮层/goal 不重同步——`navigateTree` 不发 `session_start`

**现象**：用户在跑 dgoal 时执行 `/tree` 切到另一分支后，计划浮层停在旧状态——阶段明明建检完成了还显示未完成，计时器也冻住。`/fork` 不受影响。

**做法**：查 pi 主程序 `agent-session.js` 的 `navigateTree`——它是**原地切 session 分支**，只发 `session_before_tree`（可取消）+ `session_tree`（通知）两个事件，**不发 `session_shutdown`/`session_start`**（对比 `fork`/`new`/`resume`/`switchSession` 都走 `teardownCurrent` + `createRuntime` 发 session_start）。所以 dgoal 扩展若只监听 `session_start`，tree 之后 `currentGoal` 永不重新 `loadGoal`、overlay 永不重同步。

修法：监听 `session_tree`，与 `session_start` 共用同一套重同步逻辑（抽 `resyncGoalFromSession`：清 continuation/check snapshot/auditor tracker + `loadGoal` + setStatus + overlay 重同步），保证两路径不分叉。UI 抛错用 try/catch 不阻断状态重同步。

关键认知：**不是所有 session 导航都发 `session_start`**。`/tree` 是例外——它不重建 runtime，只换分支。以后新增「跟 session 走的状态」都要同时监听 `session_tree`。

**证据**：
- `index.ts` `resyncGoalFromSession` + `pi.on("session_tree")` / `pi.on("session_start")` 共用。
- `test/session-tree-resync.test.ts` 4 case。
- pi 主程序证据：`agent-session.js` `navigateTree` 无 `teardownCurrent`/`createRuntime` 调用，只 `emit({type:"session_tree"})`。
- 2026-07-07 首次遇到。

## `startGoal` 加 abort 后 `agent_end` 与 propose 投递双发撞车

**现象**：给 `startGoal` 加「非 idle 时 `ctx.abort` 暂停当前 LLM」后，被中断 turn 的 `agent_end` 会看到刚设的 pending goal，走 `handleStartupGate` 的兜底重试路径再发一次 propose prompt，与 `startGoal` 自己的 `sendPrompt(buildProposePrompt)` 撞车→双发 propose。

**做法**：用 `startGoalInProgress` 标志包住「创建 pending goal → 投递 propose prompt」整段（try/finally），`agent_end` 的 pending 分支看到标志即 `return` 跳过 `handleStartupGate`，让 `startGoal` 自己独占 propose 投递。标志在 finally 里清零——卡 true 会永久抑制 `handleStartupGate` 导致启动闸门锁死（有专门测试守这个不变量）。

关键认知：**`agent_end` 的 pending 分支是「主代理本轮没产出 proposal」的兜底重试入口**，任何在 `startGoal` 初始化期间引发 `agent_end` 的操作（abort/换 goal）都会触发它。`startGoal` 要么不引发额外 `agent_end`，要么用标志抑制期间被误触发的兜底。

**证据**：
- `index.ts` `startGoalInProgress` 标志 + `startGoal` try/finally + `agent_end` pending 分支 `if (startGoalInProgress) return`。
- `test/startgoal-abort.test.ts` 4 case（abort 调用 + 标志清零 + propose 恰好一次）。
- 2026-07-07 首次遇到。

## pi 0.80.3 `aboveEditor` widget 不渲染——浮层/状态栏片段凭空消失

**现象**：用户报告「持续显示浮层不见了，上几个版本还有」。同时消失的还有 footer 里 dgoal 通过 `setStatus` 注册的那格 status 片段；但 `/dgoal s` 弹出的 overlay modal（`ctx.ui.custom`）能正常显示且计时正常。第一反应会怀疑 pi-dgoal 代码 regression 或 `currentGoal` 被清空，但都不是。

**做法（诊断反馈环，按顺序）**：
1. **先排除 pi-dgoal 侧**：`git diff v0.5.2..HEAD -- index.ts | grep setWidget` 证明 `renderPlanLines`/`update`/`setWidget` 显隐逻辑零改动（只有 `LoopGoal→GoalState` 类型重命名）；`bun test test/plan-overlay-render.test.ts` 全 pass；用法对照 pi 官方 `docs/tui.md` Pattern 4/5 与 `docs/extensions.md#widgets-status-and-footer` 完全一致（`setStatus`→footer 片段、`setWidget(key,lines,{placement:"aboveEditor"})`，`"aboveEditor"` 是 RPC types 明确的合法值）。代码侧没问题就别改代码。
2. **用 `/dgoal s` 区分「状态问题」vs「渲染问题」**：modal 能弹+计时正常 = `currentGoal` 有值、扩展加载正常、pi 的 overlay 渲染正常 → 浮层消失是**常驻 widget/status 渲染层**的问题，与 dgoal 状态机无关。这条判据最省时间。
3. **placement A/B 验证**：临时把 `PlanOverlay.update()` 里 `setWidget` 的 `placement` 从 `"aboveEditor"` 改成 `"belowEditor"` 重载——浮层立刻在编辑器下方恢复显示 → 坐实 `aboveEditor` 单点失效（同一份代码、同一个会话，只换 placement 值）。
4. **确认 regression 范围**：降级 `npm i -g @earendil-works/pi-coding-agent@0.80.2` 复测 aboveEditor（0.5.2 的 CHANGELOG 记录在 0.80.2 录到过 aboveEditor 浮层展开态）。若 0.80.2 正常 → 锁定 0.80.3 引入，报 pi issue 精准。

关键认知：**浮层/状态栏片段不显示时，先别动 pi-dgoal 代码**。pi 的 `ctx.ui` 有三套独立渲染路径——overlay modal（`ui.custom`）、footer status 片段（`setStatus`）、editor 上/下 widget（`setWidget`）。三者用 A/B 对比能快速定位是哪套挂了。本例 overlay 正常、aboveEditor widget + footer status 同时失效 = pi 主程序常驻渲染层 bug。pi-dgoal 侧无法修 aboveEditor（是主程序渲染问题），只能 belowEditor 兜底或降级 pi。

**证据**：
- pi-dgoal 侧：`git diff v0.5.2..HEAD` 浮层逻辑零改动；`plan-overlay-render.test.ts` 20/20 pass；`index.ts` 的 `PlanOverlay.update()` 调用 `setWidget(…,{placement:"aboveEditor"})`。
- pi 主程序：`package.json` peerDep `pi-coding-agent/pi-tui: ^0.79.10`，实际跑 0.80.3（pi-tui 0.80.3），peerDep 不满足；0.80.3 CHANGELOG 顶部 #6026「status indicators 在 clear-on-shrink 下压缩 TUI 的修复」显示这块在活跃改动。
- 实测三路径对比：`aboveEditor` 不渲染 / `belowEditor` 渲染 / `/dgoal s` overlay modal 正常。
- 2026-07-08 首次遇到，触发场景：用户升级到 pi 0.80.3 后浮层消失。

## 真实 smoke 被外部 `SIGTERM` 终止后遗留认证副本

**现象**：候选回退 smoke 为了制造 ZAI `401`，临时复制 `auth.json` 并启动 Pi。Pi 的审核子进程可能继承 stderr pipe（管道）；若 driver 用阻塞 `stderr.read()` 收尾，外部 runner 等待超时后发送 `SIGTERM`。Python 默认不会在 `SIGTERM` 时展开 `finally`，导致临时目录残留完整认证副本。即使普通 smoke 和启动失败回归都通过，也覆盖不到这条“真实子进程运行中被终止”路径。

**做法**：Pi 用独立进程组启动，收尾时终止整个组并非阻塞读取 stderr；把 `SIGTERM` / `SIGINT` 转为 `SystemExit` 以执行外层 `finally`。认证副本遵循最小权限：只复制真实 smoke 所需的 ZAI 与 MiniMax provider，ZAI key 改为无效值，文件以 `0600` 写入。用假 Pi 长驻的确定性测试发 `SIGTERM`，同时断言退出码、认证内容和所有临时目录。

**证据**：
- `test/test-auditor-fallback-smoke.py`：独立进程组、非阻塞 stderr、signal 清理与最小认证副本。
- `test/test-auditor-fallback-cleanup.py`：假 Pi + `SIGTERM` 回归；`npm run test:smoke:cleanup` 不消耗 token。
- 2026-07-11 首次遇到；此前真实终审曾发现 `pi-dgoal-fallback-agent.* / auth.json` 残留。

## 可省略的逐 phase 数组必须先按 proposal 基数补齐

**现象**：`final_only` 允许语义预审省略 phase 验收条件。真实 reviewer 合法返回 `phaseAcceptanceCriteria: []` 时，rewrite 校验却仍按 proposal 的 phase 数遍历 `rewrittenLayers[layer]`；第二层开始取到 `undefined`，工具直接抛出 `rewrittenLayers[layer] is not iterable`，pending goal 无法进入确认 UI。

**做法**：区分“输出数组为空”和“proposal 没有 phase”。对 `final_only`，先把 reviewer 提供的 phase 数组按 proposal phase 基数逐项补齐，缺失项保留原 phase 条件；若 reviewer 返回比 proposal 更多的 phase 层则 fail-closed。补齐后再做逐层新增、删除、迁移校验，并保留层数一致断言，不能在循环里假设索引一定存在。

**证据**：
- `src/runtime/index.ts` `validateSemanticReviewShape()` 的 `suppliedReviewedPhases` 归一化与 layer-count 防御。
- `test/final-only-proposal-path.test.ts`：reviewer 返回空 `phaseAcceptanceCriteria: []` 的真实 proposal 回归。
- 2026-07-15 首次遇到；Curio pending goal 连续三次被该 TypeError 阻断。

## Pi `input` transform 没有不可变原文字段

**现象**：自然语言启动若把 `input` 事件的 `event.text` 当成“用户键入的不可变原文”，会高估安全边界。Pi 按扩展加载顺序串联 transform：前一个 handler 返回的新文本就是后一个 handler 看到的 `event.text`，`before_agent_start.prompt` 也是最终文本；API 没有 `originalText`。因此后加载扩展的 transform 可通过精确 prompt 绑定识别，早于 dgoal 的 transform 则无法区分。

**做法**：只接受空闲 `interactive` / `rpc`，拒绝 `source=extension` 与 `steer/followUp`；把 dgoal 实际观察到的 input 精确保存到一次性内存状态，`before_agent_start` 不重新做宽松意图识别，而是要求 prompt 完全一致。文档明确早于 dgoal 的 transform 属于 Pi 扩展全权限信任边界——恶意扩展本就能执行任意本机代码，dgoal 不能冒充 extension sandbox。确定性入口仍保留 `/dgoal`。

**证据**：
- Pi 0.80.7 `ExtensionRunner.emitInput()`：`currentText` 按 handler 依次改写，`InputEvent` 无原文字段。
- `src/goal-runtime/state.ts` 的 `naturalLanguageStartInput` 与 `src/startup/index.ts` 的精确 prompt 绑定。
- `test/no-progress-agent-end.test.ts`：extension/mid-run 拒绝、后续 prompt transform 清除授权。
- 2026-07-15 dteam 安全审查首次指出。
