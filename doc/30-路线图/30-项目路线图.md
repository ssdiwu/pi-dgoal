## 实现切片（垂直切片，每步 TUI 可验证）

按依赖关系排序，每步端到端可验证：

1. **数据模型 + 持久化**：扩 `GoalState` 加 `plan`（Phase/Task）+ `pauseReason` + `rejectedCount` + `verification`。复用 dgoal-state custom entry。能建/存/恢复。
2. **dgoal_plan 工具**：task/phase CRUD（reducer 平移 rpiv-todo，去掉 tombstone，加 blocked）。能建 task、改状态、加 blockedBy（带环检测）。
3. **aboveEditor widget**：抄 rpiv-todo overlay。显示 phase 进度，task 默认隐藏。`Ctrl+O` 展开 task。
4. **启动闸门 + dgoal_propose**：`/dgoal` → 主代理整理 plan → `dgoal_propose` 提交 → `ctx.ui.select` 确认 UI（确认/拒绝/反馈）→ 确认进 loop。
5. **dgoal_check 阶段建检门**：phase 标 completed 走 dgoal_check（spawn 独立子进程）。通过标 completed，不过回 in_progress。
6. **dgoal_done + 终审 + 状态机**：loop_complete 改名 dgoal_done；终审走 dgoal_check；rejected/paused/pauseReason/rejectedCount×3 状态机。
7. **prompt 调和**：dgoal_plan/dgoal_check/dgoal_done guidance + before_agent_start 注入 plan 上下文 + 续跑 prompt 钉 rejected 问题。调和"先拆 plan"与"别停在计划"。
8. **loop_complete → dgoal_done 全面改名**：index.ts prompt 文本 6 处 + README/doc 同步。

## 待拷问项已全部完成（拷问 1-26）

26. **dgoal_propose 兜底（拷问 25，已决）**：主代理不调 dgoal_propose 时，降级提示重试 2 次（首次 propose 指令 + 2 次降级引导 = 3 次机会），仍无产出则中止（goal 不进 active，不污染状态机）。**不走"进 loop 自建"的纯自主兜底**——与拷问 6 否决纯自主一致。

27. **dgoal_pause 工具（拷问 26，已决）**：**不加**。task blocked + 终审 ×3 传导已覆盖 agent 表达卡住的场景。plan 是启动闸门前用户和 AI 已对齐的，进 loop 还动不动 pause 求救是没尽到责任；机制要逼 agent 在 loop 里穷尽尝试，3 次终审不过才停。pause 是用户的能力（`/dgoal pause`），不是 agent 的逃生口。未来若实测发现 agent 反复 blocked 同组 task 烧 token 无进展，再考虑加（YAGNI）。

## v0.5.2 已实现

- **建检协议重建（事件流活性 + 闸门锁定）**：已纳入 v0.5.2。`dgoal_check` 从“最终报告导向”重建为“事件流导向”，父进程显式消费 `thinking_*`、`toolcall_*`、`text_delta`、`message_end` 等事件，产出正式建检活性状态；阶段建检不过/无结论/审核器异常时 goal 保持 `active`，但进入建检闸门锁定，不允许越阶段推进或越终审推进。设计决策见 `../决策档案/0012-阶段建检闸门锁定与事件流化审核器.md`。
- **裸 `/dgoal` 承接前文共识启动**：已纳入 v0.5.2。裸 `/dgoal` 走启动闸门（路径 B：命令层发承接信号，agent 用 `dgoal_propose` 定 objective；`summarizeContext` 仍只产 `contextSummary` 背景）；看状态统一用显式 `/dgoal s`。前文为空不硬启动，提示用 `/dgoal <objective>` 或先 grill。

## 本轮进行中（未发版）

- **dgoal_propose 就绪度自检（L0-L3 字段物证 + 缺口提示）**：已在本轮实现。`dgoal_propose` 现已支持 `nonGoals` / `guardrails` / `budget` 作为 plan 级信号，启动闸门确认界面会展示 L0-L3 等级与缺口提示，优先暴露 `non-goals` 边界不足。仍待整轮目标验收、README/doc 全量同步与 smoke 验证后再进入发布切片。
- **dgoal_done 产出可核对的理解文本（降理解债）**：已在本轮实现。`dgoal_done` 现已支持可选 `whatChanged`（改动清单）与 `userReview`（仍需用户核对），完成回复信号会产出结构化的可核对文本，终审输入也包含改动清单。仍待整轮目标验收与 smoke 验证后再进入发布切片。
- **审核模型候选链与错误回退**：已在本轮实现并验证。`phaseAuditorModels` / `goalAuditorModels` 分别为阶段建检和目标终审提供最多 3 个有序 `provider/model[:thinking]` 候选；项目候选链整体优先、不混合来源，复数字段 > 单值字段 > 旧 `auditorModel`，`null` 显式继承。模板、格式校验、结构化 Pi 注册表预检、配置降级、运行时技术/协议异常回退（HTTP 400/401/403/404/408/429/5xx、网络、零输出超时、无终止标记的部分输出，以及明确的纯文本配额耗尽）均按候选顺序单次切换；有效 fallback 候选按 goal 与审核范围持久复用，并保留可追溯尝试轨迹与候选耗尽 `audit_error` 暂停。明确 `<APPROVED>` / `<REJECTED>` 与用户中断不切换，绝不静默回退执行模型。确定性测试全量通过；决策见 ADR 0028。

## 后续候选

- **持续计划浮层彩色化**：本次先不把 aboveEditor widget 升级为 theme-aware 彩色 widget，只统一状态字符与结构；下一版本再单列切片，专门处理层级基色渲染、跨浅/深主题可见性和 TUI 渲染回归测试。决策见 `决策档案/0009-TUI视觉编码改为层级靠颜色状态靠字符.md`。
- **运行可观测性（`/dgoal s` 运行计数）**：候选增强详细查询 Modal，展示当前 goal 的运行轮次、phase check 次数、final audit 次数、最近审核耗时与重试次数。先做不依赖 Pi core token 统计的轻量 counters；token / cost 只有在 Pi core 提供稳定接口或出现真实“烧 token 不可见”痛点后再接。
- **结构化运行时预算（Budget hard cap）**：把现有声明性 `budget` 逐步演进为可执行上限的候选，例如 `maxTurns` / `maxPhaseChecks` / `maxFinalAudits` / `maxWallClockMinutes`。触发上限后进入 paused，并给用户“已尝试什么 / 卡在哪里 / 建议下一步”。这会影响状态机和数据模型，实施前需补 ADR；没有真实空转样例前不排近期切片。
- **loop 命名清理（已完成）**：已在本轮统一处理。类型名 `LoopGoal`→`GoalState`、`LoopStatus`→`GoalStatus`、`LoopContext`→`DgoalContext`、`LoopStateEntryData`→`DgoalStateEntryData`；函数名 `isLooping`→`isGoalRunning`、`handleLoopCommand`→`handleDgoalCommand`；prompt 注入标签 `<loop_*>`→`<dgoal_*>`。代码、测试与当前权威文档（术语表 / 10-架构与运行 / 路线图）均已同步。归档与决策档案保留原历史命名，不追溯。

## 暂不做 / 远期

- **跨 session goal**：dgoal 会话内单目标（AGENTS.md 边界）。跨 session 是 dteam 的范畴，不在 dgoal。
- **多 goal 池**：同上，不做。
- **循环工程（定时循环节奏）**：建检循环目前只套在“直到完成”节奏（会话内单 goal）。远期可延展到“定时循环”节奏——定时跑、每轮建检通过才算完成那轮，对应循环工程的运维场景。参考开源实践（Cobus Greyling `loop-engineering` 的 7 个 patterns）：每日巡检（daily-triage）、PR 看管（PR babysitter）、CI 清理（CI sweeper）、依赖扫描（dependency sweeper）、起草更新日志（changelog drafter）、合并后清理（post-merge cleanup）、Issue 处理（issue triage）。按“先证据后优化”，真实需求出现前不排实现切片。
- **i18n**：A-line 用户可见文案已接 `pi-di18n` 软依赖；`pi-di18n` 不在时降级内置中文。模型侧 prompt / tool schema 仍保持中文为主，不做本地化。
- **evidence 的结构化增强**：现在 evidence 是文本字段，未来可能结构化（命令+预期输出对），待真实痛点出现再做。
- **建检历史可视化**：现在 check 结果落对话/状态，未来若要"回看每次 check 结果"再考虑独立存储（建检循环心智模型刻意不建模，ADR 0006）。

## 不做（明确排除）

- 不自动 Git 提交/回滚/删除。
- 不替代项目自身测试命令。
- 不做 plan mode 的"模式切换"（dgoal 把 plan 做成 goal 内结构，不引入模式概念）。
- 不做任意深度嵌套树（用 blockedBy 依赖图涌现分解，ADR 0006）。
