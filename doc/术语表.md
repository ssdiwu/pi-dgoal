# pi-dgoal 术语表

pi-dgoal 是一个 Pi 扩展，让 agent 围绕一个明确 goal（目标）持续工作，在**建检循环**里推进，直到完成并通过独立 check。

## 核心原理

**建检循环（Build-Check Loop）**：
dgoal 的第一性原理。**dgoal 定义了 goal，定义了完成后要 check——不通过则继续干活，通过则结束。**

建检循环有两个验收策略：
- **phased 阶段建检**（小建检）：一个 phase 要标 done 时触发，审该 phase 成果。通过则 phase 进 done，不通过则回 in_progress 继续干；phase done 唯一入口是 `dgoal_check`。
- **final_only 阶段进度 + 终审**：phase 只用 `complete_progress` 记录执行进度，不伪装成独立审核；所有 phase 进度完成后，由 `dgoal_done` 触发一次 goal 级独立审核。终审通过则 goal 进 done，不通过则 goal 进 rejected。

建检循环是**心智模型**，不是显式数据结构——它的两个实例（phase 门、终审回环）各有状态承载（phase 状态 / goal 的 rejected），不为心智模型重复建模。

**建检循环是验证约束，不是节奏。** “完成”必须通过独立检查（`dgoal_check` / 终审）才算数，不由 agent 自述；它比“直到完成”更高阶，已代替单纯的“直到完成”。此约束可套在两种循环节奏上：

- **直到完成（until-done）**：agent 持续运行、直到完成条件满足才停止的循环节奏。dgoal 现状即建检循环套在此节奏上（会话内单 goal）。
- **定时循环（cadence）**：agent 按固定节奏重复运行的循环节奏，不以单次“完成”为目标，适用于持续性巡检、看护类工作。建检循环套在此节奏上是 dgoal 的延伸方向（远期）。

**触发器（Trigger）**：
一次 loop 开始或再次运行的原因。可以是用户手动输入、定时调度、外部事件或 API 调用。dgoal 当前的触发器只有用户在 Pi 会话内输入 `/dgoal`、自然语言明确要求使用/启动 dgoal，以及 active/rejected goal 内部的续跑；不做定时触发、外部事件触发或云端例行触发。
_Avoid_: 把触发器和停止条件混为一谈；把 `/dgoal` 误解成 `/loop` / `/schedule` 这类定时或事件调度入口

**停止条件（Stop Condition）**：
一次 until-done loop 被允许停止的判据。dgoal 里的停止条件不是 agent 的自我感觉，而是启动闸门确认的 `acceptanceCriteria`（冻结完成门）、按策略执行的 phase 级 `dgoal_check` 或 `complete_progress`、以及 goal 级 `dgoal_done` 终审共同构成的验收口。`verification` 是 goal 级验收说明，帮助理解完成标准但不单独作为终审完成门（旧 session 缺少 `acceptanceCriteria` 时兼容沿用）。停止条件必须可核验；若只有愿望句，没有交付物、标准或证据，就不能进入 dgoal。
_Avoid_: 把“做完就停”“看情况”当成停止条件；把 `summary` 或 `contextSummary` 当成停止条件本身

**循环原语（Loop Primitive）**：
实现某类 loop 的基础能力或命令，例如 Claude Code 里的 `/goal`、`/loop`、`/schedule`，分别偏向停止条件、定时触发和云端例行程序。dgoal 只实现 Pi 会话内 goal-based build-check loop（目标导向建检循环）这一类原语，不追求覆盖所有 loop primitive。
_Avoid_: 因为外部工具有 `/loop` / `/schedule` 就把 dgoal 扩成调度器；把循环原语等同于 dgoal 的内部状态机

**循环工程（Loop Engineering）**：让 agent 按预设机制自主循环工作、替代人逐轮提示的工作范式。在 dgoal 体系中属于延伸支——建检循环套在“定时循环”节奏上的具象形态（巡检/看护 + 每轮建检）。

_Avoid_: 把建检循环当成独立机制/状态/循环结构去实现；把建检循环等同于“直到完成”——前者是验证约束（检查通过才完成），后者只是节奏（跑到完成条件即停）；把 dgoal 当前定位扩成覆盖所有循环原语的通用 loop 平台。

## 三层内容与双可见性轴

dgoal 自然有三层内容，但有**两条独立的可见性轴**：

| 层级 | 用户可见性（TUI） | AI 可见性（context） |
|---|---|---|
| **Goal**（全局目标） | 可见（持续显示浮层 heading / 详细查询 Modal heading） | 可见 |
| **Phase**（阶段） | 可见（持续显示浮层默认显示） | 可见 |
| **Task**（按需分解的细粒度执行单元） | **默认隐藏**；持续显示展开态只展开未完成 phase，详细查询 Modal 可看全量 | 可见（dgoal 把三层都喂给 AI） |

用户可见性轴：避免 TUI 被海量细 task 淹没（10 行折叠预算，预留 Pi core 截断提示空间）。用户平时主要看持续显示浮层里的 phase 进度；需要全量细节时，用 `Ctrl+O` 看持续显示展开态中的未完成 phase task，或用 `/dgoal s` 打开详细查询 Modal。
AI 可见性轴：AI 始终看全三层，知道每个 task 服务哪个 phase、哪个 goal——避免局部最优但全局无用。

## Language

**Goal（目标）**：
用户通过 `/dgoal` 命令或自然语言显式启动发起的全局目标。启动闸门确认后冻结，是整个 loop 的方向契约。
_Avoid_: objective（objective 是 goal 的文本简述字段）

**持续显示浮层**：
Pi TUI 展示面，归 **L3 常驻面板**，原语 `setWidget` + `placement:"aboveEditor"`。放在 editor 上方、持续可见，职责是持续显示 phase 主干进度；task 默认隐藏。它的展开态跟随 `Ctrl+O`，但只展开未完成 phase 的 task；done phase 只持久显示标题行。
_Avoid_: 把它和 `/dgoal s` 的详细查询 Modal 混叫成同一个"状态面板"；把持续显示浮层叫成 "overlay" / "Live Overlay"——`overlay` 专指 L2 的 `custom({overlay:true})`（详细查询 Modal），持续显示浮层是 L3 的 `setWidget`（widget），两者原语不同，不可混叫

**持续显示展开态**：
持续显示浮层在 `Ctrl+O` 打开后的展开形态。它不是"全量任务视图"，而是一个收敛视图：只展开 `pending / in_progress` phase 的 task；当前 phase 内已 done 的 task 仍显示，供 agent 与用户确认阶段内已完成项；done phase 的 task 不再显示。
_Avoid_: 说成"展开后显示所有 task"

**详细查询 Modal（Status Modal）**：
Pi TUI 展示面，归 **L2 自定义组件**，原语 `ctx.ui.custom(comp, {overlay:true})` + `anchor:"center"`。`/dgoal s` 打开的居中 overlay modal。职责是按需查看完整 plan 细节；它与持续显示浮层职责正交，保留全量 phase/task 细节，包括 done phase 的 task。
_Avoid_: 把它叫成"顶部浮层"，或误以为它应遵循持续显示展开态的裁剪规则；`overlay` 在 Pi 里专指 L2 的 `custom({overlay:true})`，不要把它和持续显示浮层（L3 `setWidget`）混叫

## TUI 展示面（对照 Pi 原生 UI 三层）

dgoal 的用户可见展示统一归到 Pi 原生 UI 三层 + 一个三层外的 toast，每个展示面固定对应一个原语，不混叫：

| 展示面 | Pi 层 | 原语 | dgoal 用途 |
|---|---|---|---|
| 启动闸门确认对话框 | L1 提问式 | `ui.select` / `ui.editor` / `ui.confirm` | 命令或自然语言显式启动时确认 / 拒绝 / 输入反馈 |
| 详细查询 Modal | L2 自定义组件 | `ctx.ui.custom(comp, {overlay:true})` + `anchor:"center"` | `/dgoal s` 按需查完整 plan（全量 phase/task） |
| 持续显示浮层（+展开态） | L3 常驻面板 | `setWidget` + `placement:"aboveEditor"` | editor 上方持续显示 phase 主干进度 |
| 状态栏 | L3 常驻面板 | `setStatus("dgoal", …)` | 底部 statusline 一行 goal 级状态 |
| notify 提示 | 三层外 toast | `ctx.ui.notify` | 一次性 toast（确认 / 中止 / 降级提示） |

**职责正交**：持续显示浮层（L3 setWidget）走收敛视图、常驻；详细查询 Modal（L2 custom overlay）保留全量、按需弹出查完即关。两者都不展示建检报告正文。

**L2 overlay vs L3 widget 不可混叫**：`overlay` 在 Pi 里专指 L2 的 `custom({overlay:true})`（详细查询 Modal）；持续显示浮层是 L3 的 `setWidget`（widget），不叫 overlay。

**启动闸门确认对话框（Startup Gate Dialog）**：
Pi TUI 展示面，归 **L1 提问式**，原语 `ui.select`（选项确认）/ `ui.editor`（反馈输入）/ `ui.confirm`（二选一）。命令或自然语言显式启动时，主代理用 `dgoal_propose` 提交 plan 后弹出，默认列 goal + verification + acceptanceCriteria + userReviewItems + readiness + 边界/缺口提示 + phases + task 数量，确认 / 拒绝 / 输入反馈三选一。阻塞、选完即关、内容短。
_Avoid_: 把它叫成"状态面板" / "Modal"——它是 L1 阻塞对话框，不是 L2 overlay modal

**状态栏（Status Bar）**：
Pi TUI 展示面，归 **L3 常驻面板**，原语 `setStatus("dgoal", …)`。底部 statusline 显示一行 goal 级状态：`🔁 active #N` / `🔁 paused` / `🔁 starting...` / `🔁 rejected ×M` / `🔁 done`。常驻不阻塞、随事件刷新。
_Avoid_: 把它和持续显示浮层混叫——两者都是 L3 常驻面板但原语不同（`setStatus` vs `setWidget`）、位置不同（底部 statusline vs editor 上方）、粒度不同（一行 goal 状态 vs phase 主干）

**notify 提示（Notification）**：
Pi TUI 展示面，归 **三层外 toast**（不是 L1/L2/L3 任何一层），原语 `ctx.ui.notify`。一次性瞬时弹窗，用于确认 / 中止 / 降级提示（如 proposal 确认、`clear` 清理、`custom` 不可用时降级）。不承载持续状态或详细内容。
_Avoid_: 用 notify 承载详细 plan 状态（那是详细查询 Modal 或持续显示浮层的职责）；把它算进 L1/L2/L3

**Objective（目标简述）**：
goal 的一句话文本描述，存在 `GoalState.objective`。

**GoalState（目标持久态）**：
dgoal 用来保存当前 goal 阶段性业务事实的主对象：如 objective、status、plan、verification、contextSummary、pauseReason、rejectedCount、建检反馈等。它是状态机与恢复的持久账本，不承载独立建检子进程的瞬时运行态。
_Avoid_: 把 GoalState 当成 TUI 展示状态、工具调用日志或运行时事件缓存

**Goal Runtime（目标运行态，vNext 已实现）**：
dgoal 在一个 Pi session 内的可变业务运行态及其生命周期协调者：持有当前 goal、持久化、命令、续跑与状态推进；不拥有 TUI 渲染、审核判卷或计划 reducer。
_Avoid_: 把 Goal Runtime 叫成泛化的 service，或让 TUI 直接持有其状态

**Isolated Pi Runtime（隔离 Pi 运行时，vNext 已实现）**：
调用 fresh context 的隔离 Pi 子进程、消费其 JSON 事件并进行模型可用性预检的公共运行能力；不决定 auditor 或 context summarizer 各自的重试和失败语义。
_Avoid_: 把它当作审核策略或背景总结策略本身

**Phase（阶段）**：
阶段性目标，展示状态由 task 的中间态聚合（有 in_progress 的 task→in_progress；有 blocked 且无 in_progress→blocked）；全 task 终态只表示该 phase 已具备建检条件，recompute 不会自动把 phase 写成 done。将 phase 正式记为完成的唯一入口是 `dgoal_check`（阶段建检门），通过后才写入 done。TUI 浮层默认显示的单元。尚未拆 task 的空 phase 允许直接标 blocked（整阶段判定不可行）。

最小复杂度规范（ADR 0017，**vNext 已实现**）：phase 只表示可独立建检、且通过后会降低后续不确定性或解锁后续推进的交付里程碑；默认只建一个。普通实施步骤、代码、测试、文档与验证优先归为其下 task。
_Avoid_: 用 phase 机械分组普通实施步骤；因代码、测试或文档属于不同工作面就拆 phase

**Task（任务）**：
按需递归分解（借鉴 ADaPT）的细粒度执行单元。默认对用户隐藏；在持续显示展开态里只随未完成 phase 展开，在详细查询 Modal 里可查看全量。四态：pending → in_progress → done | blocked。done 不回退；blocked 可回退 in_progress；blocked 时新建子 task（blockedBy 指向当前 task）接续——这就是按需递归分解的涌现。task 间用 `blockedBy` 依赖图表达分解关系，深度不限。

最小复杂度规范（ADR 0017，**vNext 已实现**）：代码改动、配套测试与相关文档等同一交付的工作，优先作为同一 phase 下的 task 一起完成。
_Avoid_: step（dgoal 内统一叫 task）

**层级基色（Layer Base Color）**：
dgoal TUI 中按内容层级固定分配的基础颜色：goal、phase、task 各自有稳定基色，用来表达信息层级，不表达运行状态。
_Avoid_: 状态色、完成色

**状态字符（Status Glyph）**：
dgoal TUI 中表达 pending / in_progress / done / blocked 的单字符前缀，如 `○ / ◐ / ✓ / ⚠`。它负责表达状态，不负责表达层级。
_Avoid_: 状态色、phase emoji

**接续 Task（Follow-up Task）**：
某 task 标 done 后发现错了/回归，不回退，新建的 task，`blockedBy` 指向原 task。保持历史不失真，浮层可见反复。

**Blocked（阻塞态）**：
- **Task 级 blocked**：这步因 X 原因无法完成，必须带 reason。可回退 in_progress。不触发 goal 停止。
- **Phase 级 blocked**：空 phase（未拆 task）整体判定不可行时可直接标 blocked；非空 phase 的 blocked 由其下 task 聚合得出。
- **Goal 级不进 blocked**：goal 级的停止态是 paused（见下）。

**单阶段完成建检（Single-Phase Completion Check，vNext 已实现）**：
单 phase goal 的唯一独立审核：在该 phase 的 task 已完成后，同时核验 phase 与 goal 的冻结验收条件；通过时记录统一 goal 审核凭据并标记 phase 完成，随后 `dgoal_done` 只关闭 goal，不重复审核。它取代单 phase 下先阶段建检、再目标终审的重复双审；多 phase goal 不使用该捷径。
_Avoid_: 把单阶段完成建检理解为跳过独立审核

## 状态机

**Goal 状态**：`pending | active | rejected | paused | done`
- pending：启动中，START prompt 未发。
- active：loop 正常推进。
- rejected：终审不过的回环态。每轮 prompt 钉着未过问题，agent 修完可再触发；`bounded` 按结构化修复预算和一次宽限暂停，旧 goal 无预算时才兼容三次拒绝暂停。
- paused：停止续跑。三类语义（见 PauseReason）。
- done：终审通过，goal 关闭，loop 结束。

**终审修复（Goal Repair，vNext 已实现）**：
`rejected` 的展示阶段：所有 phase 已完成，但 goal 正根据终审反馈修复，随后重审。它只显示在持续显示浮层、状态栏和详细查询 Modal；不是 phase、task 或额外状态机状态，不进入 plan 或触发额外建检。结构化有界预算耗尽时为 `paused(budget_exhausted)`；`paused(audit_failed_3x)` 仅是旧 goal 的兼容兜底。
_Avoid_: 把终审修复建模为免建检 phase 或 goal 级 task

**Paused（暂停态，三类语义）**：
goal 停止自动续跑。用 `GoalState.pauseReason` 区分，resume 时按 reason 决定是否清零 rejected 计数：
- 异常中断（user_abort / model_error / audit_error / no_progress）：resume 不清零，恢复原状继续。
- 有界预算耗尽（budget_exhausted）：宽限耗尽后暂停，resume 不清零用量或 rejected 计数。
- 旧 goal 能力到顶（audit_failed_3x）：仅无结构化 runtimeBudget 时触发，resume 清零 rejected 计数。
- agent 主动暂停（agent_blocked）：agent 调 `dgoal_pause` 声明遇到需要用户决策才能继续的死锁（如冻结验收条件与目标冲突）。resume 不清零 rejected，恢复继续；原因记在 `pauseReasonDetail`。

**PauseReason（暂停原因）**：
`GoalState.pauseReason` 字段，取值 `user_abort | model_error | audit_error | audit_failed_3x | no_progress | agent_blocked | budget_exhausted`。`budget_exhausted` 仅表示有界运行预算及一次宽限均耗尽。

**Phase 状态**：`pending | in_progress | done | blocked`（`phased` 下中间态由 task 聚合，done 由 `dgoal_check` 写入；`final_only` 另用 `progressCompleted` 记录进度划线，不代表独立审核通过；空 phase 可直接 blocked）

**Task 状态**：`pending | in_progress | done | blocked`（done 不回退，blocked 可回退）

## 工具（统一 `dgoal_` 前缀）

- **dgoal_propose**：启动闸门提交 goal + phases（+ 可选初始 task），触发确认 UI；冷会话中若当前真实用户输入已明确要求使用/启动 dgoal，也可先建立一次性显式 pending goal 再提交。
- **dgoal_plan**：task/phase 的 CRUD（建/改状态/依赖/标 task done/标空 phase blocked）。纯本地快操作。
- **dgoal_check**：phase done 的唯一入口（阶段建检门，spawn 独立验收子进程审计，fresh 上下文 + 受限核验工具）。单 phase 时一次审核同时核验 phase 与 goal，记录统一审核凭据；多 phase 仍逐 phase 建检。
- **dgoal_done**：按验收策略关闭 goal：`phased` 要求所有 phase 通过 `dgoal_check`，`final_only` 要求所有 phase `progressCompleted`，并携带 `verificationBundle`；随后触发一次 goal 级终审，通过才关 goal。完成时产出结构化的可核对文本（`summary` + `verification` + 可选 `whatChanged` 改动清单 + `userReview` 仍需用户核对），帮助用户核对结果与理解变更，降低理解债。
- **dgoal_pause**：agent 主动暂停出口。遇到需要用户决策才能继续的死锁时调用，立即 `paused(agent_blocked)` 并记录原因，不等 `no_progress` 连续 3 轮兜底。与消极空转互补：`no_progress` 仍是 agent 不懂事时的兜底。

## 其他

**启动闸门（Startup Gate）**：
`/dgoal` 命令或自然语言显式启动时，主代理用 `dgoal_propose` 提交 plan（goal + phases + 冻结验收条件 + 可选用户复核项）→ 弹确认 UI（确认/拒绝/输入反馈）→ 确认后进 loop。确认后冻结 goal 方向、验收策略/预算与验收条件；phase/task 层只允许调整执行进度、task 分解和 evidence，不提供验收条件更新入口。主代理可在 proposal 中按需提交可选 Context Summary，启动前不运行独立背景摘要。

**语义预审（Semantic Preflight）**：
`dgoal_propose` 写入 `pendingProposal` 前用当前会话模型对计划做的语义检查：人工体验/主观条件不进入冻结 `acceptanceCriteria`，改写到 `userReviewItems`。以流式接收模型响应，默认 60s **idle timeout**（无有效流事件才超时，收到任意事件重置；可通过 `pi-dgoal.json` 的 `proposalSemanticReviewIdleTimeoutSeconds` 配置）。终态拆为 `approved`/`rewritten`/`rejected`（语义，`isError:false`）与 `technical_error`（认证/超时/网络/JSON 解析等基础设施失败，`isError:true`）。预审不启动隔离子进程、不授予工具、不走审核候选链。
_Avoid_: 把预审技术失败（超时/网络）当成“计划内容有问题”让 agent 改计划；把预审与 `dgoal_check` 隔离建检混为一谈

**用户激活边界（User Activation Boundary，vNext 已实现）**：
默认情况下，只有用户输入 `/dgoal` 或在真实用户消息中明确要求“使用/启动 dgoal”，才授予 AI 使用 dgoal 的执行权。AI 看见 `dgoal_*` 工具不等于可自行启动或探测它们；未获授权时只能建议用户启动。ADR 0034 的全局配置授权是受限例外，激活后工具仍分别受 goal/phase 状态约束。
_Avoid_: 把工具已注册、文本里仅提到 dgoal、能力询问或 extension 注入理解为 AI 已获 dgoal 启动授权

**自然语言显式启动（Natural-Language Explicit Start）**：
真实用户在当前 `interactive` / `rpc` 输入中以明确动作词要求使用或启动 dgoal 后，获得的一次性显式启动权。它只在冷会话内存中短暂存在；`dgoal_propose` 消费后创建普通 pending goal，并继续语义预审与阻塞式确认 UI。它可承载 `phased` / `unbounded` 及计划内外部动作，因为这些仍要由用户确认；不等于 `implicit=true`，也不会因 extension 注入、讨论、能力询问或否定句触发。详见 ADR 0036。
_Avoid_: 把自然语言显式启动叫成隐式启动；把一次性启动授权理解为跳过确认或直接执行外部动作

**隐式轻量启动（Implicit Final-Only Start）**：
用户在全局 `pi-dgoal.json` 显式开启 `implicitFinalOnlyStart` 后，LLM 对范围具体、可独立验收的轻量任务，直接建立并执行 `final_only` 有界 goal 的受限权限。它允许本地测试、构建、解释器脚本、项目文件修改、本地 Git 变更和只读外部信息；`tool_call` 执行前策略门会阻止已识别的工作仓库 / `.git` 破坏、Git 远端写入、消息发送、发布/部署、外部写入、账号/权限修改与付费命令，并要求改走显式授权或保持禁止。该护栏是 best-effort 策略检查而非 OS sandbox，无法证明获准脚本内部没有隐藏副作用。默认预算为 24 个基础执行 turn、60 分钟基础有效运行时长、1 个终审修复回合，并在首次触顶后自动使用一次额外 24 turn 的预授权宽限，可由全局 `implicitFinalOnlyBudget` 覆盖。它跳过用户审核 plan，但不跳过 proposal 校验、语义预审、独立终审、持续显示浮层或用户的暂停/清除入口；项目配置不能授予此权限。详见 ADR 0035。
_Avoid_: 把它理解为默认自动接管、`unbounded` 权限，或项目自行开启的能力

**冷启动（Cold Start）**：
当前 session 没有 `currentGoal` 的状态；它不同于 `pending`（启动闸门已启动但尚未确认）、`active`、`rejected` 或 `paused`。`/dgoal help` 只在冷启动或 paused 可用。
_Avoid_: 把 pending 叫作冷启动

**Task Plan（任务计划）**：
goal 的"怎么做"的结构化脚手架，即 phase + task 两层。不是 plan mode 的计划文件，不是持久 Kanban。

## 验收与预算策略（ADR 0032–0034，v0.7.0）

**验收策略（Verification Policy）**：
同一 dgoal 提案在启动闸门中由主 agent 推荐、用户确认的建检密度选择。它不是 Plan / Goal 两套系统的开关；两种策略共享 Task Plan、确认 UI、自动推进、浮层、暂停和 goal 级独立终审。确认框默认选中 LLM 推荐值，用户可直接确认或覆盖；新 goal 的实际选择与 plan 一起冻结，运行期不能由 agent 静默切换。
_Avoid_: 把策略推荐当作不可改的 agent 决定；把 `final_only` 误称为无审核模式

**终审一次（`final_only`）**：
验收策略之一。phase 仅组织 task、浮层和上下文，不要求 phase 级独立验收条件，也不逐阶段调用 `dgoal_check`。主 agent 在某 phase 的 task 全部完成后回顾该阶段，可将其标为**阶段进度完成**；所有 task 都完成后，仍必须由 `dgoal_done` 触发一次 goal 级独立终审。
_Avoid_: 把阶段进度完成当作阶段建检通过，或把主 agent 自测当作独立审核结论

**分阶段建检（`phased`）**：
验收策略之一。仅用于每个 phase 有真实独立验收价值、通过会降低后续不确定性或解锁后续工作的计划。每个 phase 冻结独立验收条件，`dgoal_check` 通过才算阶段已建检完成，随后才可推进后续 phase 与 goal 终审。
_Avoid_: 因代码、测试、文档属于不同工作面就机械选择 `phased`

**阶段进度完成（Phase Progress Completion）**：
`final_only` 下主 agent 对一个 phase 的 task 全部完成并回顾后写入的进度事实；TUI 与正常完成 phase 一样以划线展示，便于用户读取计划主干。它不是独立建检，不是当前“phase done”的同义词，不构成终审证据；实现必须与“已通过阶段建检”保持不同业务语义。
_Avoid_: 写作“phase done”而不区分它是进度完成还是已通过 `dgoal_check`

**验证包（Verification Bundle）**：
主 agent 每次准备 goal 终审前提交给审核器的结构化定位材料：本轮改动、冻结完成门到命令/测试/工件的映射、最后一次改动后的自测、已知风险和未覆盖边界。它帮助审核器少从零定位，不是独立通过证据；可复用的独立工具事实仍受审核检查点边界约束。
_Avoid_: 把验证包或“测试已过”的主 agent 自述直接当作审核通过结论

**诊断审（Diagnostic Audit）**：
`final_only` 的首次 goal 级独立审核。它针对冻结完成门一次集中找全 blocker、实际回归和高风险证据缺口，不报告无关优化、偏好或 nits。
_Avoid_: 把诊断审理解为允许审核器临时增加新的完成门

**窄确认审（Narrow Confirmation Audit）**：
诊断审拒绝后，主 agent 修复并重新自测后的 goal 级独立审核。只核上轮 blocker、修复后新增 diff、受影响回归测试与少量全局保护测试；不得追加无关 nits 或新完成门，但新 diff 真实造成的回归仍可拒绝。
_Avoid_: 认为窄确认审等于不再独立复核最后一次修改

**执行轮次（Turn）**：
一次主 agent 执行回合：模型开始处理当前 goal，直到产生一次 `agent_end` 结果；`toolUse`、`length` 或正常 `stop` 都会结束当前模型执行并可能触发下一次续跑。它不是单个工具调用、消息 token 数或用户聊天消息。active goal 下每次非用户中断的 `agent_end` 都计入 `budgetUsage.turns`，因此该字段反映实际主模型执行次数，而不是只反映最终 `stop` 次数。

**预算策略（Budget Policy）**：
同一 dgoal 提案在启动闸门中由用户确认的运行成本边界。它与验收策略并列，不能只作为文本说明；确认框默认选中 LLM 推荐值，用户可直接确认或覆盖，确认后冻结并持久化。
_Avoid_: 把 LLM 推荐的预算误解为不可改；把预算策略混同为后台调度能力

**有界预算（Bounded Budget）**：
用户确认执行轮次、墙钟时长和/或审核修复回合上限的预算策略。首次耗尽发出一次非阻塞提示、状态栏持续标记“宽限中”，再消耗一次用户预授权的自动宽限；宽限耗尽才暂停并保留证据与反馈。
_Avoid_: 把首次预算告警直接当作不可恢复的失败，或让 agent 无限制自行追加宽限

**无上限预算（`unbounded`）**：
用户在提案阶段显式选择的预算策略：取消执行轮次、墙钟时长与固定终审拒绝次数造成的预算暂停，只在 Pi 会话持续运行时有效。它不取消模型错误、无进展、审核技术错误、用户暂停或真实冲突的安全出口；真实冲突必须 `paused(agent_blocked)` 保留证据等待用户决定。
_Avoid_: 把 `unbounded` 误解为跨 session / 后台运行，或把未验收目标标为 done

**Context Summary（启动背景固化）**：
`GoalState.contextSummary` 中保存的可选持久背景，补充范围、约束、风险和验收线索等“后续每轮都别忘”的信息；它是背景纪要，不是最终完成契约。v0.7.0 起由主 LLM 在 `dgoal_propose` 中按需提交，缺失不阻塞启动；生产启动路径不运行独立摘要子进程。
_Avoid_: 把 contextSummary 当成 verification 本身，或误认为必须另起摘要模型才可建立 plan

**Verification（全局验收说明）**：
`GoalState.verification` 中保存的 goal 级验收说明，回答“最后凭什么算整个 goal 完成”。它可参考或继承 contextSummary 里的“验收标准”部分，但必须在 `dgoal_propose` 阶段由主代理显式确认或改写。新 goal 的冻结完成门只有结构化 `acceptanceCriteria`；`verification` 是帮助 agent 和用户理解完成标准的总结性说明，不单独作为终审完成门。旧 session 缺少 `acceptanceCriteria` 时，终审兼容沿用 `verification` 作为完成门。
_Avoid_: 把 verification 当作新 goal 的唯一完成门（新 goal 必须有结构化 acceptanceCriteria）；把 verification 降格成普通背景备注

**验收口（Acceptance Gate）**：
一个 goal 被允许进入 `/dgoal` loop 之前，必须能归纳出的完成判据：交付什么、满足什么标准、凭什么算完成。新 goal 的验收口由 `dgoal_propose` 的结构化 `acceptanceCriteria` 冻结：goal 条件始终必需，`phased` 还要求每个 phase 的 criterion + evidence，`final_only` 的 phase 只记录进度；`verification` 是 goal 级验收说明，帮助理解完成标准。旧 session 可参考 `objective` 本身、`contextSummary` 的补充、以及 `verification`。验收口只收 **独立验收条件**，不收需要人判断的体验项。
_Avoid_: 把“先研究一下”“先看看再说”这类愿望句当成有验收口的 goal；把人工 TUI/体验确认写成 phase 或 goal 的通过前提

**独立验收条件（Independently Verifiable Criterion）**：
LLM 能通过项目工件、受限工具、可复现命令或可观察外部状态独立核验的完成条件。它可进入 phase 建检或 goal 终审，并决定 `done`。无法由 LLM 可靠观察或证明的条件不属于这一类。
_Avoid_: 把“用户实际觉得顺手”“人工看过界面”当成独立验收条件

**用户复核（User Review）**：
在 dgoal 已通过独立验收并结束后，明确交给用户确认的体验、视觉或实际使用事项。它可在启动闸门通过 `userReviewItems` 预先声明，也可由 agent 的 `userReview` 或审核器的非阻塞建议补充，最终统一通过完成回复交付；它不阻塞 phase / goal 完成，也不触发 `REJECTED` 或自动续跑；必须诚实说明该项尚未由 LLM 验证。
_Avoid_: 把用户复核伪装成“已通过”的审核证据，或把它变成 LLM 无法完成的建检门

**建检模式（Build-Verify）**：
dgoal 遵循的原则——生成（推进 task/phase）与验证（check）分离，验证是独立验收子进程（fresh 上下文 + 受限核验工具），不让学生判卷。关键是上下文隔离（不带主会话上下文），不是模型隔离——模型可复用当前会话模型，也可经 `phaseAuditorModels` / `goalAuditorModels` 为阶段建检与目标终审分别配置最多 3 个有序 `provider/model[:thinking]` 候选；项目候选链整体优先、不混合来源，同来源复数字段 > 对应单值字段 > 旧 `auditorModel`，`null` 显式继承当前会话模型并阻断继续降级；旧 `phaseAuditorModel` / `goalAuditorModel` / `auditorModel` 保持单候选兼容。候选先由与审核 child 同隔离边界的 Pi 结构化模型注册表预检，预检失败保留候选交运行时判定。详见建检循环与审核模型候选链。

**审核模型候选链（Auditor Model Candidate Chain）**：
同一审核范围内，用于在独立审核器发生技术/协议异常或无终止标记的部分输出时按顺序接替的有序模型集合。一次审核中每个候选最多调用一次；产生有效结论的候选会在当前 goal、同一审核范围内持久复用，候选全部不可用时审核暂停，而不是由执行模型静默接管。
_Avoid_: 把候选链理解成多个模型对审核结论投票、业务审不过就换模型，或执行模型兜底

**背景总结模型候选链（Context Summarizer Candidate Chain，已移除）**：
v0.7.0 不再配置或运行启动前独立背景摘要候选链。主 LLM 可在 `dgoal_propose` 中直接提交 `contextSummary`；缺失或不提交都不阻断启动。
_Avoid_: 把旧版本的背景摘要候选链重新接回生产启动路径

**建检反馈（Check Feedback）**：
`dgoal_check` 或终审未通过后，检查 agent 返回给主 agent 继续修正的原始失败报告。它是 agent-facing 的修复输入，不是 user-facing 的过程展示；用户只关心启动前确认、goal 完成结果，以及建检循环连续失败到暂停这类需要介入的节点。阶段建检反馈服务于当前 phase 的重做，终审反馈服务于 goal 级 rejected 回环；报告完整保存并完整钉回，不再生成 summary、不压缩。
_Avoid_: 把建检反馈当成普通聊天摘要、二手转述、用户状态面板内容或 agent 自述

**审核检查点（Audit Checkpoint）**：
独立审核 child（子进程）在 `tool_execution_start/end` 事件中产生的脱敏工具执行事实，按 `phase / goal` 审核范围与 workspace fingerprint（工作区指纹）持久化。只有同指纹、成功结束的精确命令可供下一候选或 resume（恢复）复用；running、failed、unknown 与工作区变化都不构成完成证据。
_Avoid_: 把审核检查点当成主 agent 自述、完整会话历史或测试通过声明

**终审修复账本（Final Audit History，vNext 已实现）**：
goal 级终审每次拒绝的追加式历史：保存原始报告、该轮完成声明与时间。它保留终审修复脉络，但不是 task、不是 phase，不参与 task 状态或额外建检；运行时仅注入最新完整报告与历史的紧凑索引，避免上下文膨胀。
_Avoid_: 将终审修复账本建模为 goal 级 task 或免建检 phase

**用户介入节点（User Intervention Point）**：
dgoal 中需要用户显式关注或决策的节点。当前包括：启动闸门确认、goal 完成结果、建检循环连续失败到暂停；普通阶段建检失败和终审 rejected 回环不是用户介入节点，主要服务于主 agent 继续修复。
_Avoid_: 把每次建检失败都设计成要求用户处理的交互节点

**就绪度分级（L0-L3）**：
评估一个 plan/loop 够不够格让 agent 自主执行的就绪度框架。自主性每升一级，要补齐更多工程约束。四级：
- **L0 Draft**：只有文档化的意图（目标 + 范围）。
- **L1 Report**：分诊→写状态，不自动行动（补齐目标范围 + 调度节奏 + skills + 状态）。
- **L2 Assisted**：带验证器的小自动修复（加验证 + 护栏 + 连接器）。
- **L3 Unattended**：自主跑到完成，无需人盯（加预算 + 可观测 + 安全的全部约束）。

核心交易：自主性换约束——要 agent 更自主，就得补更多约束（验证/护栏/预算/可观测）。是“没有约束的 loop 只是把错误自动化”的就绪度版。

**dgoal 的就绪度现状**：验证维度超 L3（`dgoal_check` 独立子进程 + 受限核验工具），状态/可观测到 L3；调度/skills/连接器 dgoal 不做（N/A）。启动闸门支持 `nonGoals` / `guardrails` / `budget` 作为 plan 级信号，并在确认 UI 显式展示就绪度等级与缺口提示；v0.7.0 进一步把预算策略落实为用户确认的结构化运行预算，并为全局授权隐式启动增加 fail-closed 动作护栏。

_Avoid_: 把 L0-L3 当成运行时自主性切换档位——它是 plan 提交前的就绪度评估，不是运行中切换的机制；把它和“用户介入节点”（运行时边界）混为一谈——L0-L3 评的是 plan 够不够格自主跑，用户介入节点定的是运行中人何时出现。

**建检闸门锁定（Check-Gate Lock）**：
阶段建检未通过、无结论或异常结束时的推进边界：goal 仍保持 active，但 loop 只能围绕当前 phase 补证据、修正并重试 `dgoal_check`。锁定的重点是**不能越阶段推进、不能越终审推进**；它不是对所有工具做广义拦截。它首先是推进规则，不要求为此把工具调用中的瞬时状态持久化进 `GoalState`。
_Avoid_: 把这类卡点叫成 paused、后台继续跑、先做后补状态，或误解成“所有 write/edit/bash 都要拦”

**越闸门推进（Gate-Jumping Progression）**：
当前 phase 尚未通过建检时，提前去推进后续 phase，或提前触发终审/完成声明的行为。典型表现包括：对后续 phase 调 `dgoal_plan` / `dgoal_check`，或在当前 phase 未过时调用 `dgoal_done`。
_Avoid_: 用“副作用越界”笼统概括这类问题

**意图边界（Intent Boundary）**：
建检闸门锁定时判断是否越界的标准：看一次改动是否服务于当前 phase 的建检目标与失败反馈，而不是看它碰到了哪个文件或目录。只要改动是为当前 phase 通过建检所必需，即使触及看似后续 phase 会改的文件，也不算越界；只有当它实质上构成越闸门推进时，才应被拦。
_Avoid_: 用文件归属、目录归属代替建检语境判断，或把意图边界扩成广义工具审查

**建检活性状态（Check Liveness State）**：
独立建检子进程在运行中的正式状态投影，用来表达它现在是启动中、思考中、调工具中、流报告中，还是已通过 / 未通过 / 异常结束。它服务于主 agent 和 TUI 的可观测性，不等于原始 thinking 正文；默认属于运行时观察层，不写入 `GoalState` 这类阶段性持久态。
_Avoid_: 把建检活性状态等同于原始思考文本、单纯的最终报告或需要长期持久化的业务事实

**参考注意力（Reference Attention）**：
dgoal 注入机制的理论坐标（类比自 R-SWA / Reference Sliding Window Attention，参考滑动窗口注意力）。goal 与 context 是全局可见的"参考层"，每轮注入、不随进度退化；当前 phase 与其下 task 是"工作记忆"，聚焦当下进度。它回答"dgoal 注入机制在概念上是什么"，不定义具体注入函数。详见 ADR 0010。
_Avoid_: 把参考注意力当成一个独立模块或数据结构

**软遗忘（Soft Forgetting）**：
dgoal 注入的一个边界规则：done phase（建检通过）在注入里只保留 phase 标题行，其下 task 的 subject 与 evidence 不再注入。软遗忘只作用于注入投影层，不改持久化（`goal.plan` 全量保存）、不改建检/终审子进程的可见性（读持久化全量）、不另建回查工具。软遗忘的时机是 phase 整体 done，不是单个 task done。详见 ADR 0010。
_Avoid_: 把软遗忘当成"删除"——信息没丢，只是不占每轮注意力
