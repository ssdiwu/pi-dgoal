# pi-dgoal 术语表

pi-dgoal 是一个 Pi 扩展，让 agent 围绕一个明确 goal（目标）持续工作，在**建检循环**里推进，直到完成并通过独立 check。

## 核心原理

**建检循环（Build-Check Loop）**：
dgoal 的第一性原理。**dgoal 定义了 goal，定义了完成后要 check——不通过则继续干活，通过则结束。**

建检循环有两个粒度，统一用 `dgoal_check` 机制：
- **阶段建检**（小建检）：一个 phase 要标 done 时触发，审该 phase 成果。通过则 phase 进 done，不通过则回 in_progress 继续干。
- **终审**（大建检）：最后一个 phase 标 done 时，check 升级为审全 goal。这就是 `dgoal_done` 内部走的 check。终审通过则 goal 进 done，不通过则 goal 进 rejected。

建检循环是**心智模型**，不是显式数据结构——它的两个实例（phase 门、终审回环）各有状态承载（phase 状态 / goal 的 rejected），不为心智模型重复建模。

_Avoid_: 把建检循环当成独立机制/状态/循环结构去实现。

## 三层内容与双可见性轴

dgoal 自然有三层内容，但有**两条独立的可见性轴**：

| 层级 | 用户可见性（TUI） | AI 可见性（context） |
|---|---|---|
| **Goal**（全局目标） | 可见（浮层 heading） | 可见 |
| **Phase**（阶段） | 可见（浮层默认显示） | 可见 |
| **Task**（按需分解的细粒度执行单元） | **默认隐藏**，`Ctrl+O` 展开看细节 | 可见（dgoal 把三层都喂给 AI） |

用户可见性轴：避免 TUI 被海量细 task 淹没（10 行折叠预算，预留 Pi core 截断提示空间），用户平时只看 phase 进度，需要细节时 `Ctrl+O` 展开。
AI 可见性轴：AI 始终看全三层，知道每个 task 服务哪个 phase、哪个 goal——避免局部最优但全局无用。

## Language

**Goal（目标）**：
用户通过 `/dgoal` 发起的全局目标。启动闸门确认后冻结，是整个 loop 的方向契约。
_Avoid_: objective（objective 是 goal 的文本简述字段）

**Objective（目标简述）**：
goal 的一句话文本描述，存在 `LoopGoal.objective`。

**LoopGoal（目标持久态）**：
dgoal 用来保存当前 goal 阶段性业务事实的主对象：如 objective、status、plan、verification、contextSummary、pauseReason、rejectedCount、建检反馈等。它是状态机与恢复的持久账本，不承载独立建检子进程的瞬时运行态。
_Avoid_: 把 LoopGoal 当成 TUI 展示状态、工具调用日志或运行时事件缓存

**Phase（阶段）**：
阶段性目标，状态**由其下 task 聚合**（有 in_progress 的 task→in_progress；全终态→done；有 blocked 且无 in_progress→blocked）。TUI 浮层默认显示的单元。标 phase done 的唯一入口是 `dgoal_check`（阶段建检门）。尚未拆 task 的空 phase 允许直接标 blocked（整阶段判定不可行）。

**Task（任务）**：
按需递归分解（借鉴 ADaPT）的细粒度执行单元。默认对用户隐藏（Ctrl+O 展开）。四态：pending → in_progress → done | blocked。done 不回退；blocked 可回退 in_progress；blocked 时新建子 task（blockedBy 指向当前 task）接续——这就是按需递归分解的涌现。task 间用 `blockedBy` 依赖图表达分解关系，深度不限。
_Avoid_: step（dgoal 内统一叫 task）

**层级基色（Layer Base Color）**：
dgoal TUI 中按内容层级固定分配的基础颜色：goal、phase、task 各自有稳定基色，用来表达信息层级，不表达运行状态。
_Avoid_: 状态色、完成色

**状态字符（Status Glyph）**：
dgoal TUI 中表达 pending / in_progress / done / blocked 的单字符前缀，如 `○ / ◐ / ✓ / ⚠`。它负责表达状态，不负责表达层级。
_Avoid_: 状态色、phase emoji

**接续 Task（Follow-up Task）**：
某 task 标 done 后发现错了/回归，不回退，新建的 task，`blockedBy` 指向原 task。保持历史不失真，浮层可见反复。

**Blocked（阻塞态）**：
- **Task 级 blocked**：这步因 X 原因无法完成，必须带 reason。可回退 in_progress。不触发 goal 停止。
- **Phase 级 blocked**：空 phase（未拆 task）整体判定不可行时可直接标 blocked；非空 phase 的 blocked 由其下 task 聚合得出。
- **Goal 级不进 blocked**：goal 级的停止态是 paused（见下）。

## 状态机

**Goal 状态**：`pending | active | rejected | paused | done`
- pending：启动中，START prompt 未发。
- active：loop 正常推进。
- rejected：终审（最后 phase 的大建检）不过的回环态。每轮 prompt 钉着未过问题，agent 修完可再触发。连续 3 次终审不过转 paused。
- paused：停止续跑。两种语义（见 PauseReason）。
- done：终审通过，goal 关闭，loop 结束。

**Paused（暂停态，两种语义）**：
goal 停止自动续跑。用 `LoopGoal.pauseReason` 区分，resume 时按 reason 决定是否清零 rejected 计数：
- 异常中断（user_abort / model_error / audit_error）：resume 不清零，恢复原状继续。
- 能力到顶（audit_failed_3x）：resume 清零 rejected 计数，给 agent 重新机会。

**PauseReason（暂停原因）**：
`LoopGoal.pauseReason` 字段，取值 `user_abort | model_error | audit_error | audit_failed_3x`。

**Phase 状态**：`pending | in_progress | done | blocked`（由其下 task 聚合，空 phase 可直接 blocked）

**Task 状态**：`pending | in_progress | done | blocked`（done 不回退，blocked 可回退）

## 工具（统一 `dgoal_` 前缀）

- **dgoal_propose**：启动闸门提交 goal + phases（+ 可选初始 task），触发确认 UI。
- **dgoal_plan**：task/phase 的 CRUD（建/改状态/依赖/标 task done/标空 phase blocked）。纯本地快操作。
- **dgoal_check**：phase done 的唯一入口（阶段建检门，spawn 独立验收子进程审计，fresh 上下文 + 受限核验工具）；最后一个 phase 的 check 即终审。通过才标 done，不通过则 phase 回 in_progress（终审不通过则 goal 进 rejected）。
- **dgoal_done**：≈ 标最后一个 phase done（走 dgoal_check 终审）+ 关 goal。agent 调它即触发终审。

## 其他

**启动闸门（Startup Gate）**：
`/dgoal` 启动时主代理用 `dgoal_propose` 提交 plan（goal + phases）→ 弹确认 UI（确认/拒绝/输入反馈）→ 确认后进 loop。确认的是 goal 层（冻结），phase/task 层进 loop 后可调。

**Task Plan（任务计划）**：
goal 的"怎么做"的结构化脚手架，即 phase + task 两层。不是 plan mode 的计划文件，不是持久 Kanban。

**Context Summary（启动背景固化）**：
`LoopGoal.contextSummary` 中保存的启动背景摘要。它来自 objective（目标简述）之外的前文讨论提炼，用来补充范围、约束、验收线索等“后续每轮都别忘”的信息；它是背景纪要，不是最终完成契约。
_Avoid_: 把 contextSummary 当成 verification 本身

**Verification（全局验收说明）**：
`LoopGoal.verification` 中保存的 goal 级完成口径，回答“最后凭什么算整个 goal 完成”。它可参考或继承 contextSummary 里的“验收标准”部分，但必须在 `dgoal_propose` 阶段由主代理显式确认或改写；它是完成契约，不是背景摘要。
_Avoid_: 把 verification 降格成普通背景备注

**验收口（Acceptance Gate）**：
一个 goal 被允许进入 `/dgoal` loop 之前，必须能归纳出的完成判据：交付什么、满足什么标准、凭什么算完成。验收口可以来自 `objective` 本身、`contextSummary` 的补充、以及 `dgoal_propose` 的显式 `verification`；但若三者合起来仍归纳不出，就应在启动闸门打回。
_Avoid_: 把“先研究一下”“先看看再说”这类愿望句当成有验收口的 goal

**建检模式（Build-Verify）**：
dgoal 遵循的原则——生成（推进 task/phase）与验证（check）分离，验证是独立验收子进程（fresh 上下文 + 受限核验工具），不让学生判卷。详见建检循环。

**建检反馈（Check Feedback）**：
`dgoal_check` 或终审未通过后，检查 agent 返回给主 agent 继续修正的原始失败报告。它是 agent-facing 的修复输入，不是 user-facing 的过程展示；用户只关心启动前确认、goal 完成结果，以及建检循环连续失败到暂停这类需要介入的节点。阶段建检反馈服务于当前 phase 的重做，终审反馈服务于 goal 级 rejected 回环；报告完整保存并完整钉回，不再生成 summary、不压缩。
_Avoid_: 把建检反馈当成普通聊天摘要、二手转述、用户状态面板内容或 agent 自述

**用户介入节点（User Intervention Point）**：
dgoal 中需要用户显式关注或决策的节点。当前包括：启动闸门确认、goal 完成结果、建检循环连续失败到暂停；普通阶段建检失败和终审 rejected 回环不是用户介入节点，主要服务于主 agent 继续修复。
_Avoid_: 把每次建检失败都设计成要求用户处理的交互节点

**建检闸门锁定（Check-Gate Lock）**：
阶段建检未通过、无结论或异常结束时的推进边界：goal 仍保持 active，但 loop 只能围绕当前 phase 补证据、修正并重试 `dgoal_check`。锁定的重点是**不能越阶段推进、不能越终审推进**；它不是对所有工具做广义拦截。它首先是推进规则，不要求为此把工具调用中的瞬时状态持久化进 `LoopGoal`。
_Avoid_: 把这类卡点叫成 paused、后台继续跑、先做后补状态，或误解成“所有 write/edit/bash 都要拦”

**越闸门推进（Gate-Jumping Progression）**：
当前 phase 尚未通过建检时，提前去推进后续 phase，或提前触发终审/完成声明的行为。典型表现包括：对后续 phase 调 `dgoal_plan` / `dgoal_check`，或在当前 phase 未过时调用 `dgoal_done`。
_Avoid_: 用“副作用越界”笼统概括这类问题

**意图边界（Intent Boundary）**：
建检闸门锁定时判断是否越界的标准：看一次改动是否服务于当前 phase 的建检目标与失败反馈，而不是看它碰到了哪个文件或目录。只要改动是为当前 phase 通过建检所必需，即使触及看似后续 phase 会改的文件，也不算越界；只有当它实质上构成越闸门推进时，才应被拦。
_Avoid_: 用文件归属、目录归属代替建检语境判断，或把意图边界扩成广义工具审查

**建检活性状态（Check Liveness State）**：
独立建检子进程在运行中的正式状态投影，用来表达它现在是启动中、思考中、调工具中、流报告中，还是已通过 / 未通过 / 异常结束。它服务于主 agent 和 TUI 的可观测性，不等于原始 thinking 正文；默认属于运行时观察层，不写入 `LoopGoal` 这类阶段性持久态。
_Avoid_: 把建检活性状态等同于原始思考文本、单纯的最终报告或需要长期持久化的业务事实

**参考注意力（Reference Attention）**：
dgoal 注入机制的理论坐标（类比自 R-SWA / Reference Sliding Window Attention，参考滑动窗口注意力）。goal 与 context 是全局可见的"参考层"，每轮注入、不随进度退化；当前 phase 与其下 task 是"工作记忆"，聚焦当下进度。它回答"dgoal 注入机制在概念上是什么"，不定义具体注入函数。详见 ADR 0010。
_Avoid_: 把参考注意力当成一个独立模块或数据结构

**软遗忘（Soft Forgetting）**：
dgoal 注入的一个边界规则：done phase（建检通过）在注入里只保留 phase 标题行，其下 task 的 subject 与 evidence 不再注入。软遗忘只作用于注入投影层，不改持久化（`goal.plan` 全量保存）、不改建检/终审子进程的可见性（读持久化全量）、不另建回查工具。软遗忘的时机是 phase 整体 done，不是单个 task done。详见 ADR 0010。
_Avoid_: 把软遗忘当成"删除"——信息没丢，只是不占每轮注意力
