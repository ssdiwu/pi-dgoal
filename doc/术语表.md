# pi-dgoal 术语表

pi-dgoal 是一个 Pi 扩展：用同一套 Task Plan 数据结构承载三档 Plan，并按 Plan 类型增加用户确认和独立审核保障。

## 核心原理

**建检循环（Build-Check Loop）**：
先建造，再由 fresh context 的独立审核器检查；不通过就继续修，满足冻结条件后才允许完成。它是验证约束，不是定时调度。Phase Plan 只在 goal 级应用，Goal Plan 在 phase 与 goal 两级应用，Task Plan 不启用独立审核。
_Avoid_: 把主 agent 的“我跑过测试”当成独立审核；把 dgoal 扩成定时任务或后台调度器

**直到完成（Until Done）**：
agent 在当前 Pi session 内持续推进同一个 Plan，直到状态守卫允许 goal done，或进入 paused 等停止态。pi-dgoal 不做跨会话后台运行、多目标池或定时触发。

**三档 Plan（Three-Plan Runtime）**：
共享 goal / phase / task 数据、持久化和 TUI 的三种保障强度。Plan 类型由 `GoalState.planType` 冻结，不在运行中切换。

| Plan | 建立方式 | 结构 | 完成保障 |
|---|---|---|---|
| **Task Plan**（`task`） | agent 可主动调用 `task_plan` | 隐藏单 phase + 多 task | 最后一个 task 带 evidence 进入 done 时自动完成 goal；无独立审核 |
| **Phase Plan**（`phase`） | 显式 dgoal 激活后调用 `phase_plan` | 1..N phase + task + goal 验收契约 | phase 要求 task 全部带 evidence 进入 done；goal 需 `goal_check` approved |
| **Goal Plan**（`goal`） | 显式 dgoal 激活后调用 `goal_plan` | 1..N phase + task + phase/goal 验收契约 | phase 另需 `phase_check` approved；goal 需 `goal_check` approved |

**Task Plan（任务计划）**：
普通、明确、多步执行或 AFK、有界、低风险探索的默认结构化脚手架。它可承载当前 frontier：稳定的是用户意图 / 最终效果，当前 objective 与 tasks 可随证据整份替换。它不要求 `/dgoal`、语义预审、用户确认或独立审核，也不扩大宿主 agent 的执行权限。纯讨论、解释、能力问答、单步回答和必须由用户选择的 HITL 决策不应为了形式而建计划。再次调用 `task_plan` 会原子替换 objective、goal description 与全部 task，不合并旧 task，也不保留完整探索历史。
_Avoid_: 把 Task Plan 当成完整 dgoal 审核；每条用户消息都机械建计划；把用户决策伪装成探索 task

**Phase Plan（阶段计划）**：
适合需要冻结 goal 验收契约、独立 goal 终审，但 phase 只是执行主干的目标。phase 不携带独立完成门，不调用 `phase_check`。所有 phase 完成后运行 `goal_check`。
_Avoid_: 因为叫 Phase Plan 就给每个 phase 建检

**Goal Plan（目标计划）**：
适合 phase 本身是可独立验收的交付里程碑，且通过会降低后续不确定性或解锁推进的目标。每个 phase 与整体 goal 都有冻结验收条件，分别由 `phase_check` 与 `goal_check` 独立审核。
_Avoid_: 因代码、测试、文档属于不同工作面就机械升级为 Goal Plan

## 三层内容

**Description（执行说明，ADR 0042）**：
goal、用户可见 phase 与 task 的必填文本，回答“为什么存在、对上层目标有什么作用、为什么采用当前方法、应避免哪些方法偏移”。objective / subject 继续承担短标题；description 不得同义复述标题，也不写实时进度。它是主 Agent 必须遵循的权威执行说明，但不是独立审核完成门；硬方法边界仍写入 guardrails / acceptanceCriteria。Phase/Goal Plan 的 goal description 随确认冻结，phase/task description 可显式修订。Task Plan 的隐藏 phase 不生成独立 description；内部字段复用 goal description，但不作为 phase 展示。
_Avoid_: 把 description 写成“正在处理”之类运行态文案；把历史讨论、日志或旧 prompt 塞进 description；凭 description 临时增加审核条件

**Goal（目标）**：
Plan 的全局方向与生命周期对象。`objective` 是 goal 的一句话文本字段，不是另一个层级。

**Phase（阶段）**：
一组 task 的阶段性主干。Task Plan 内部有一个隐藏 phase，但它不出现在工具结果、prompt 和 TUI。Phase/Goal Plan 的 phase 在启动时冻结；运行中不新增 phase。

**Task（任务）**：
可执行、可更新、可提供 evidence 的细粒度单元。四态为 `pending → in_progress → done | blocked`；不能跳过 in_progress 直接 done，done 必须带可复验证据且不回退，发现问题时新建 follow-up task；blocked 可回到 in_progress，且必须带原因。task 用 `blockedBy` 表达依赖，不能成环，也不能依赖后续 phase。

**接续 Task（Follow-up Task）**：
已 done task 后发现缺陷时新增的 task，通常以 `blockedBy` 指向原 task。它保留历史真实性，避免把 done 回退。

**隐藏 Phase（Hidden Phase）**：
Task Plan 为复用现有 reducer 和持久化结构而使用的内部容器。工具层只暴露平铺 task；TUI heading 显示 task 完成数；它不构成用户可见层级。

**Plan ID 命名空间（Plan ID Namespace）**：
phase 与 task 各自使用独立数字命名空间，二者都从 `1` 开始；task ID 在整个 Plan 内全局唯一，不按 phase 重置，以便 `blockedBy` 跨阶段引用。工具通过 `target` 或专用参数区分 phase/task，同号不表示同一对象。`TaskPlan.nextId` 只表示下一个 task ID；旧 Plan 保留原编号。
_Avoid_: 要求 phase/task 数字全局不重复；让每个 phase 的 task 都重新从 `1` 编号

## 八个公共工具

工具名遵循“两词原则”，不使用 `dgoal_` 前缀；`dgoal` 只保留为产品名与 `/dgoal` 用户命令。

### 建立 Plan

- **`task_plan`**：直接建立或整份替换 Task Plan；参数为 objective + goal description + 带 description 的初始 tasks。
- **`phase_plan`**：显式 dgoal 启动入口；提交 Phase Plan、goal description/验收契约和带 description 的初始 phase/task，经语义预审与确认 UI。
- **`goal_plan`**：显式 dgoal 启动入口；在 Phase Plan 基础上要求每个 phase 的冻结验收条件；三层 Description 同样必填。

### 管理 Plan

- **`plan_create`**：只新增 task。Task Plan 不需要 phase 参数；Phase/Goal Plan 必须定位已有 phase。不能新增 phase。
- **`plan_read`**：纯读；读取 Plan、goal、phase 或 task。Task Plan 返回平铺 task，不暴露隐藏 phase。
- **`plan_update`**：唯一 agent 执行状态写工具；更新 task 状态/证据/依赖，写 phase done，写 Phase/Goal Plan 的 goal done，或以 `target=goal,status=paused,reason=...` 主动暂停。Task Plan 的最后一个 task 完成时由本工具调用自动收口；check 只写审核记录。

### 独立审核

- **`phase_check`**：仅 Goal Plan 可用；独立审核一个 phase，写 `CheckRecord`，不写 phase done。
- **`goal_check`**：Phase/Goal Plan 在所有 phase done 后运行；写 `CheckRecord`，不写 goal done。

**Check / Update 分离**：
`check` 只负责判卷并记录结果；`plan_update` 才负责确认完成并改变浮层。`approved` 不等于 done。
_Avoid_: 在 check 内顺手标完成；把 `plan_update` 当作绕过审核的后门

## 完成守卫

**Task Plan 完成链**：
`task_plan → plan_create/plan_update(task) → 最后一个 task done 自动关闭 goal`。goal done 要求全部 task 带 evidence 进入 done；blocked 不算完成。

**Phase Plan 完成链**：
`phase_plan → plan_update(phase, done) × N → goal_check → plan_update(goal, done)`。phase done 要求其 task 全部带 evidence 进入 done；blocked 表示尚未完成。goal done 要求所有 phase done 且当前 revision 的 goal check approved。

**Goal Plan 完成链**：
`goal_plan → [phase_check → plan_update(phase, done)] × N → goal_check → plan_update(goal, done)`。phase done 额外要求对应 phase check approved；goal done 要求当前 goal check approved。

**Plan Revision（计划修订号）**：
任何改变受审事实的写操作都会增加 `TaskPlan.revision`，并清理旧审核检查点。check 记录启动时的 revision；审核运行中若 Plan 已变化或清除，迟到结果直接丢弃，必须重审。
**Phase Revision（阶段修订号）**：
Goal Plan 中单个 phase 的受审事实版本。只要该 phase 的 task、Description 或独立验收相关事实变更就递增，并使该 phase 的 `phase_check` 失效；其他 phase 的批准保持有效。它不替代全局 Plan Revision，后者仍使 `goal_check` 失效。


**CheckRecord（审核记录）**：
一次 phase 或 goal 审核的持久结果：`approved | rejected | audit_error`、report、checkedAt、revision。它是审核事实，不是完成状态。

**冻结验收契约（Frozen Acceptance Contract）**：
Phase/Goal Plan 启动确认时冻结的 `acceptanceCriteria`。goal 条件两者都必须有；phase 条件只属于 Goal Plan。每项包含 `criterion` 与可独立复验的 `evidence`。运行中不能扩张契约。

**Verification（全局验收说明）**：
帮助用户与 agent 理解“整体凭什么算完成”的文字说明；它不替代结构化 `acceptanceCriteria`。

**独立验收条件（Independently Verifiable Criterion）**：
可由项目工件、受限工具、可复现命令或可观察外部状态核验的条件。主观体验不属于这一类。

**用户复核（User Review）**：
完成后交给用户人工观察、体验或偏好判断的事项。它不进入机器完成门，也不能被描述为已经验证。

## 激活与提案

**用户激活边界（User Activation Boundary）**：
agent 可主动建立 Task Plan，但不能自行调用 `phase_plan` / `goal_plan` 升级保障强度。Phase/Goal Plan 需要用户输入 `/dgoal`，或以明确祈使句要求使用/启动 dgoal。
_Avoid_: 因工具可见就认为 AI 获得 dgoal 启动权

**自然语言显式启动（Natural-Language Explicit Start）**：
真实用户在空闲的 interactive / RPC 输入中明确要求使用 dgoal，形成一次性显式授权。能力问句、引用、解释讨论、否定、标识符后缀、处理中追加与 extension 注入不授权。它仍进入语义预审和确认 UI，不是隐式执行。

**启动闸门（Startup Gate）**：
显式 dgoal 激活后，主 agent 读相关上下文，推荐 Phase Plan 或 Goal Plan，调用对应入口；语义预审成功后弹确认 UI。用户可确认、拒绝、反馈或要求改用另一 Plan 类型。Task Plan 不经过启动闸门。

**语义预审（Semantic Preflight）**：
`phase_plan` / `goal_plan` 写 pending proposal 前，由当前会话模型判断计划能否自主闭环，并把候选条件分为独立验收条件、非阻塞用户复核与真实人工 blocker。默认 60 秒 idle timeout，可通过 `proposalSemanticReviewIdleTimeoutSeconds` 配置。技术失败与语义拒绝分开返回。

**轻提案、硬执行（Thin Proposal, Hard Execution）**：
代码校验结构、状态和授权；LLM 判断 proposal 语义；真实动作仍受宿主工具权限和执行护栏约束；独立审核器只核冻结结果。Task Plan 不经过 proposal，也不因此获得额外权限。

## 状态机

**Goal 状态**：
- `pending`：Phase/Goal Plan 正在提案或等待确认。
- `active`：Plan 正在推进。
- `paused`：自动续跑停止，等待恢复或用户处理。
- `done`：完成守卫通过后收口；Task Plan 由最后一个 task 的 `plan_update` 自动收口，Phase/Goal Plan 由 `plan_update(target=goal,status=done)` 收口；持久化随后清空活动 Plan。

业务 rejection 只存在于 `CheckRecord`；Goal 保持 `active`，不建立平行 rejected 状态。

**Phase 状态**：`pending | in_progress | done | blocked`。中间状态可由 task 聚合；done 只能由 `plan_update(target=phase,status=done)` 在守卫通过后写入。

**Task 状态**：`pending | in_progress | done | blocked`。

**Paused（暂停态）**：
显式 Phase/Goal Plan 的 `user_abort`、`model_error`、`audit_error`、`no_progress` 是固定暂停出口；Task Plan 的普通用户中断只停止当前响应、清理旧 continuation，并保持 `active` 等待下一条用户输入，不要求 `/dgoal resume`。agent 遇到只有用户能决策的死锁时，用 `plan_update(target=goal,status=paused,reason=...)` 写 `agent_blocked`。不再有用户可选的 bounded/unbounded 预算或 `budget_exhausted` 新路径。

## 独立审核

**建检模式（Build-Verify）**：
执行与验证分离；审核器运行在 fresh context 的隔离 Pi 子进程，可使用受限核验工具。模型默认继承当前会话，也可通过 `phaseAuditorModels` / `goalAuditorModels` 配置最多三个有序候选。

**审核模型候选链（Auditor Model Candidate Chain）**：
只在技术/协议异常时切换，不因业务 rejected 换模型。一次审核中每候选最多调用一次；产生有效结论的候选在同 goal、同 scope 复用；候选耗尽则 `audit_error` 暂停。

**审核检查点（Audit Checkpoint）**：
按 phase/goal scope 与 workspace fingerprint 持久化的脱敏成功工具事实。只有审核终态或可恢复的稳定 checkpoint 写入会话；逐工具进行中的活性状态只在内存投影。只有同 scope、同指纹、明确成功的精确命令可供下一候选或 resume 复用；任一相关 Plan 写操作、running、failed、unknown 与工作区变化都不是证据。

**建检反馈（Check Feedback）**：
rejected check 的原始报告，作为 agent 后续修复输入。Plan 改动后旧 check 失效，再次审核产生新记录；普通 rejected 不要求用户逐次介入。

**建检活性状态（Check Liveness State）**：
审核子进程的启动、思考、工具执行、报告流与终态投影。它服务于可观测性，不把原始 thinking 写入业务持久态。

## TUI 与可见性

**持续显示浮层**：
Pi `setWidget(..., placement:"aboveEditor")` 常驻面板。Task Plan 默认显示 task；Phase/Goal Plan 默认显示 phase；heading 同时显示聚合 task 完成数。`Ctrl+O` 展开未完成 phase 的 task。

**详细查询 Modal（Status Modal）**：
`/dgoal s` 打开的居中两层 overlay。列表页展示完整 goal description 与可选择的 phase/task；详情页展示所选项的完整 description 与 status、dependency、evidence、blocked reason。Task Plan 仍只显示平铺 task，不显示隐藏 phase；详情返回列表时保留选中位置。

**状态栏（Status Bar）**：
`setStatus("dgoal", …)` 的一行 goal 状态。它与持续显示浮层是不同展示面。

**软遗忘（Soft Forgetting）**：
Phase/Goal Plan 的 done phase 在每轮 prompt 中只保留标题，其 task 不再占工作上下文；持久化和详细查询仍保留全量。Task Plan 不按隐藏 phase 软遗忘单个 task。

## 持久化与边界

**持久化键**：
三档 Plan 使用 `dgoal-plan-v2` custom entry。旧 `dgoal-state` / `dgoal-goal-vnext` / `dgoal-plan-v1` 不读取、不迁移；升级后旧活动目标需重新建立。

**会话内单 Plan**：
同一 Pi session 同时只维护一个当前 Plan。Task Plan 可整份替换；Phase/Goal Plan 运行中不新增 phase、不静默切换类型。

**固定技术熔断**：
连续无进展、模型连续错误、审核器异常、审核 timeout、用户中断与 agent 主动暂停继续保留。它们是可靠性边界，不是用户可配置运行预算。
