# ADR 0016：阶段与目标只接受 LLM 可独立验收条件

phase（阶段）与 goal（目标）的完成前提只能是 LLM（大语言模型）可用 dgoal 的工具与独立审核器实际核验的条件。TUI 体验、视觉质量及其他必须由人判断的事项不进入 phase / goal 的通过门；它们作为 dgoal 完成回复里的 `userReview`（用户复核）明确交给用户。

## 背景

一次修复任务的启动计划明确把真实模型 smoke 和人工 TUI smoke 列为 non-goal，并将确定性自动化测试写入 verification（验收说明）。阶段审核器却根据项目 `AGENTS.md` 把人工 TUI smoke 追加为通过前提。

这造成两个问题：

- 已确认并冻结的验收契约，被审核器在运行中扩容；
- 人工 TUI smoke 不能由隔离审核子进程或 LLM 自行完成、观察并诚实证明，phase 因而会在 `REJECTED`（拒绝）回环中永远无法完成。

严格审核不等于让不可独立核验的要求成为 dgoal 的停止条件。否则 dgoal 会把“需要用户判断”伪装成“agent 再努力即可解决”的实现失败。

## 决策

### 1. 完成门只收独立可验收条件

启动闸门确认的完成契约必须区分两类内容：

- **独立验收条件**：LLM 能通过项目工件、受限工具、可复现命令或可观察外部状态独立核验。它们可以进入 phase 建检和 goal 终审，并决定 `done`。
- **用户复核项**：需要用户实际使用、视觉判断、主观体验判断或其他 LLM 当前无法可靠观察的事项。它们不能进入任何 phase / goal 的通过前提。

若某项“必须完成”条件无法被 LLM 独立验收，提案不是可自主完成的 dgoal；启动闸门必须要求重新表述为可验收条件，或把它降为用户复核项。dgoal 不承担依赖人工签字的发布门禁；那是外部发布流程的职责。

### 2. 用户复核在完成后交付，不伪装为通过

`dgoal_done` 终审通过后，完成回复必须保留并清晰呈现 `userReview`：

- 明确写出尚未由 LLM 证明的体验/界面风险和建议操作；
- 不写“已验证 UI 体验”之类的虚假成功结论；
- 不阻止 goal 进入 `done`，也不触发 continuation（自动续跑）或 `REJECTED` 回环。

这让用户获得必要的复核提示，同时保持 dgoal 的 `done` 语义诚实：它只表示冻结的独立验收契约已通过。

### 3. 审核器不得运行时扩容完成契约

阶段审核与终审可以：

- 否定已有独立验收条件的证据；
- 发现实现、文档或测试与已确认目标不一致；
- 报告额外风险，并将不可独立验证的体验项建议为用户复核。

审核器不得仅因 `AGENTS.md`、README 或自身推断发现额外人工验证要求，就把它添加为 `FAIL`（失败）并阻塞 phase / goal。项目规范中可独立执行的强制命令，应在启动计划里显式成为独立验收条件；人工要求则应成为用户复核项。

## 实现约束

当前 `verification`、`nonGoals` 和 `userReview` 仍主要是文本字段；后续实现应在启动闸门将“独立验收条件 / 用户复核项”结构化并持久化，使审核 prompt（提示词）只以冻结契约作为 completion blocker（完成阻塞项）。具体字段形状和迁移策略另行设计；本 ADR 不要求迁移既有 session。

需要同步收紧 `buildPhaseCheckTask()` / `buildAuditorTask()`：不得再以“补充隐含验收条件”或泛化的文档检查，把未冻结的人工验证扩容为拒绝理由。

## 理由

- 建检循环的停止条件必须是 agent 可完成、审核器可判定的事实；否则循环不具备终止性。
- 用户复核保留体验与视觉质量的责任，不让自动化“通过”掩盖其未验证状态。
- 启动闸门承担范围确认，审核器承担独立判卷；后者不能反向改写前者确认的契约。
- 把人工发布门禁留在外部流程，避免 dgoal 从会话内建检循环膨胀为发布审批系统。
