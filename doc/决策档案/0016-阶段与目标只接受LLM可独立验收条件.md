# ADR 0016：阶段与目标只接受 LLM 可独立验收条件

> Status：独立验收 / 用户复核分离与审核器不得扩容完成门继续有效。ADR 0037 覆盖 evidence 词形启发式；ADR 0038 规定 goal 条件用于 Phase/Goal Plan，phase 条件只用于 Goal Plan，并把旧 `dgoal_propose` / `dgoal_done` / `dgoal_plan` 工具边界替换为八工具链。

phase（阶段）与 goal（目标）的完成前提只能是 LLM（大语言模型）可用 dgoal 的工具与独立审核器实际核验的条件。TUI 体验、视觉质量及其他必须由人判断的事项不进入 phase / goal 的通过门；它们作为 dgoal 完成回复里的 `userReview`（用户复核）明确交给用户。

## 背景

一次修复任务的启动计划明确把真实模型 smoke 和人工 TUI smoke 列为 non-goal，并将确定性自动化测试写入 verification（验收说明）。阶段审核器却根据项目 `AGENTS.md` 把人工 TUI smoke 追加为通过前提。

这造成两个问题：

- 已确认并冻结的验收契约，被审核器在运行中扩容；
- 人工 TUI smoke 不能由隔离审核子进程或 LLM 自行完成、观察并诚实证明，phase 因而会在 `REJECTED`（拒绝）回环中永远无法完成。

严格审核不等于让不可独立核验的要求成为 dgoal 的停止条件。否则 dgoal 会把“需要用户判断”伪装成“agent 再努力即可解决”的实现失败。

## 决策

### 1. 完成门只收独立可验收条件

启动闸门确认的完成契约必须区分两类内容：

- **独立验收条件**：LLM 能通过项目工件、受限工具、可复现命令或可观察外部状态独立核验。它们可以进入 phase 建检和 goal 终审，并决定 `done`。
- **用户复核项**：需要用户实际使用、视觉判断、主观体验判断或其他 LLM 当前无法可靠观察的事项。它们不能进入任何 phase / goal 的通过前提。

若某项“必须完成”条件无法被 LLM 独立验收，提案不是可自主完成的 dgoal；启动闸门必须要求重新表述为可验收条件，或把它降为用户复核项。dgoal 不承担依赖人工签字的发布门禁；那是外部发布流程的职责。

### 2. 用户复核在完成后交付，不伪装为通过

`dgoal_done` 终审通过后，完成回复必须保留并清晰呈现 `userReview`：

- 明确写出尚未由 LLM 证明的体验/界面风险和建议操作；
- 不写“已验证 UI 体验”之类的虚假成功结论；
- 不阻止 goal 进入 `done`，也不触发 continuation（自动续跑）或 `REJECTED` 回环。

这让用户获得必要的复核提示，同时保持 dgoal 的 `done` 语义诚实：它只表示冻结的独立验收契约已通过。

### 3. 审核器不得运行时扩容完成契约

阶段审核与终审可以：

- 否定已有独立验收条件的证据；
- 发现实现、文档或测试与已确认目标不一致；
- 报告额外风险，并将不可独立验证的体验项建议为用户复核。

审核器不得仅因 `AGENTS.md`、README 或自身推断发现额外人工验证要求，就把它添加为 `FAIL`（失败）并阻塞 phase / goal。项目规范中可独立执行的强制命令，应在启动计划里显式成为独立验收条件；人工要求则应成为用户复核项。

## 实现约束

启动闸门已将"独立验收条件 / 用户复核项"结构化并持久化：

- `AcceptanceCriterion`（`{ criterion, evidence }`）—— `criterion`（完成条件）与 `evidence`（可独立复验的证据）均为必填非空字符串。goal 与每个 phase 都必须提供 `acceptanceCriteria: AcceptanceCriterion[]`；`dgoal_propose` 在 schema 与 execute 层强制校验，任一缺失或脏条件（空、非字符串、混入空 criterion/evidence）整体拒绝提案，不静默删除。`normalizeAcceptanceCriteria` 采用严格模式。
- `userReviewItems: string[]`（可选）—— 完成后交给用户复核的事项，不进入 phase/goal 通过门。启动闸门确认时可见，终审通过后与 agent 补充及审核器非阻塞建议合并输出。
- 这两个字段在 `GoalState` / `Phase` / `PlanProposal` 上持久化；旧 session 缺字段按空数组兼容加载，不迁移、不重编号。
- `dgoal_plan` 不提供修改 goal/phase `acceptanceCriteria` 的入口——验收条件只在 proposal 确认激活时写入，运行期不可扩容。
- `validateProposalInput` 做必填结构校验（非空、格式），不再用 evidence 词形启发式判断语义（此实现细节被 ADR 0037 覆盖）。校验通过后，`dgoal_propose` 在写入 `pendingProposal` 前调用当前会话 LLM 做计划级语义预审：人工条件与可复验证据的组合必须被拒绝或改写到 `userReviewItems`；`rewrite` 对每个被移除或改变的原始 criterion 必须返回精确 `sourceCriterion` → `userReviewItem` 迁移映射，缺失或无关映射同样 fail-closed；模型异常、非正常 `stopReason`、用户中断或无模型时 fail-closed。`buildProposePrompt` 继续负责提交前引导，审核器只对已冻结契约做终审兜底。
- 新 goal 的冻结完成门只有 `acceptanceCriteria`；`verification` 是 goal 级验收说明，不单独作为终审完成门。旧 session 缺少 `acceptanceCriteria` 时，终审兼容沿用 `verification` / task evidence / 本次 `dgoal_done` 证据。

审核 prompt 的动态输入边界：所有进入 `<dgoal_acceptance_contract>` 块及 phase/goal 审核 task 的动态字段（criterion/evidence、旧 verification、phase subject、task subject/evidence/blockedReason、goal summary/verification/whatChanged/userReview、userReviewItems）都经过 `escapeXml()`，防止动态内容闭合 XML 标签边界。`escapeXml()` 只防止标签闭合，不消除普通文本形式的 prompt injection；系统指令进一步约束审核器只审冻结契约。`buildPhaseCheckTask()` / `buildAuditorTask()` 与两级 system prompt（`PHASE_CHECK_SYSTEM_PROMPT` / `AUDITOR_SYSTEM_PROMPT`）明确禁止从 AGENTS、README 或人工体验要求扩容完成门；重审时上一轮反馈中的越权人工体验完成门不继续作为完成门，只按冻结契约重审。旧 session 缺少结构化契约时沿用 `verification` / task evidence / 本次 `dgoal_done` 证据作为兼容验收依据。

## 理由

- 建检循环的停止条件必须是 agent 可完成、审核器可判定的事实；否则循环不具备终止性。
- 用户复核保留体验与视觉质量的责任，不让自动化“通过”掩盖其未验证状态。
- 启动闸门承担范围确认，审核器承担独立判卷；后者不能反向改写前者确认的契约。
- 把人工发布门禁留在外部流程，避免 dgoal 从会话内建检循环膨胀为发布审批系统。
