# ADR 0038：三档 Plan 与八工具职责分离

> Status：已实现，待下一版本发布。

## 背景

pi-dgoal 原先把所有计划都建模成显式启动的复合 goal，并以 `final_only` / `phased`、有界 / 无上限预算、隐式 proposal 等参数组合表达保障强度。五个旧公共工具同时承担计划建立、状态改写、审核、完成和暂停。实践中出现了三个问题：

1. 普通任务没有轻量、默认的结构化计划入口；agent 要么不建计划，要么必须启动完整 dgoal。
2. `check` 同时负责“独立判卷”和“写完成状态”，审核结论与业务状态耦合，难以清楚表达“审核通过但尚未确认完成”。
3. 策略、预算、隐式授权与工具动作形成组合爆炸；工具名也从产品前缀而非动作职责出发，agent 难以稳定选对入口。

Task Plan 数据结构、持续显示浮层、独立审核器和 `/dgoal` 启动闸门本身仍有价值，不应另造一套 Plan Extension。

## 决策

### 1. 将保障强度收敛为三种 Plan 类型

三档 Plan 共用同一份 goal / phase / task 数据结构、持久化、浮层和状态机，以 `planType` 冻结差异：

| 类型 | 激活边界 | 结构 | 独立审核 |
|---|---|---|---|
| **Task Plan**（`task`） | agent 可为明确的多步执行任务主动建立 | 一个不对工具与 TUI 暴露的隐藏 phase + 多 task | 无 |
| **Phase Plan**（`phase`） | 用户显式 `/dgoal` 或明确要求使用 dgoal 后，经语义预审与确认 | 1..N phase + 多 task + goal 验收契约 | 仅 `goal_check` |
| **Goal Plan**（`goal`） | 同 Phase Plan | 1..N phase + 多 task + phase/goal 验收契约 | `phase_check` + `goal_check` |

Task Plan 是日常自动规划的默认入口，但不是所有回复的强制仪式：纯讨论、解释、能力问答和单步回答不建计划。AI 不能自行把任务升级成 Phase Plan 或 Goal Plan；需要冻结验收契约或独立审核时，只能说明理由并建议用户启动 `/dgoal`。

### 2. 公共工具面固定为八个两词工具

不再用 `dgoal_` 产品前缀命名 agent 工具；`dgoal` 只保留为产品名和用户命令。

| 职责 | 工具 |
|---|---|
| 建立三档 Plan | `task_plan`、`phase_plan`、`goal_plan` |
| 管理现有 Plan | `plan_create`、`plan_read`、`plan_update` |
| 独立审核 | `phase_check`、`goal_check` |

`plan_create` 只新增 task；三档 Plan 运行中都不新增 phase。`plan_read` 是纯读。`plan_update` 是 agent 可调用的 task / phase / goal 执行状态、证据、依赖和暂停原因写入口。Task Plan 可由再次调用 `task_plan` 原子替换 objective 与全部 task；旧 task 不合并、不保留运行历史。

### 3. `check` 只记录审核，`update` 才写完成

`phase_check` / `goal_check` 只运行独立审核并写入带 `revision` 的 `CheckRecord`：`approved | rejected | audit_error`。它们不把 phase 或 goal 标为 done，也不改变浮层完成显示。

`plan_update` 才能写完成状态：

- Task Plan：全部 task 带可复验 evidence 进入 done 后，可 `plan_update(target=goal,status=done)`。
- Phase Plan：phase 的 task 全部带 evidence 进入 done 后，可 `plan_update(target=phase,status=done)`；blocked 不算完成。全部 phase done 后运行 `goal_check`，当前 revision 为 approved 才可更新 goal done。
- Goal Plan：phase 的 task 全部带 evidence 进入 done 且当前 revision 的 `phase_check` approved 后，才可更新 phase done；blocked 不算完成。全部 phase done 后运行 `goal_check`，当前 revision 为 approved 才可更新 goal done。

任何会改变受审事实的 plan 写操作都增加 `revision`，并使对应的旧 check 与审核检查点失效。check 启动时固定本轮 revision；审核运行中若 Plan 已变化或被清除，结果直接丢弃，不得写回新 revision 或复活旧 Plan。由此明确表达：**check 通过不等于完成；完成是经过状态守卫的显式 update。**

Check 结果与 Goal 状态正交：Goal 只保留 `pending | active | paused | done`，phase/task 只保留 `pending | in_progress | done | blocked`；旧 `completed` 别名删除。`rejected` 只存在于 `CheckRecord.status`，Plan 继续保持 `active`；业务拒绝不再建立 Goal `rejected` 状态，也不按固定拒绝次数进入专用暂停。`audit_error` 才进入 `paused(audit_error)`。若 `goal_check` 指出已完成 phase 仍有问题，只有在当前 goal check 为 rejected 时，才允许 `plan_update(target=phase,status=in_progress)` 重开该 phase；该写操作提升 revision 并清除 goal check。

### 4. 删除策略与预算组合

新 Plan 不再暴露或持久化以下运行时选择：

- `verificationPolicy`、`final_only`、`phased`；
- `budgetPolicy`、`bounded`、`unbounded`、`runtimeBudget`、宽限与预算用量；
- `implicitFinalOnlyStart`、`implicitFinalOnlyBudget`、proposal 的 `implicit` 参数和隐式启动权限。

保障强度只由 `planType` 表达。固定技术熔断仍保留：用户中断、模型错误、审核器错误、审核子进程 timeout、连续无进展，以及 `plan_update(target=goal,status=paused,reason=...)` 的 agent 主动暂停。

### 5. 复用现有启动闸门与展示层

`/dgoal` 仍由当前会话模型读取上下文、推荐 Phase Plan 或 Goal Plan，并调用对应入口提交。两者继续经过 proposal 语义预审和用户确认；确认 UI 可要求 agent 改用另一入口重新提交，但不再切换验收/预算策略。

三档 Plan 复用现有 Task Plan reducer、持久化和 TUI：Task Plan 默认展示 task；Phase/Goal Plan 默认展示 phase，并显示聚合的 task 完成数。Task Plan 的隐藏 phase 不出现在工具结果、prompt 或 TUI。

## 覆盖关系

- 保留 ADR 0006 的 goal / phase / task 三层与建检循环原则，但覆盖其中 phase 自动聚合完成、单一 check 工具与 check 直接写完成的旧机制；改由 `planType` 选择审核密度、`plan_update` 显式写完成。
- 覆盖 ADR 0002 的旧 proposal 工具名、ADR 0003 的单 check 工具、ADR 0018 的单 phase 合并双审实现。
- 覆盖 ADR 0004 与 ADR 0023 的 Goal `rejected` / 固定三次拒绝暂停 / Goal Repair 展示投影；check rejection 改为与 Goal 状态正交。
- 部分覆盖 ADR 0005：保留 blocked 与必填原因，但 blocked 不再作为 phase/goal 完成放行条件。
- 保留 ADR 0011 的失败报告持久化和 ADR 0028 的审核候选切换，但覆盖其中依赖 Goal `rejected` 与固定次数暂停的部分。
- 扩展 ADR 0019：Task Plan 成为无需用户激活的日常默认；Phase/Goal Plan 仍受显式 dgoal 激活边界保护。
- 覆盖 ADR 0031 的独立暂停工具：暂停并入 `plan_update`。
- 覆盖 ADR 0032 的验收/预算策略，以及 ADR 0034/0035 的隐式轻量启动与预算护栏。
- 保留 ADR 0036 的自然语言显式 dgoal 授权，以及 ADR 0037 的 proposal 语义职责分层。

## 后果

- agent 只需先判断任务是否值得建 Task Plan；用户显式启动 dgoal 后再判断 Phase Plan 与 Goal Plan，工具选择路径变短。
- 审核结论和完成状态可分别观察、持久化和测试，旧批准不会在计划变化后被误用。
- 删除旧五工具、策略/预算字段和隐式权限会造成一次持久化与 API 破坏性升级；新运行时使用 `dgoal-plan-v1` entry，不迁移旧状态。
- 固定熔断不是用户可配置预算；dgoal 仍是会话内单 Plan，不是后台调度器或多目标池。
