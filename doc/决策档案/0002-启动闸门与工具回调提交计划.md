# ADR 0002：启动闸门 + 工具回调提交计划

> Status：原则保留、工具形态被 ADR 0038 覆盖。Phase/Goal Plan 仍经启动闸门确认，入口改为 `phase_plan` / `goal_plan`；Task Plan 不经过启动闸门。

## 背景

`/dgoal` 启动时需要先产出结构化 plan（goal + 初始 steps）给用户确认，确认后才进 loop。两个子决策：plan 由谁产出、用什么形式提交。

## 决策

1. **启动闸门**：`/dgoal <objective>` → 主代理整理 plan → 弹确认 UI（列出 goal + 初始 steps + 确认/拒绝/输入反馈）→ 确认后进 loop。不走纯自主模式（agent 直接进 loop 自建 plan）。
2. **工具回调**：主代理用 `dgoal_propose` 工具提交结构化 plan（goal + steps 数组），不靠文本解析。工具 execute 存参数 + 触发确认 UI；`agent_end` 检测产出。

## 为什么

**启动闸门**：纯自主模式的失败是 plan 跑偏在 loop 内不可见，用户只能等结束或中途打断才发现——南辕北辙往往体现在步骤拆解里。启动闸门用一次人工确认把这个成本前置付掉。

**工具回调**：steps 是数组结构（id/subject/blockedBy），工具 schema 能强制结构，文本解析保证不了嵌套字段且格式漂移会失败（plan mode 示例的 "no plan unprompted" 报错即此问题）。工具调用本身也是"主代理整理完毕"的信号，兜底简单（没调工具 → 降级提示）。

## 权衡

启动闸门备选是纯自主（无确认）。纯自主契合 goal 范式自主性、支持无人值守，但 plan 质量和方向失控，与 dgoal"可控"诉求冲突。

工具回调备选是文本解析（主代理输出 markdown，正则提取）。零额外工具但解析脆，`blockedBy` 等嵌套字段无法可靠提取。工具回调多一次工具调用，但这次调用就是"整理 plan"的显式动作，比自由文本可靠得多。`dgoal_propose` 与 `dgoal_done` 对称（提交计划 / 提交完成），模式统一。

## 代价

主代理启动时多产出 steps + 多调一次工具。主代理不调 `dgoal_propose`（跑偏无产出）的兜底（拷问 25 决）：降级提示重试 2 次（首次 propose 指令 + 2 次降级引导 = 3 次机会），仍无产出则中止（goal 不进 active，pending 直接清除，不污染状态机）。**不走"进 loop 自建"的纯自主兜底**——纯自主在拷问 6 已被否决（plan 跑偏不可见）。中止而非 paused：这是启动阶段失败（未进 active），不是运行中卡住。重试阈值 2 与现有 `MAX_ERROR_RETRIES=3`、终审 ×3 保持心智一致。
