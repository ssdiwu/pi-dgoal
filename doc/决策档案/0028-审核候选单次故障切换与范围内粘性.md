# ADR 0028：审核候选单次故障切换与范围内粘性

## 状态

已采用，取代 ADR 0015 中“同一模型部分输出最多续审三次”的运行时规则；ADR 0015 保留作历史记录。

## 决策

审核候选链把“模型故障切换”和“业务拒绝修复”分开：

- 同一次审核调用中，每个候选最多尝试一次。
- 候选出现技术/协议错误、无终止标记的部分输出或零输出错误时，按候选顺序切换下一个；不在同一候选上重复三次。
- 候选产生有效 `APPROVED` / `REJECTED` 后立即结束本次审核。
- 在当前 goal、同一审核范围（phase 或 goal）内，产生有效结论的候选被持久化为健康候选，后续同范围审核优先复用；已故障候选也按范围隔离记录。
- `/dgoal resume` 从 `audit_error` 恢复时只清除产生错误的审核范围（phase 或 goal）的候选故障记录，保留另一范围的健康候选；旧 goal 缺少范围字段时兼容为全量重置。

业务拒绝不等于模型故障：

- phase 的有效 `REJECTED` 持续回到 `in_progress`，不受三次上限约束。
- goal 终审的有效 `REJECTED` 才累计 `rejectedCount`，连续第三次进入 `paused(audit_failed_3x)`。

## 理由

“同模型三次续审”把一次审核的技术故障预算和 goal 修复回环混在一起，导致最多 `3 × 3` 次调用、等待时间放大，并且成功 fallback 后下一轮又从坏候选开始。按候选单次切换可以快速绕过故障；持久化范围内健康候选可以让后续修复回环稳定使用已验证可用的审核器；phase/goal 分层则保留建检阶段可持续修复、终审三次暂停的不同语义。

## 边界

- 不改变候选配置格式、审核 prompt 或 acceptanceCriteria。
- 不把多个候选的业务结论当作投票。
- 不因业务 `REJECTED` 换模型。
- 候选故障状态属于当前 goal，不跨 goal 继承。
- TUI 只展示切换进度，不拥有候选状态事实。
