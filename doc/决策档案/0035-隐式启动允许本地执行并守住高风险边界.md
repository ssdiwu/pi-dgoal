# ADR 0035：隐式启动允许本地执行并守住高风险边界

> Status：已接受（Unreleased）。覆盖 ADR 0034 中“隐式启动不允许 shell 执行 agent 可编辑脚本或测试代码”的限制；其余全局授权、`final_only + bounded`、语义预审与预算边界继续有效。

## 背景

ADR 0034 为隐式轻量启动采用了“本地文件工具可读写，但 shell 只放行枚举式只读命令”的保守边界。真实自举测试证明，这会在 dgoal 已成功自主启动后立即拦截 `bun test`，导致任何以测试或构建为独立验收证据的代码任务都无法在隐式模式闭环。

全局 `implicitFinalOnlyStart` 已是用户主动授予的高权限事实。用户确认其期望不是“隐式目标只能读”，而是“agent 可以在本地完整工作，但高风险仓库破坏和外部动作必须得到显式授权”。继续禁止测试、构建和本地脚本会让隐式 dgoal 只适合极窄的文档读取任务，与独立验收目标冲突。

## 决策

- 全局授权的隐式 dgoal 允许执行本地测试、构建、解释器脚本、项目文件增删改和 package manifest 修改。
- 允许本地 Git 变更，包括 `add`、`commit`、`reset`、`checkout`、`tag` 和仅修改本地配置的操作。
- 下列动作必须改走用户显式 `/dgoal`：
  - 删除整个当前工作仓库；
  - 删除、移动或直接破坏 `.git`；
  - `git push` 等把提交写到远端的动作。
- 发布、部署、外部写入、账号或权限修改、付费动作继续禁止隐式执行；外部只读访问仍可使用。
- 路径型文件工具继续限制在受信任项目目录内，并保留符号链接 / 多链接逃逸防护。shell 允许运行本地程序；运行时必须使用 Pi 的 `tool_call` 执行前 preflight，而不是 `tool_execution_start` 后 abort，对已识别的高风险命令做 fail-closed 拦截。
- proposal 动作扫描覆盖包括 `contextSummary`、`nonGoals`、`guardrails` 和 `userReviewItems` 在内的全部自由文本字段，并按 clause 区分命令式动作与 `不 / 禁止 / must not` 等否定式边界，既不反向误判安全声明，也不让自由文本成为命令注入通道。
- 隐式启动仍只允许 `final_only + bounded`，不改变全局配置授权、semantic preflight、独立终审、运行预算和用户随时 pause / clear 的约束。

## 后果

隐式 dgoal 可以真正完成“改代码 → 跑测试 / 构建 → 独立终审”的本地闭环，自举测试不再需要为了执行 `bun test` 改走显式模式。本地执行能力因此扩大，用户承担项目代码自身副作用的风险；运行时护栏是策略检查而不是 OS sandbox，不能证明任意脚本内部绝无隐藏外部副作用，因此 proposal 语义预审、已知外部动作拦截和全局 only 授权仍必须同时保留。

高风险动作的判断从“是否执行代码”转为“是否破坏仓库身份 / 整体工作区，或把状态写到本机之外”。这条边界需由 prompt、proposal 校验、运行时工具钩子和回归测试共同保持。
