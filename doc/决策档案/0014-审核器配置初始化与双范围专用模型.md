# ADR 0014：审核器配置首次初始化与双范围专用模型

阶段建检与目标终审改用两个独立的 `pi-dgoal.json` 字段：`phaseAuditorModel` 与 `goalAuditorModel`。首次实际审核时，如果全局和受信任项目级配置文件都不存在，原子创建全局模板；模板以两个 `null` 值显式表示继承当前会话模型。

## 背景

ADR 0013 已决定审核器配置归 dgoal 专属 `pi-dgoal.json`，并把“不自动创建配置文件”作为当时边界：用户不需要选模时不产生文件副作用。

实际使用中，阶段建检和目标终审需要独立选择模型和 Pi 思考等级。只靠一次性路径提示，用户需要自行记住字段、格式和两级配置优先级；主会话换模或 Pi 重载后，也难以一眼确认审核器是否已配置为稳定的专用模型。

## 决策

1. 新增专用字段 `phaseAuditorModel` 与 `goalAuditorModel`，分别只供 `dgoal_check` 与 `dgoal_done` 的独立审核子进程使用。具体 `provider/model[:thinking]` 值是持久化的专用设置，不随主会话换模或 Pi 重载变化。
2. 保留旧 `auditorModel` 作为兼容共享回退：同一配置来源中，范围字段优先于共享字段；来源优先级先于字段优先级，即受信任项目级配置优先于全局配置。
3. `null` 是显式继承当前会话模型；当前范围未配置、为 `null`，或字段缺失/非法且全链降级后仍无有效值时，才使用当次 `ctx.model`。它是模板默认值，不是强制产品默认模型。
4. 首次**实际**审核时，若全局和受信任项目级 `pi-dgoal.json` 都不存在，使用独占写入（`wx`）原子创建全局模板；不在扩展加载时写文件，不覆盖已有文件。创建失败、配置不可读、坏 JSON 或字段非法时，提示后继续按优先级安全降级，不阻断审核。

这项决策明确反转 ADR 0013 的“配置文件不被自动创建”边界；ADR 0013 关于配置文件落点、信任边界和不影响主执行线程的结论仍然有效。

## 理由

- 两个审核范围有不同的成本和严谨度需要，字段分离让设置的归属和生效范围直接可见。
- 模板把可用字段、`provider/model[:thinking]` 格式与 `null` 继承语义写入用户实际编辑的位置，减少一次性提示的记忆成本。
- `wx` 独占创建同时保留首次初始化便利和“绝不覆盖用户配置”的安全边界。
- 不在代码中硬编码审核模型，用户可在全局或受信任项目级配置中固定自己的专用模型。

## 取舍

- **代价**：首次审核会产生一个可见的全局配置文件副作用，扩展也需要维护字段级校验、提示去重和模板写入失败降级。
- **换来**：用户能持久化两个明确的审核模型设置；具体设置与主会话模型解耦，并且首次配置可发现、可编辑、可验证。

## 边界

- 仅全局和受信任项目级配置文件均不存在时创建模板；已有但不可读或格式错误的文件只告警和降级，不重写、不替换。
- 提示去重按消息类型、配置文件路径和字段区分：不同字段的问题分别报告，同一问题不在每次审核重复刷屏。
- 项目 `.pi/pi-dgoal.json` 仍受 `ctx.isProjectTrusted()` 保护。
- 该配置不改变主执行线程模型，也不引入自动换模、备用模型或容灾策略。
