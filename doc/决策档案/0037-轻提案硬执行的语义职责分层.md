# ADR 0037：轻提案、硬执行的语义职责分层

> Status：语义职责分层继续有效；ADR 0038 已把公共 proposal/check/done 工具边界替换为三档 Plan 与 `phase_check` / `goal_check` / `plan_update`。

## 背景

`dgoal_propose` 的语义预审最初用于解决一个真实死锁：计划把人工体验、用户签字或主观判断冻结进 `acceptanceCriteria`，独立审核器无法自行完成或证明这些条件，goal 因而永远卡在 rejected 回环。正确目标是让模型理解计划语义，把人工体验迁移到 `userReviewItems`，并在确实缺少只有用户能提供的决策、信息或授权时提前暴露 blocker。

后续实现又在 LLM 语义预审之前叠加了 evidence 关键词启发式和 proposal 全自由文本动作扫描。一次本地只读真实工具链自测因此经历多次无意义重提：否定式 `nonGoals` 被当成待执行动作，真实 `dgoal_plan` 工具响应因缺少特定证据词形被拒，校验失败后还留下 pending goal。终审又一度要求当前 `dgoal_done` 的成功响应在当前审核作出批准前预先存在，形成因果循环。

这些问题说明：结构、语义、安全动作和结果验收被混在同一层重复判断。继续扩写正则只会增加词形摩擦，不能获得可靠语义，也不能替代执行前安全边界。

## 决策

### 1. 采用“轻提案、硬执行”分层

代码、LLM、执行护栏和审核器各自只承担一类责任：

- **代码管结构与授权**：字段类型/必填、状态机、验收与预算策略组合、预算数值、用户激活来源和一次性授权绑定；这些确定性事实继续 fail-closed。
- **LLM 管提案语义**：语义预审是 proposal 含义的唯一判定者，重点判断任务能否自主闭环、哪些条件需要人工介入，以及 evidence 是否在当前工具边界内可独立复验。
- **执行护栏管真实动作**：安全硬边界以 `tool_call` 的真实工具名、参数和项目边界为准；LLM 不是安全边界。
- **审核器管冻结结果**：阶段/目标审核只核用户已确认的冻结完成门和真实回归，不重新扩张契约。

“轻提案”不等于取消 Task Plan、验收条件、语义预审或显式确认；“硬执行”也不等于把安全交给模型。

### 2. LLM 将人工依赖分为三类

语义预审对每条候选条件作以下语义归类：

1. **独立验收条件**：可由项目工件、受限工具、可复现命令或可观察外部状态核验，保留在 `acceptanceCriteria`。
2. **用户复核项**：视觉、体验、偏好或其他主观确认，迁移到 `userReviewItems`，不阻塞 goal 完成。
3. **真实人工 blocker**：缺少只有用户能提供的决策、信息、凭据或授权，且不改变目标就无法迁移；隐式路径若显式确认即可补足授权，应先降级显式，否则才拒绝 proposal，并明确需要用户补什么。

预审不得因为“更完整”而扩张目标，也不得把普通用户复核升级成启动阻塞。

### 3. 移除提案层的泛化语义硬编码

`validateProposalInput` 只保留结构和组合不变量。evidence 非空仍是结构要求，但不再用命令名、文件扩展名、`API response JSON` 等词表作为语义通过门；真实可复验性由 LLM 语义预审判断，并由最终独立审核兜底。

proposal 不再把 `objective`、task、`nonGoals`、`guardrails`、`userReviewItems` 等自由文本按关键词作泛化语义硬拒；当前会话 LLM 负责理解这些字段，若显式确认即可补足授权，则返回 `requiresExplicitConfirmation` 并降级为普通 pending goal。真正执行时仍由 `tool_call` preflight 对已识别的高风险动作 fail-closed；即使 LLM 漏判，这些动作也不能越过执行边界。

此项覆盖 ADR 0016 中“轻量 evidence 形态启发式作为硬拒绝”的实现细节，以及 ADR 0035 中“全自由文本 proposal 扫描作为语义硬门”的部分；不覆盖两者的独立验收边界、全局隐式授权和运行时高风险动作拦截。

### 4. Proposal 状态写入必须具备事务边界

新隐式 goal 必须在权限、结构校验与语义预审成功后再原子写入 goal + pending proposal。失败结果不得留下不可见的半启动 goal；若产品明确选择保留可重提 pending，则返回结果必须显式说明其状态和下一次调用方式，不能让重试只看到“需要冷会话”。

### 5. 终审不得要求当前结果预先存在

`dgoal_done` 内部审核发生在当前 tool result 生成之前。审核器应核验调用前已完成的冻结条件、工件和回归；其 `<APPROVED>` 决策本身触发 `finalizeGoal` 并生成成功响应。不得把“当前 `dgoal_done` 已批准响应”设为当前审核的前置证据，也不得因调用前 goal 仍是 active/rejected 就拒绝。

## 后果

- 原始防死锁目标保留：人工体验不再进入机器完成门，真实人工 blocker 仍在启动前暴露。
- 提案阶段减少依赖魔法词和正则的重复摩擦；显式确认与运行时硬护栏继续承担真实权限和动作风险。
- LLM 语义预审成为更关键的语义依赖，可能存在模型波动；通过窄任务、结构化 `approve / rewrite / reject` 协议、精确迁移映射、技术失败分流和最终独立审核约束，而不是再叠一套语义词表。
- 实现与测试应分别覆盖：人工条件三分流、否定式边界不误报、隐式不确定项降级显式、失败 proposal 不残留半启动状态，以及当前 `dgoal_done` 结果不构成自指完成门。
