# ADR 0011：建检反馈持久化到 LoopGoal

建检反馈（`dgoal_check` 或终审未通过后，检查 agent 返回给主 agent 的原始失败报告）必须持久化到 `LoopGoal`，而不是只依赖上一轮 tool result 或对话 transcript。原因是 dgoal 的基本盘是不靠 transcript 存活：session compact、恢复或多轮 rejected 回环后，主 agent 仍必须看到完整、保真的失败反馈，避免因报告丢失而瞎修。建检反馈主要是 agent-facing 的修复输入，不是要求用户介入的过程展示；用户只关心启动前确认、goal 完成结果，以及建检循环连续失败到暂停这类需要介入的节点。工具结果仍可完整展示原始报告，让用户偶尔可读、AI 也能读到当轮结果，但本决策不要求在 `/dgoal s` modal 或 aboveEditor 浮层额外展示原始报告。报告保存检查 agent 给出的原文，不再生成 summary，也不压缩；如果报告过长，应约束检查 agent 的报告格式，而不是在反馈链路里二手转述。阶段建检反馈与终审反馈生命周期不同：阶段建检反馈服务于当前 phase 的重做，通过该 phase 后必须清除；终审反馈服务于 goal 级 rejected 回环，终审通过或 goal 清除后自然结束；终审连续 3 次失败进入 `paused(audit_failed_3x)` 时仍保留终审反馈，`/dgoal resume` 清零 `rejectedCount` 但不清除 `finalFeedback`。阶段建检反馈在数据形态上使用 `phaseFeedbackById` 按 phaseId 定位，但这不表示允许多个 phase 并行失败；dgoal 仍保持阶段顺序执行，`active` 时只注入当前 phase 的阶段反馈，`rejected` 时只注入终审反馈，旧 phase 通过后清理对应反馈。Datawhale 推文里的“同一失败 / 无进展 / 回归”等停止规则不在本决策中建自动判定器：dgoal 只负责把检查 agent 的原始报告稳定交还给主 agent，主 agent 基于报告自行判断是否同源失败、是否无进展、是否需要拆 task 或升级为 blocked。备选方案是只把失败报告返回到当前工具结果中，但这无法跨 compact/恢复稳定钉回，违背“建检反馈闭环增强”的目标。
