# ADR 0036：自然语言显式启动复用启动闸门

> Status：已实现（v0.7.1）。覆盖 ADR 0019 中“只有字面 `/dgoal` 命令才能创建显式启动授权”的入口限制；不改变 ADR 0034/0035 的隐式轻量启动边界。ADR 0037 进一步规定冷启动必须在结构与语义成功后才写入 goal 并消费授权。

## 背景

用户已经在自然语言里明确说“请用 dgoal”“启动 dgoal”或“你可以用 dgoal 和 dteam 自己处理掉”时，主 agent 能理解授权，却仍可能因为当前没有 `/dgoal` 命令创建的 pending goal 而提交失败，随后要求用户再输入一次 `/dgoal`。

这既重复确认，也会让复杂目标卡在两条路之间：它们需要 `phased`、`unbounded` 或 GitHub/发布等外部动作，不能走受限的 `implicit=true`；但自然语言授权又不能打开原有显式启动闸门。

## 决策

1. **自然语言明确指令可授予一次显式启动权**：Pi `input` 事件只对空闲状态下的真实用户来源（`interactive` / `rpc`）识别“祈使动作 + 独立 dgoal token”的明确指令；能力询问、引用/代码示例、解释讨论、仅否定使用 dgoal 的表达、标识符后缀及 `extension` 注入消息不授权；“不是……而是需要你用 dgoal……”这类后半句明确授权的转折祈使可命中。处理中追加的 `steer` / `followUp` 也不授权。
2. **授权精确绑定 dgoal 实际观察到的输入**：输入命中后，在 `before_agent_start` 要求实际 prompt 与该文本完全一致；后加载扩展若 transform 输入，授权 fail-closed 清除。授权只存在内存中，在消费、session/tree 生命周期结束或 agent settled 后清除，不持久化。
3. **复用显式启动闸门**：冷会话中，带该授权的 `dgoal_propose` 不设置 `implicit` 时，运行时先执行结构校验与语义预审；成功后才消费授权并原子创建 pending goal，再进入用户确认 UI。失败不留半启动 goal，用户也无需补输 `/dgoal`。
4. **完整策略仍由用户确认**：这条路径可提交 `phased` / `final_only`、`bounded` / `unbounded`，也可在计划中声明外部写入、推送或发布动作；它们只有在用户确认启动闸门后才进入普通显式 dgoal 执行语义。
5. **不扩张隐式权限**：`implicit=true` 仍只允许 ADR 0034/0035 定义的全局授权 `final_only + bounded` 本地/只读路径及其高风险动作护栏。自然语言显式启动不是隐式启动的绕过。
6. **只处理冷会话启动**：已有 pending/active/rejected/paused goal 时，不用自然语言授权静默替换；替换或状态控制继续走现有 `/dgoal` 命令与确认流程。
7. **不冒充不可变原始输入边界**：Pi 0.80.7 的 `input` transform 会按扩展加载顺序串联，事件没有不可变 `originalText`。因此 dgoal 能阻止 `source=extension` 及自身之后发生的 transform，却无法区分自身之前由受信任扩展改写的文本。Pi 扩展本就拥有完整本机代码权限，恶意扩展不属于 dgoal 可建立的安全隔离边界；这项限制必须显式记录，不能宣传成对其它扩展的 sandbox。

## 权衡

- 意图识别可能有假阴性；用户仍可用 `/dgoal` 作为确定性兜底。
- 假阳性不会直接执行任务：结构与语义通过后最多建立 pending proposal，仍有阻塞式确认 UI。
- 不把“文本里出现 dgoal”或问句视为授权，牺牲部分礼貌问句覆盖，换取默认不接管的安全边界。
- 早于 dgoal 执行的受信任 input transform 仍可能改变其观察到的文本；确定性、不可被 transform 歧义影响的入口仍是 `/dgoal`。

## 结果

用户明确说“用 dgoal 处理”后，主 agent 可以一次提交完整计划。安全轻量任务仍可选择全局授权的隐式路径；复杂或含外部动作的任务直接进入显式确认，不再要求重复输入 `/dgoal`。
