# ADR 0032：提案阶段确认验收与预算策略

> Status：已实现（v0.7.0）。

## 背景

当前 dgoal 把所有多 phase 计划都固定为“逐阶段独立建检 + goal 终审”：每个 phase 都必须有独立 `acceptanceCriteria`，通过 `dgoal_check` 才能继续，最后还要走 `dgoal_done` 终审。对确有中间交付门的工作，这是必要的；但普通复合任务常把代码、测试、文档等工作面机械拆成 phase，于是同一交付承担多次独立审核，执行时长和 token 成本失控。

当前 `budget` 也只是启动闸门里的文本信号，不能影响运行时；固定的“终审连续 3 次 rejected 即暂停”不能表达用户愿意让一个会话内长程任务持续运行的选择。

问题不是缺一套独立 Plan Extension（计划扩展）。dgoal 已有用户确认的计划、持续显示浮层、自动推进、暂停与完成交付；需要让用户在确认同一份提案时，选择**建检密度**与**运行预算**，而不是由 agent 静默决定。

## 决策

### 1. 同一 dgoal，不分裂为 Plan / Goal 两套系统

保留一个 dgoal、同一份 Task Plan、启动闸门、持续显示浮层、自动推进和完成回复。差异叫作 **Verification Policy（验收策略）**，不是新的运行模式或独立扩展。

不引入“资格判定后把任务分流到另一套计划系统”的前置流程；提案仍须经过现有启动闸门和用户确认。

### 2. 验收策略与预算策略必须在提案阶段由用户选择

`dgoal_propose` 应携带主 agent 的策略推荐及其理由；推荐不是隐藏决定。启动闸门的**同一确认框**必须同时展示并允许用户覆盖：

- 验收策略：`final_only` 或 `phased`；
- 预算策略：有界预算或 `unbounded`；
- 有界预算的具体上限与一次预授权宽限。

两个控件默认选中 LLM 推荐值；用户可直接确认该推荐，或改选任一策略后再确认，不增加第二个阻塞对话框。确认后的实际选择与 plan 一起冻结、持久化；运行期不能由 agent 静默切换。旧 goal 不迁移，继续沿用创建时的现有 phased 语义。

### 3. `final_only`：phase 组织进度，goal 保留独立终审

`final_only` 适合没有真实中间验收门的普通复合任务：

- phase 只组织 task、浮层和上下文；不要求 phase 级 `acceptanceCriteria`，也不逐 phase 调 `dgoal_check`；
- 一个 phase 的 task 均完成后，主 agent 回顾该阶段工作，可将其标为**阶段进度完成**；TUI 与正常完成 phase 一样划线显示，帮助用户读进度；
- 阶段进度完成不是独立建检通过，不得复用当前 `done`（“已通过阶段建检”）的业务语义，也不得作为独立证据；其具体持久态编码由实现保持这一语义边界；
- 所有 task 完成且各 phase 已阶段进度完成后，才由 `dgoal_done` 发起一次 goal 级独立终审。终审仍是唯一的 goal 完成门。

因此，轻量化减少的是不必要的阶段审核，不是删除独立验收。

### 4. `phased`：保留现有的逐阶段建检

`phased` 只用于每个 phase 都有独立验收价值、通过后能降低后续不确定性或解锁下一步的计划：

- phase 仍须冻结独立 `acceptanceCriteria`；
- `dgoal_check` 仍是 phase 进入已建检完成的唯一入口；
- 后续 phase 与 goal 终审仍受建检闸门锁定。

这是用户可选的高保证策略，不再是普通多 phase 计划的隐式成本。

### 5. 终审采用“诊断审 + 窄确认审”收敛

`final_only` 的首次 goal 审核是**诊断审**：审核器应一次找全冻结完成门的 blocker、实际回归和高风险证据缺口，不报告无关优化或 nits。

若诊断审拒绝，主 agent 修复并重新自测后，下一次是**窄确认审**，只核验：

1. 上轮 blocker 是否修复；
2. 修复后新增的 diff；
3. 受影响回归测试与少量全局保护测试。

窄确认审不得新增冻结完成门、偏好或无关 nits；但新增 diff 确实造成的回归仍可拒绝。后续修复回合沿用这个窄确认边界，直至通过、预算暂停，或发生需要用户决定的真实冲突。

`final_only` 不新增审核专用面板或历史视图：阶段进度完成沿用 phase 划线；goal 终审复用既有建检活性快照，以及拒绝后的 `Goal Repair` 标题/状态栏投影。诊断审与窄确认审只约束审核范围，不增加常驻 UI 噪音，审核报告正文仍只服务主 agent 修复。

主 agent 每次准备终审前提供 **Verification Bundle（验证包）**，至少说明改动、冻结条件到命令/工件的映射、最后一次改动后的自测、已知风险和未覆盖边界。验证包只帮助审核器定位；它不是主 agent 自述的通过结论，不能替代独立审核。

ADR 0030 的边界不变：可复用的独立证据仍仅限同一 workspace fingerprint 下审核 child 成功产生的工具事实。

### 6. 预算是运行策略，不只是文本说明

- **有界预算**：用户确认具体的执行/时长/审核修复回合上限。首次耗尽时发出一次非阻塞提示，并在状态栏持续标记“宽限中”，随后自动使用一次用户已预授权的宽限；宽限也耗尽才暂停，并保留已做工作、审核反馈与下一步建议。
- **`unbounded`**：用户显式选择后，取消执行轮次、墙钟时长和“终审三次 rejected”这类预算暂停；它只在 Pi 会话保持运行时有效，不引入后台 worker、定时调度或跨 session 执行。
- 无论何种预算，模型错误、无进展、用户暂停、审核技术错误和真实的用户决策死锁仍保留原有安全出口。
- `unbounded` 遇到无法靠 agent 自己解决的真实冲突时，必须 `paused(agent_blocked)`，保留证据与需要用户作出的决定；不得把未满足完成门伪装成 done。

## 后果

- `dgoal_propose`、启动闸门、持久态、状态机、工具约束、审核 prompt、TUI 文案和测试接缝都需要按冻结的策略分支执行。
- 新 proposal 的 phase `acceptanceCriteria` 只在 `phased` 下是必填；goal 级独立 `acceptanceCriteria` 始终必填。
- 现有 `auditCheckpoints` 的独立性与 workspace 边界不放松；验证包只降低审核定位成本。
- 实现必须为“阶段进度完成”和“已通过阶段建检”保留不同业务语义，即使二者在浮层都可使用完成划线展示。

## 非目标

- 不新建独立 Plan Extension，不复制第二套状态机或计划 UI。
- 不取消启动闸门、用户确认、自动执行或 goal 级独立终审。
- 不把主 agent 自测声明当作审核通过证据。
- 不做跨 session goal、后台 worker、定时循环或多 goal 池。
- 不让重审以“新发现的偏好”无限扩张冻结完成门。
